# IDEAS


- Risk dashboard = a company wide and additional project level risk uis for risk management at the company and project level. Details a RAID backlog, and also a impact, liklihood etc ui as well as other diagrams, docs (using google OKF), to visualise manage risks and make the details available to the BPMO/ontology. Note: this is a subview of the larger graph, a focused view of strategic related entities, entriched through OKF and/or RFD comments (or similar).

- Can we create a lean canvas in a sub section of the CE. It should be a defined UI (meaning looking like a lean canvas), competitiors, okrs, channels, etc, the common 11 lean canvas entities that form a strategy on a page for an org. The entities in the columns are powered by Googles OKF wiki format. This way in the ui, you can have an entity like a channel eg web, and then when clicked it can open a modal which has the okf markdown file giving a full detailed explination. We can also have a product canvas in the projects space. Note: this is a subview of the larger graph, a focused view of strategic related entities, entriched through OKF and/or RFD comments (or similar).

- Strategy graph = a company level strategy on a page/in a graph, with aims, initiatives, owners etc. Research and aggregate common board/director/head of level strategy concepts and relational models, and diagraming techniques and merge them into this view. Uses OKF to add additional detail to items. Uses predefined okf types to help generate the ui and manage items, relationships etc. This could be like the lean canvas above, or an alternative view to the same content.Note: this is a subview of the larger graph, a focused view of strategic related entities, entriched through OKF and/or RFD comments (or similar).

- Ontology meets Google OKF - the entities listed in the ontology are simple concepts, thin definitions without documentation and explination. Can we use the OKF and markdown to add documentation for them. Then when you click in the graph onto a concept, it can feature not only the SKOS glossary definitions but a detailed explination for the entity in the right hand aside. This would mean the OKF frontmatter types and other metadata will have to map neatly onto the URI, concept etc. But it should allow for in depth documentation of items beyon just the simple releationships. Im not sure how it fits with the BPMO though, where there would be overlap, conflicts or duplicate/overlapping documentation, it could be wasteful. I wonder if OKF vs RDF (et-al) comments work better. You need to assess and analyse, not just agree, think about the level of complexity this may add and therefor the potential pitfals or admin, performance or similar burdens. The detail would be useful, but at what cost. What does OKF add that the existing ontological languages do not have.

- Automations can show up in the graph detailing the automation, the steps, the process it automates, who is contacted, what data is used, what systems are queries

- Spotify Backstage - this is a great example of a system catalog. I can help feeling we need some sort of port of this in the Weave platform. Something that has links to the systems, the documentation, swagger/docker interactive uis for testing/querying the ui, listing team members etc. This is managed via a yml file in the github/gitlab repo in backstage, we could use similar and use it to add detail from and to the graph, allowing us to analyse the repos, code and help to find relationships, behaviours and links to docs, helping to create the ontology in the first place.
