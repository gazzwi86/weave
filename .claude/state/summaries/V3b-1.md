# TASK V3b-1 ‚Äî Wire Ask-the-model bar + populate KPI strip

Branch: `feature/v3b-askbar-kpi`. Two commits, one per item, per the task brief.

## Item 1 ‚Äî Ask-the-model bar

New `packages/frontend/components/explorer/canvas-ask-bar.tsx` (`CanvasAskBar`)
mounts the existing `AskBar` molecule bottom-centre inside
`ExplorerInteractions` (new last line in its returned fragment). It reuses
`useAskLifecycle` from `app/ce/query/use-ask-lifecycle.ts` verbatim (same
`POST /api/query/nl` call the `/ce/query` page already makes) ‚Äî no new
backend call, no new hook. Codebase precedent for a `components/**` file
importing a hook from `app/**` already exists (`explorer-tour.tsx` imports
`useOnboardingPath` from `app/settings/onboarding-path`), so this isn't a
new layering pattern.

On success/error it renders one feedback line under the bar, sourced from
the pre-existing (previously unwired) `lib/explorer/ask-view.ts` mapper
(`toAskAnswerView(...).sentence`) for success, and the lifecycle's own
`errorMessage` for provider-missing/timeout/error. `role="status"` on that
line so screen readers pick up the async result (matches the `role="status"`
pattern already used for the submitting state in `app/ce/query/ask-panel.tsx`).

**Deliberately out of scope** (named, not built): a full canvas answer
overlay ‚Äî entity chips positioned on nodes, collapsible SPARQL disclosure.
`ask-view.ts` already returns `entities`/`sparql` for that, but wiring it to
node positions is the separate de-hairball/graph-curation task the brief
explicitly excludes.

## Item 2 ‚Äî KPI strip

`canvasKpiItems()` (`components/explorer/canvas-kpi-items.ts`) now takes a
second parameter, the canvas's loaded `CytoscapeElement[]` (or `null`).
Entities/relations are derived from it using the same node-vs-edge
discriminator already established in `use-filter-panel.ts`
(`element.data.source !== undefined` ‚áí edge/relation, else node/entity).
Violations has no client-side SHACL source and stays `"Pending"`
unconditionally.

Caller: `canvas-filter-chrome.tsx` passes `adapter?.listElements() ?? null`.
No new fetch, no new hook, no threading through `use-explorer-canvas.ts` ‚Äî
`CanvasFilterChrome` only ever renders once `adapter` is loaded (gated by
`explorer-canvas.tsx`'s `{adapter && <ExplorerInteractions .../>}`), so
`listElements()` is always real data by the time it's read.

**Nuance worth flagging for QA/architect**: the old code comment and test
explicitly argued *against* ever substituting the canvas's own render count
in for entities/relations, because it's capped/filtered/
expansion-dependent (`MAX_VISIBLE_NODES`), not the true `CE-METRICS-1`
aggregate (an M2-only contract with no M1 producer). This task brief
explicitly and knowingly reverses that call for entities/relations only ‚Äî
the CONTEXT section named the current "Pending" placeholders as the thing
to fix. I kept the rationale honest rather than deleting it: the doc
comment and the progress note here both still say this is the *loaded/
rendered* count, not the true aggregate. Violations was left untouched
(still "Pending") since there's no client-side SHACL data to substitute in
regardless of this reversal.
