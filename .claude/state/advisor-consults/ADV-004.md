## Planned change reviewed

Allow the coordinator to run multiple epic lanes concurrently (cap 3) when the dependency graph's
ready-set exceeds one task: worktree per epic lane, within-epic sequential (unchanged), sibling PRs
where the dep graph shows no path between epics, coordinator-serialized shared state, per-lane
compose/port isolation, all per-task/epic/phase gates unchanged.

## Blocking items and how each is addressed (all in the committed skill wording)

1. **Ahead-of-phase clause dropped as default.** Lanes draw only from the current phase's
   ready-set; work in a later engine/phase requires explicit per-instance human opt-in. *Recorded
   opt-in:* user explicitly approved cross-engine overlap ("stage 1+2 … go", 2026-07-05, this
   session) — GE-EPIC-001 / BE root-task lanes this session run under that recorded approval.
   Every engine phase gate still fires, in order; nothing merges past an unapproved gate.
2. **State discipline written in:** `.claude/state/**` is read/written/committed only by the
   coordinator in its primary checkout (state commits ride the coordinator's current branch, as
   today); lane subagents never run `progress.sh update` and never commit `.claude/state/**` on
   lane branches; coordinator writes task summaries from lane reports.
3. **Restack topology:** with siblings open, `restack.sh` is invoked with explicit branch
   arguments matching the real dependency topology (siblings rebase onto main/parent, never onto
   each other); bare discovery mode is chain-only. Script unchanged; invocation rule codified.
4. **Ledger assignment:** each open ledger finding is assigned to at most one lane; epic-close
   remediation touches only assigned findings.
5. **Multi-lane resume/escalation/budget:** one resume point per in-progress lane (reuse its
   worktree; prune after the epic PR merges); HITL escalations serialized — a blocked lane pauses
   only itself, no new lane starts while an escalation is pending; /goal budget becomes per-lane
   60 turns with a global 3×60 cap.
