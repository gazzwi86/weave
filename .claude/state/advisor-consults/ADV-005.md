# ADV-005 — mutation gate: collapse to a single 60% bar everywhere

- **Date:** 2026-07-06
- **Change under review:** human-directed re-levelling of Weave's own mutation gate to a **single 60%
  threshold**, enforced identically by both CI tiers (per-PR unit-only + `mutation-strict`
  main-push-with-services), replacing the two-number design (per-PR 60 floor + strict 70) from
  ADV-003. Plus the doc/spec sweep of the ~70% references and the fix that unblocks the strict runner.
- **Advisor model:** fable (`advisorModel`)
- **Verdict:** APPROVE-WITH-CHANGES
- **Human authorisation:** yes — the operator explicitly directed "one 60% gate everywhere … so there
  is only one consistent PR and main-level 60% score" (AskUserQuestion, this session). This is the
  HITL sign-off for the re-levelling; the advisor consult is the governance record.

## What changed

- `mutation_gate.py`: `DEFAULT_THRESHOLD` 70.0 → **60.0**. The `MUTATION_SCORE_THRESHOLD` env override
  mechanism stays in the code but CI sets **no** override — both tiers now enforce the 60 default.
- `ci.yml`: removed the per-PR `MUTATION_SCORE_THRESHOLD: "60"` env (redundant now the default is 60);
  the `mutation` (per-PR, unit-only) and `mutation-strict` (main-push, live services) jobs both stay
  **BLOCKING** and both enforce the same 60% bar. The tiers now differ only in *depth* (unit-only vs
  services), not in the number.
- Pinning tests updated to assert `DEFAULT_THRESHOLD == 60.0` and "one bar, no per-PR override".
- Doc/spec sweep: Weave's-own mutation references moved 70 → 60 (CLAUDE.md, testing standards,
  phase-gate/implement/arch-quality/po-strategy skills, QA + tech-architect agents, spec-templates,
  weave-spec.md's own-testing bar, per-engine testing-strategy tables, dev-environment.md).

## Honest framing (advisor advisory 1 — recorded, not softened)

70-with-live-services was **never disproven**: the strict tier ran on main-push run 28725517537
(2026-07-05) and scored **77.3% vs the 70% bar — PASS** (PROJ-005). So this is **not** a capitulation
to an unreachable bar — it is a deliberate lowering of the strict main-push bar from 70 → 60, chosen
by the operator to get **one consistent number** so per-PR (unit-only, ~64.5% measured) and main
(services, ~77.3% measured) pass the *same* gate. 60 sits **below both measured realities**, so:

- main stays green; the gate is a real regression floor (a >4.5pt per-PR drop or a >17pt strict drop
  still reds the gate), not decorative;
- both tiers remain blocking; line coverage stays ≥ 80% (untouched);
- **60 is a floor, not a ceiling.** Revisit-upward trigger: when the unit-killable mutant pool grows
  or the strict runner is stably green, ratchet the bar back up (toward 70) — the ADV-003 ratchet
  rule (move up only, fix noise with evidence never a threshold drop) carries forward.

Per `git-safety.md` rule 4 this is a **legitimate human-authorised re-levelling**, not a silent gate
removal: the gate stays, blocking, at a lower but still-meaningful floor, with a recorded path back up.

## Blocking items (advisor) — dispositions

1. **Sweep the enforcement code + its pinning tests, not just docs.** DONE — `DEFAULT_THRESHOLD` and
   both `test_mutation_gate.py` / `test_ci_workflow.py` pins moved to 60 in the same change (verified:
   16 passed). Without this CI would contradict itself at the code level immediately after merge.
2. **State how `mutation-strict` actually goes green.** RESOLVED — the advisor's assumed cause (docker
   namespace contention from a parallel worktree) was **not** the blocker. The real cause of the
   recent main redness (run 28761059585) was a mutmut **collection crash**: `test_ce_perf_benchmark.py`
   (CE-TASK-008, just merged) resolved the benchmark dir via a hard-coded `parents[4]`, which breaks
   under mutmut's `mutants/` copy (one level deeper) → `ModuleNotFoundError: run_benchmark` → 0
   killed/survived → the gate correctly failed as "interrupted". Lowering 70→60 alone would **not**
   have fixed this. Fixed by replacing the fixed depth with a depth-independent upward search (verified:
   test green normally; 700 tests collect clean under the strict marker filter; resolver proven to
   find the real dir at mutmut depth). With collection unblocked, strict produces a real score
   (~77.3% measured) which clears 60.
3. **Supersede ADV-003 and PROJ-005 by annotation, not rewrite.** DONE — ADV-003 carries an appended
   "SUPERSEDED by ADV-005" note (original body intact); PROJ-005 updated to record that the strict-70
   backstop is replaced by the single-60 bar (this ADV-005 rationale) and closed.

## Carve-out (NOT changed — flagged for the operator)

The **Build Engine generated-app safety gate** (FR-029: the mutation bar Weave imposes on applications
it *generates* for customers — `build-engine.md`, `build-engine/**`, `docs/standards/secrets-scanning.md`,
and weave-spec.md's "M1 safety gates … delta-scoped mutation" line) was **left at 70%**. Rationale: it
is a *different* gate with the opposite pressure (you want generated code *more* tested, not less), it
does not gate `main` (the Build Engine is unbuilt), and silently weakening a product-quality promise is
harder to undo than to add. If the operator wants that gate at 60 too, it is a one-line follow-up sweep.
