# ADV-003 — mutation gate: two-tier (per-PR floor + deterministic strict-70)

> **SUPERSEDED by [ADV-005](ADV-005.md) (2026-07-06).** The two-number design below (per-PR 60 floor
> + strict 70) was collapsed to a **single 60% bar** enforced by both tiers, on explicit operator
> authorisation. The strict-70 backstop this record established is replaced by the single-60 bar; the
> ratchet-up-only rule carries forward. This record is retained intact as history — do not edit its
> body.

- **Date:** 2026-07-04
- **Change under review:** making the backend CI mutation gate blocking (human request), which
  surfaced that per-PR unit-only mutation is structurally capped ~62%.
- **Advisor model:** fable (`advisorModel`)
- **Verdict:** APPROVE-WITH-CHANGES

## Design endorsed

Two-tier, both blocking: per-PR CI mutation job blocks at a **60% regression floor** (unit-only,
fast); the strict **70%** bar is enforced by a deterministic **`mutation-strict`** job (main-push,
boots services, mutmut exercises the integration suite) **and** the phase gate. blocking-60 is
strictly stronger than the previous advisory (continue-on-error) 70, so this is a legitimate
altitude relocation under `git-safety.md` rule 4 — **conditional on** the strict bar also living in a
deterministic runner, not solely the model-executed phase gate.

## Advisor corrections to the loop's premises (all verified)

1. **`packages/backend/uv.lock` exists and is tracked** — deps ARE pinned; the "dep drift" variance
   argument was false. (The hook's "no uv.lock" warning is about the repo-root path, not the backend
   package.)
2. **CI already boots services** — the `integration` job runs `docker compose up -d --wait postgres
   redis oxigraph localstack`. So the strict 70% CAN run deterministically in CI; it doesn't have to
   be model-only.
3. **`docs/standards/testing-py.md` claimed a single "CI blocks at 70%"** — the two-tier design made
   that rubric line false; it had to be corrected in the same change (harness-gated doc, covered here).

## Required changes — dispositions

1. **Add a deterministic strict-70 runner.** DONE — `mutation-strict` job (main-push, services,
   `PYTEST_ADDOPTS` includes the integration suite, no threshold-lowering env → strict 70 default,
   blocking, 30-min timeout). Runtime is **unverified locally**; PROJ-005 tracks validating the first
   main-push run (may need timeout tuning or a nightly `schedule` trigger).
2. **Update the rubric doc.** DONE — `testing-py.md` now describes both tiers honestly.
3. **Rewrite `test_ci_pr_gates_pass`.** DONE — asserts real invariants: gate script invoked, no
   `continue-on-error` at job or gate-step level, `pipefail` + no `|| true`, floor ∈ [60, 70], and a
   sibling assertion that `mutation-strict` boots services + enforces the strict default.
4. **Fix `test_mutation_gate`.** DONE — `monkeypatch.delenv` for the default-threshold test, a new
   env-override test, and a cleaned subprocess env.
5. **Keep floor 60 + ratchet-up-only.** DONE — floor stays 60; PROJ-005 records the ratchet rule
   (floor moves up only; noise is fixed with evidence, never a threshold drop).

## Rejected alternatives (advisor)

- Services in the **per-PR** mutation job — blows the 10-min PR budget (mutmut × integration).
- Scoping `paths_to_mutate` to unit-killable modules — reports a prettier number by exempting the
  security-critical SQL/auth code mutation testing exists to protect. Disguised weakening.
