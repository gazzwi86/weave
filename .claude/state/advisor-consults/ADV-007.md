## Blocking items (all addressed in the implementing commit)

1. `design.md` carries an explicit write-boundary (never writes `.claude/**`, `CLAUDE.md`,
   `docs/standards/**` — it reads `docs/standards/design/*` as rubric and must not be able to
   "fix" the standard it grades against) and a minimal tool list (Read, Glob, Grep, Write; no
   Bash).
2. QA wiring wording is strictly additive: the design agent can add findings or raise severity,
   never downgrade/dedupe-away/substitute for QA Categories 15 (design-system conformance) and
   17 (ui_verify); `ui_verify.sh` exit ≠ 0 remains a hard FAIL regardless of design-agent output;
   QA still executes every category itself.
3. "UI-bearing" defined once by reference to Category 17's smart-detection criterion (task touches
   a screen/component/page) so brief-time and QA-time triggers cannot disagree; in
   `arch-task-brief` the appended Design-requirements section lands BEFORE the existing HITL
   sign-off so the fable architect + human still own the final brief.
4. `.claude/HARNESS.md` regenerated in the same commit (`harness_manifest.py generate`) to satisfy
   the pre-push `check-harness-manifest` gate.

## Advisory items (adopted)

1. Every appended requirement line cites its source (token, JTBD entry, F-D-NNN, design-doc
   section); uncited lines are advisory notes, not acceptance criteria.
2. QA-time dedupe: findings already produced by Category 15/17 are cross-referenced, not
   re-reported.
3. Graceful degradation: missing JTBD entry for a surface → flag the gap, never invent success
   criteria.
4. Name the exact skill (`arch-task-brief`).
5. HITL approval for these paths recorded as **voluntary** (rule 2 covers only the enforcement
   core); noted to avoid diluting rule 2's meaning by precedent.

## Rationale (advisor's summary)

Additive and correctly aimed; both hooks ride existing gates (`/qa`, `ui_verify`) rather than
creating or relaxing any. Design agent's write surface is spec/state content, not harness
machinery, so no generation-tier agent gains authority over its own constraints provided the
write-boundary is explicit. Real risks — silent delegation of QA's design categories, trigger
drift between hooks, manifest-parity push failure — all cheaply fixed in authoring; no redesign
needed.
