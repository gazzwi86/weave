# /anatomy — repository anatomy generator

You maintain the project's **anatomy map** — a navigable, semantic index of the codebase that
lets an agent traverse one or two files instead of grepping the whole tree. It has two layers:

- **`ANATOMY.md`** (repo root) — the top-level map: one line per area with a one-sentence purpose,
  plus a pointer to its wiki page. Think "table of contents for the whole repo."
- **`docs/wiki/<area>.md`** — one page per area: the files in that area, the key symbols
  (functions/classes/exports) in each, and a one-line semantic description of what each does and why.

`<area>` is derived from a file's path exactly as `.claude/scripts/modules/common.py::area_for_path`
does it (e.g. `apps/web/...` → `web`, `packages/core/...` → `core`, `src/api/...` → `src-api`,
`infra/...` → `infra-<sub>`). Only files under the tracked roots `apps/ packages/ infra/ src/ lib/`
have wiki pages — everything else (docs, prototypes, .claude config) is intentionally excluded.

## Arguments

`$ARGUMENTS`

- **`refresh <file1> <file2> …`** — incremental mode (this is how the pre-commit / pre-push hooks
  call you). Only the areas containing those files are touched.
- **no arguments** — full regeneration of `ANATOMY.md` and every `docs/wiki/<area>.md`.

## Procedure

1. **Determine target areas.**
   - refresh mode: map each passed file to its area via the rule above. Skip files not under a
     tracked root. If no files map to an area, **make no changes and exit** (nothing to do — this is
     the common case for docs/config/prototype-only commits).
   - full mode: discover all areas under the tracked roots (`Glob` `apps/**`, `packages/**`,
     `src/**`, `lib/**`, `infra/**`).

2. **Gather structure per area.** Prefer jcodemunch when the repo is indexed
   (`resolve_repo` → if `indexed`, use `get_file_outline` / `search_symbols` for accurate symbol
   lists). Otherwise read the files directly. Capture, per file: its purpose, and each exported/
   public symbol with a ≤1-line description of what it does. Be concise — this is a map, not docs.

3. **Write `docs/wiki/<area>.md`** for each target area using this shape:

   ```markdown
   # <area>

   _Anatomy — regenerated by `/anatomy`. Last refresh: <UTC date>._

   ## <relative/file/path>
   <one-line file purpose>
   - `symbolName(args) -> ret` — what it does
   - `ClassName` — responsibility
   ```

   **Remove every `<!-- stale: <path> -->` marker** for files you just refreshed (the
   `mark-anatomy-stale` PostToolUse hook adds these; clearing them is what marks the page fresh).

4. **Update `ANATOMY.md`.** Keep it to a single table — one row per area — and a short intro.
   In refresh mode, update only the rows for touched areas (add a row if the area is new); leave
   the rest untouched.

   ```markdown
   # Repository Anatomy

   Navigable map of the codebase. Each area links to its wiki page; open that before grepping.

   | Area | Purpose | Wiki |
   |---|---|---|
   | web | Next.js SPA — graph explorer + spec UI | [web](docs/wiki/web.md) |
   ```

5. **Keep edits minimal and idempotent.** Don't reflow unchanged sections. Don't touch files
   outside `ANATOMY.md` and `docs/wiki/`. Never fail the commit — if you cannot complete, write
   nothing and report briefly.

## Output

End with a one-line summary: which areas were refreshed (or `no tracked source files — skipped`).
