# ADR-003: A non-superuser weave_app role is required for RLS to have any effect

## Status

Accepted

## Context

PLAT-TASK-003's mandatory cross-tenant isolation requirement (AC-6) is enforced at the
Postgres layer with `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY` +
`current_setting('app.tenant_id')`-scoped policies on `workspaces`, `workspace_members`,
`settings`, and `audit_events`.

Postgres's own documentation states this plainly, but it is easy to miss until it is
verified against a real database (as done here, not assumed): row-security policies are
**never** applied to a superuser connection, regardless of `FORCE ROW LEVEL SECURITY`.
`FORCE` only changes behaviour for the table owner when that owner is *not* a superuser —
it does nothing for a superuser session. Locally, the docker-compose `postgres` service's
`weave` user (`POSTGRES_USER=weave`) is a superuser by default, and the existing migration
tooling (`db/migrate.py`) intentionally connects as that role because it needs
`CREATE ROLE`/DDL privileges the runtime role must not have.

Verified by running the tenancy migration and cross-tenant isolation test against a real
local Postgres instance (not assumed from documentation alone): a raw, unfiltered
`SELECT * FROM workspaces` issued over a `weave` (superuser) connection returns every
tenant's rows regardless of any `app.tenant_id` setting, while the identical query over a
`weave_app` (non-superuser) connection returns only the calling tenant's rows.

## Decision

- `migrations/0001_tenancy.sql` creates `weave_app` (`LOGIN NOSUPERUSER NOCREATEDB
  NOCREATEROLE NOINHERIT`) idempotently via a `DO` block, and grants it exactly
  `SELECT, INSERT, UPDATE, DELETE` on the four tenancy/settings/audit tables — no DDL, no
  `BYPASSRLS`.
- `db/pool.py`'s application connection pool (`get_app_pool`) connects as
  `POSTGRES_APP_USER` (default `weave_app`) and always sets
  `set_config('app.tenant_id', ..., is_local=true)` (`SET LOCAL` semantics, scoped to one
  transaction) before yielding a connection from `tenant_connection()`.
- `db/migrate.py` continues to connect as `POSTGRES_MIGRATION_USER` (default `weave`,
  the compose superuser) since DDL (including `CREATE ROLE`) requires it.
- The real AWS deployment (Aurora Postgres) must provision an equivalent non-superuser
  application role via Terraform/Secrets Manager — the app must never be granted the
  Aurora master/superuser credential for the same reason this ADR exists locally.

## Consequences

- Any future table holding tenant-scoped data must add its own `tenant_isolation` RLS
  policy *and* an explicit `GRANT ... TO weave_app` — RLS alone does not grant access;
  the grant plus the policy together define what's visible.
- A developer running ad-hoc `psql` queries as the `weave` superuser (e.g. for local
  debugging) will see all tenants' data with no policy enforcement — this is expected and
  matches Postgres's documented behaviour, not a bug in the policies themselves. Anyone
  debugging an isolation issue must reproduce it through `weave_app`, not `weave`.
- If a future migration ever needs to run application-level queries as part of a data
  migration (backfill), it must do so through `weave_app` (or bypass explicitly and
  document why), not the superuser connection the migration runner itself uses for DDL.
