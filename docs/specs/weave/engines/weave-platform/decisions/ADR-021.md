# ADR-021: 410 re-baseline re-seed is label-ordered, not true-recency-ordered

## Context

TASK-024's AC-3 (recent-edits collaboration widget) requires that an aged-out cursor
(`410 Gone` from the CE-EVENT-1 seq feed) re-baselines via a **CE-READ-1** "recently-updated
entities" re-seed, per contracts.md's CE-EVENT-1 retention rule.

The graph carries no per-entity modified-timestamp predicate — no PROV `generatedAtTime` (or
equivalent) is asserted per entity today. `graph_change_events` (the Postgres table backing
CE-EVENT-1) does carry per-event timestamps, but its own retention window is exactly what just
expired, so it cannot be the re-seed source either.

## Decision

The re-seed (`dashboard/coverage_gap.py::recently_updated_entities`) reuses the same real,
existing `POST /api/sparql` SELECT-only surface (no new CE endpoint), with the same query
shape as `instances/browse.py`'s existing browse query — `GRAPH`-scoped, `ORDER BY LCASE(?label)`,
bounded `LIMIT`. This is **not** ordered by true recency; it is a bounded, honestly-labelled
substitute so AC-3's "never a silent empty page" holds without fabricating a recency signal CE
does not currently produce.

## Consequences

- The 410 truncation notice ("history truncated — showing activity from now") is accurate about
  *what happened* (history was lost) but the re-seed rows are not literally "most recently
  touched" — they are a stable, bounded sample of existing entities.
- Upgrade path: once CE asserts a per-entity modified-timestamp predicate (e.g. `prov:generatedAtTime`
  on the entity's latest revision), swap the `ORDER BY` clause — no shape/contract change needed
  on the platform side.
- Follow-up flagged for whoever owns CE's provenance model next, not invented here (same posture
  as ADR-019's billing `group_by` gap).
