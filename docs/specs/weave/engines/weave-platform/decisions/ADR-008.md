# ADR-008: `GET /api/search`'s `workspace_id` is optional, falling back to the caller's active session workspace

## Status

Accepted

## Context

PLAT-TASK-005's API contract is `GET /api/search?q={query}&workspace_id={wid}`. The
task brief does not say what happens when `workspace_id` is omitted, and M1 has
no workspace-switcher UI for the command palette to source one from — the
Cmd+K palette has to work today against whichever workspace the user is
already in.

`/api/sparql` already solved the identical shape of problem
(`_resolve_named_graph`): resolve an explicit `workspace_id` if given,
otherwise fall back to the session's active workspace; 404 on a
missing/foreign workspace before the role check so a non-member can't
distinguish "doesn't exist" from "exists, you can't see it" (ledger item 3 --
membership-authz parity with settings/sparql).

## Decision

`_authorize_search` in `routers/search.py` mirrors that exact resolution
order:

1. `workspace_id = requested_workspace_id or get_active_workspace(tenant_id, sub)`.
2. No active workspace and none supplied -> `400 no_active_workspace` (there is
   nothing to search).
3. Workspace looked up by id, tenant-scoped -> `404 workspace_not_found` if
   absent (covers both "doesn't exist" and "exists in another tenant").
4. `enforce_workspace_role(min_role="read")` -> `403` for a real member with an
   insufficient role, and for a *tenant* member with no membership row in that
   workspace at all (`enforce_workspace_role` resolves no role the same as too
   low a role).

Only after all four gates pass does the SPARQL search run, scoped to
`workspace.named_graph_iri`.

## Consequences

- The command palette needs no workspace-picker state in M1; it searches
  "wherever I currently am," matching every other workspace-scoped surface
  (settings, SPARQL console).
- A tenant member who is not in the target workspace gets `403`, not a
  filtered/empty result set that could be mistaken for "no matches" — proven
  by `test_search_cross_tenant_returns_403` and the sibling non-member test in
  `tests/integration/test_search_tenancy.py`.
- When M2 adds a workspace switcher / multi-workspace search, this ADR's
  fallback path stays as the zero-param default; explicit `workspace_id` keeps
  working unchanged for that case, so no follow-up migration is needed here.
