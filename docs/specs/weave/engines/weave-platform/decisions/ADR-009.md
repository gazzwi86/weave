# ADR-009: no configured budget cap means unmetered / fail-open

## Status

Accepted

## Context

PLAT-TASK-008 AC-2 requires rejecting an AI call once `consumed >= cap`. That
comparison is only meaningful once a cap has been set at some scope in the
PLAT-SETTINGS-1 cascade (company/domain/workspace/project). The brief gives
no default cap value and no test exercises "a tenant that has never called
`PUT /api/billing/caps`". Two options exist: (a) fail closed -- refuse every
AI call until a cap is explicitly configured, or (b) fail open -- treat the
absence of any cap as "no budget limit configured, allow the call."

## Decision

`enforce_budget` catches `SettingNotFound` from `resolve_setting` and returns
immediately (no exception, no notification, no metering skipped). Usage is
still recorded after the call regardless of whether a cap exists --
metering and enforcement are independent (AC-3/AC-4 do not depend on AC-2).
`get_usage_summary`'s `cap_utilisation_pct` mirrors this: no cap resolves to
`0.0`, not an error.

Fail-open was chosen because a brand-new tenant/workspace with no cap set
yet must not be blocked from using the product on day one -- that would be a
surprising, support-ticket-generating default with no corresponding UI
prompt in M1 to tell the caller why. Billing/finance can configure a cap
once cost control becomes a concern; until then, usage is metered but not
gated.

## Consequences

- A tenant that never calls `PUT /api/billing/caps` has unlimited AI usage,
  metered but unenforced. This is the correct default for M1's thin-proof
  scope; a future milestone may want a platform-wide default cap applied at
  tenant-creation time -- tracked as a follow-up, not built here.
- `GET /api/billing/usage`'s `cap_utilisation_pct` is `0.0` (not `null` and
  not an error) when no cap is configured, keeping the response schema total
  rather than optional.
- Proven by `test_enforce_budget_no_cap_configured_allows_call` (unit) --
  asserts no exception raised and the downstream AI-call mock is invoked.
