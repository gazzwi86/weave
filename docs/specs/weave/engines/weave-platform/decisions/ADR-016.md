# ADR-016: Connector sync scheduling is a DB-driven due-poller worker, not EventBridge

## Status

Proposed — Platform v1 architecture pass, 2026-07-08.

## Context

E7-S1 captures a per-connector **sync frequency**; something must fire syncs on that
cadence, in every environment. Constraints: Law F (tests never touch real cloud — the
scheduler must run under the local stack: Postgres, LocalStack, mock connector servers);
the M1/M2 platform already runs Aurora Postgres with proven RLS and has no queueing or
scheduling infrastructure to inherit; sync volume at v1 is small (7 connector types ×
tenants, minutes-level frequencies — not a fan-out problem).

## Decision

1. **Schedule state lives on the connector-config row**: `sync_frequency` (interval) and
   `next_sync_at` (timestamptz), in the existing Aurora database under the M1 RLS family.
2. **A dedicated sync worker** (same backend codebase, separate entrypoint/container) loops:
   claim due rows with `SELECT … WHERE next_sync_at <= now() FOR UPDATE SKIP LOCKED`, run
   the sync (ADR-015 pipeline), write the sync-run record + health outcome, set
   `next_sync_at = now() + sync_frequency`. `SKIP LOCKED` makes multiple worker instances
   safe with zero coordination infrastructure; v1 deploys one instance (Fargate service).
3. **Manual "sync now"** (`POST /api/connectors/{type}/sync`) sets `next_sync_at = now()`;
   the worker picks it up on the next tick (default tick 15 s, tunable). No second code path.
4. **No EventBridge, no APScheduler, no SQS.** The poller is plain SQL + a loop — it runs
   identically under docker-compose, CI, and Fargate.

## Consequences

**Positive:** the whole scheduling story is testable with a Postgres container (Law F holds
with nothing mocked); one mechanism serves scheduled and manual syncs; crash recovery is
free (an unfinished claim releases its lock; `next_sync_at` is unchanged until success is
recorded — the run re-fires).

**Negative:** scheduling precision is bounded by the tick (±15 s — irrelevant at
minutes-level frequencies); a long-running sync of one connector occupies a worker slot —
per-connector sync timeout (default 10 min, tunable) bounds it.

**Neutral / upgrade path:** if per-tenant connector counts ever outgrow a worker pool,
move firing to EventBridge Scheduler targeting the same claim-and-run code — the
`next_sync_at` contract is unchanged, so the swap is transport-only.

## Alternatives considered

- **EventBridge Scheduler per connector** — rejected for v1: real-cloud dependency in the
  dev/test loop (LocalStack emulation of Scheduler is partial), IaC churn per connector
  config change, and it solves a fan-out scale problem v1 does not have.
- **APScheduler in the API process** — rejected: schedules held in process memory need
  rebuild-on-deploy and leader election; the API tier (Lambda-compatible) is the wrong home
  for a long-lived loop. The DB poller keeps state where it already is.
- **SQS delay queues** — rejected: delay cap (15 min) forces re-enqueue bookkeeping that
  the `next_sync_at` column already does for free.
