# ADR-017: v1 write-back is Atlassian + ServiceNow only; conflict policy is check-then-write, reject on drift

## Status

Proposed — Platform v1 architecture pass, 2026-07-08. Resolves OQ-07.

## Context

OQ-07 left open which of the 7 connectors support write-back in v1 (Atlassian + ServiceNow
already confirmed by the PRD) and the conflict-resolution policy detail. FR-033 fixes the
mechanics: idempotency key per write, bounded retry (default 3, tunable) on target 4xx/5xx,
then `PLAT-NOTIFY-1` + `PLAT-AUDIT-1`; out-of-band conflicts are rejected and surfaced,
never silently overwritten.

The other five: Snowflake / Databricks / AWS / Azure Data Lake are analytical stores —
Weave-initiated writes there have no v1 use case and a large blast radius. Slack's write
surface is notification delivery, which is `PLAT-NOTIFY-1` channel scope (EPIC-006), not
graph write-back.

## Decision

1. **Write-back allowlist (v1): `atlassian`, `servicenow`.** Connector config validates
   `sync_direction`: `write` or `bidirectional` on any other connector type is rejected at
   save (422, message names the allowlist). The allowlist is engine data, so widening it
   post-v1 is a config + driver change, not a contract change.
2. **Conflict policy — check-then-write, reject on drift:** every ingested record stores the
   source's change marker (Jira `updated`, ServiceNow `sys_updated_on`) alongside the
   external-id (ADR-015). A write-back re-reads the target record, compares the marker to
   the stored one; on mismatch the write is **rejected**, surfaced via `PLAT-NOTIFY-1` and
   recorded to `PLAT-AUDIT-1` with both markers. No merge, no last-writer-wins, no
   overwrite. Re-sync then ingests the remote change; the caller re-issues against fresh
   state.
3. **Idempotency + retry per FR-033:** each write-back carries an idempotency key
   (`sha256(tenant_id, connector_type, external_id, operation_hash)`; sent natively where
   the target supports it, else used for local at-most-once suppression of the retried
   batch). Target 4xx/5xx → bounded backoff retry (default 3 attempts, tunable), then
   `connector-degraded` `PLAT-NOTIFY-1` + `PLAT-AUDIT-1` failure record — fail closed and
   observable, per the EPIC-007 epic AC.
4. **Slack is read/notify-only as a connector**; its outbound surface is the EPIC-006
   notification channel, out of write-back scope.

## Consequences

**Positive:** the risky surface (writing into customer systems) is confined to the two
connectors with a confirmed use case and native change markers; the conflict rule is
mechanically testable (set marker, mutate remote fixture, assert reject + notify + audit);
no false "connected" or silent overwrite path exists.

**Negative:** check-then-write is not atomic — a remote change landing between check and
write can still race. Both targets tolerate this (Jira/ServiceNow updates are
last-writer-wins server-side and the subsequent re-sync surfaces the drift); recorded
honestly rather than engineered around at v1 volume.

**Neutral:** widening the allowlist (e.g. Databricks post-v1) requires a driver `write()`
implementation + allowlist entry + the same conflict tests — no architectural change.

## Alternatives considered

- **All 7 writable in v1** — rejected: no use case for warehouse write-back, five extra
  fail-closed audit surfaces for the security review, contradicts the PRD's "bidirectional
  v1 for Atlassian + ServiceNow families".
- **Last-writer-wins on conflict** — rejected: FR-033 explicitly forbids silent overwrite.
- **Server-side compare-and-set only** (rely on target ETags) — rejected as the sole
  mechanism: ServiceNow/Jira conditional-update support is inconsistent across resources;
  the stored-marker check works uniformly and degrades to the same reject behaviour.
