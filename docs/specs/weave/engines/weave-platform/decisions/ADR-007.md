# ADR-007: settings writes gate on the author role, not admin

## Status

Accepted

## Context

PLAT-TASK-004 (AC-3: "every endpoint checks role") required closing a
within-tenant lateral-movement gap: `get_setting_route`, `set_setting_route`,
`run_sparql_route`, and `switch_workspace_route` authenticated the caller and
matched their tenant, but never checked workspace membership or role. A tenant
member with **zero** membership in the target workspace could read/write its
settings, run SPARQL against it, and switch into it.

The invite/revoke member routes (fixed earlier in this task) gate on `admin`
because they mutate *who has access*. The four routes above are not
access-control mutations, so reusing `admin` would be wrong in both
directions: too strict (an ordinary contributor could not change a workspace
setting they legitimately own) and conceptually muddled (a config write is not
a membership change).

The `ROLE_RANK` hierarchy is `read (0) < author (1) < publish (2) < admin (3)`.

## Decision

Each workspace-scoped route gates on the minimum role that matches the nature
of the operation, enforced by `enforce_workspace_role` against the workspace
the operation actually targets (derived via `workspace_of` from the
scope/context IRI for settings and SPARQL, from the target id for switch — not
a blanket path parameter):

| Route | Ceiling | Rationale |
|---|---|---|
| `GET /api/settings/{key}` | `read` | Reading a resolved config value is a read. |
| `PUT /api/settings/{key}` | `author` | A settings write is a config change any contributor may make; not a membership change. |
| `POST /api/sparql` | `read` | Querying the workspace graph is a read. |
| `POST /api/workspaces/{id}/switch` | `read` | Switching in requires being a member; `read` is the membership floor. |

A non-member (no `workspace_members` row at all) is rejected the same as an
insufficient role — `enforce_workspace_role` resolves no role and returns 403.

Company- and domain-scoped settings have no workspace segment; `workspace_of`
returns `None` for them and they remain tenant-match-only (no workspace role to
check). This is intentional and not an escalation path: they are still
RLS-isolated by `tenant_id` and guarded by `_require_own_tenant_scope`.

## Consequences

- A read-only member can read settings and query, but cannot write settings
  (403) — verified by `test_read_role_member_cannot_write_settings_author_ceiling`.
- A member of workspace A operating on workspace B they do not belong to is
  rejected (the guard resolves B's role, not A's) — verified by
  `test_workspace_role_enforced_against_the_requested_workspace_not_another`.
- If a future requirement makes settings privileged (e.g. billing or security
  config that only workspace admins should change), that key class needs a
  higher ceiling than the blanket `author` here — worth a follow-up ADR keyed
  on setting sensitivity rather than route.
