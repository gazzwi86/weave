# ADR-015: Connector ingestion maps to the graph via declarative per-connector profiles over CE-WRITE-1

## Status

Proposed — Platform v1 architecture pass, 2026-07-08.

## Context

E7-S3 (FR-033) requires connector data to reach the graph **only** through `CE-WRITE-1`
validated operations, under a connector-scoped principal, committing only on SHACL pass.
Two constraints shape the mapping layer:

1. **Ontology rule (`.claude/rules/ontology-standards.md`):** Weave ships the BPMO grammar,
   not a populated client taxonomy. The canonical kind/relationship set is served by
   `GET /api/ontology/types` (CE-READ-1) — a connector must not hard-code domain classes.
2. **Idempotent re-sync:** connectors poll on a schedule; the same Jira issue or ServiceNow
   record must update its existing node, never mint a duplicate. `CE-WRITE-1` dedups on
   case-insensitive `label`+`kind` and accepts an idempotency key, but labels can change at
   the source, so label-dedup alone is insufficient.

## Decision

1. **Each connector driver ships a declarative mapping profile** (YAML, versioned with the
   driver, one per connector type): source record selector → target kind (by ontology-type
   IRI or SKOS label), field → property mappings, relationship rules (source field → edge
   kind + target node lookup). Profiles are data, reviewed like code.
2. **Kinds are resolved at sync time** against `GET /api/ontology/types`. A profile entry
   whose kind is absent from the tenant's ontology is skipped and counted in the sync-run
   report (a warning, not a batch failure) — the grammar stays open, nothing is hard-coded.
   The skipped count is **surfaced through the health read API** as a dimension; sustained
   skips (same kind skipped over N consecutive runs, default 3, tunable) read as `degraded`,
   and a later resync recovers the records once the kind is added (contract amendment,
   `PLAT-CONNECTOR-1`).
3. **Ingestion emits `CE-WRITE-1` op batches** (`add_node`/`update_node`/`add_edge`) under
   the connector-scoped agent principal (`PLAT-IDENTITY-1`). Every ingested node carries a
   stable **external-ref property**
   (`weave:externalId = "<connector_instance_handle>:<source_id>"`, parse first-colon-only) —
   scoped to the connector **instance handle** (unique per tenant; defaults to the connector
   type at v1, where each type has one instance), never bare type, so two future connectors of
   the same type (e.g. two Jira sites) cannot collide. No `tenant_id` in the value — the graph
   is the tenant boundary. Re-sync resolves external-id → existing node IRI first (CE-READ-1
   lookup) and emits `update_node`; only unseen external-ids emit `add_node`.
4. **Per-batch idempotency key** = `sha256(tenant_id, handle, batch_content_hash)` where
   `batch_content_hash` is over the canonicalised source records (ids + change markers) —
   **content-stable, never `sync_run_id`**. A run-scoped key gives ZERO crash recovery (the retry
   run mints a new `sync_run_id` → new key → double-apply); a content-stable key lets a
   crashed-and-retried batch replay into `CE-WRITE-1`'s 24 h idempotency window and return the
   original `201`. Same key + different payload → `409` (batch failure + audited). (Matches
   TASK-018 AC-2 — the authoritative statement.)
5. **SHACL is the sole commit gate.** A `422` from `CE-WRITE-1` fails that batch only; the
   sync run records the violations, emits `PLAT-NOTIFY-1` (`connector-degraded` taxonomy
   family) + `PLAT-AUDIT-1`, and continues with the next batch. The platform never writes
   the graph store directly.

## Consequences

**Positive:** one ingestion engine + seven data profiles instead of seven bespoke pipelines;
the security/ontology review audits profiles, not transform code; adding a connector's
mapping needs no engine change; the FR-033 epic AC ("platform never writes the graph
directly") holds by construction.

**Negative:** profiles express only structural mapping — computed/derived fields need a
named transform function registered in the engine (kept to a small whitelist); deep
per-tenant mapping customisation is out of v1 scope (profiles are per connector type, not
per tenant) — recorded as the post-v1 upgrade path.

**Neutral:** external-id lookup adds one CE-READ-1 query per batch; batch size (default 100
ops, tunable) bounds it.

## Alternatives considered

- **Code-per-connector transforms** — rejected: seven divergent pipelines to audit,
  and the path of least resistance leads to hard-coded domain classes (ontology-rule
  violation).
- **LLM-mapped ingestion** (model proposes ops per record) — rejected for v1:
  nondeterministic output under a SHACL gate means unreproducible sync failures, and
  per-record token cost lands inside `PLAT-BILLING-1` caps with no user-visible value over
  declarative profiles. Viable later as a profile-*authoring* assistant, not a runtime path.
- **Raw staging store + user-driven later mapping** — rejected: defers the v1 demonstrable
  outcome (Atlassian data in the graph, SHACL-committed) and adds a second store with its
  own isolation story.
