# ADR-005: Agent-token minting resolves tenant via a narrow SECURITY DEFINER function

## Status

Accepted

## Context

PLAT-TASK-004's `POST /api/auth/agent-token` contract is fixed as
`{sts_token, workspace_id}` -- no `tenant_id` field. But every tenancy table
(`workspaces` included, per ADR-003) enforces `FORCE ROW LEVEL SECURITY` with
a policy scoped on `current_setting('app.tenant_id', true)`, and the
non-superuser `weave_app` runtime role the app always connects as (ADR-003)
gets zero rows back for *any* query until `app.tenant_id` is already set for
that transaction.

Agent-token minting needs exactly one fact before it can do anything else:
which tenant owns `workspace_id`, so it can set `app.tenant_id` correctly for
every subsequent tenancy-scoped write (the `principals` upsert, the audit
event). There is no tenant context to set yet -- that's precisely the value
being looked up.

## Decision

Add `resolve_workspace_tenant(p_workspace_id UUID) RETURNS TEXT`, a
`SECURITY DEFINER` SQL function (`migrations/0002_identity.sql`) that
returns only the `tenant_id` column for one workspace row. `SECURITY
DEFINER` runs the function body with the *owner's* privileges -- the
migration/admin role, which is a real Postgres superuser locally (ADR-003)
and therefore already bypasses RLS the same way any other superuser session
does. `weave_app` is granted `EXECUTE` on the function only, never
broadened table access.

This is the narrowest possible RLS carve-out: it can only ever answer "which
tenant owns this workspace_id", never list workspaces, never return any
other column, never accept a tenant_id to widen the query.

## Consequences

- `identity/registry.py`'s agent-mint path calls
  `SELECT resolve_workspace_tenant($1)` once, then opens a normal
  `tenant_connection(tenant_id)` (RLS back in force) for every subsequent
  write in the same request.
- In a real deployment (Aurora), the migration/admin role may **not** be a
  superuser the way local Postgres's `weave` is -- `SECURITY DEFINER` alone
  does not bypass RLS for a non-superuser owner unless that owner is also
  exempted (e.g. via `BYPASSRLS` or being the table's owner without `FORCE`).
  This needs a follow-up decision when Aurora provisioning lands (out of
  scope for this task, which is bound by Law F to local Postgres only) --
  flagging for architect/QA review, same posture as ADR-001's Cognito-floor
  follow-up.
- Any future "look up X before tenant is known" need should reuse this same
  narrow-function pattern rather than granting `weave_app` broader
  superuser-adjacent access.
