# ADR-022: Pin operates on already-persisted widget rows

## Context

TASK-014's brief pseudocode specifies `POST /api/dashboard/widgets { spec, column_span?,
position? }` inserting a brand-new `widget_instances` row from a client-held spec, framed as the
step that first persists a generated widget server-side.

By the time TASK-014 was picked up, TASK-011 (`generate.py::generate_widget_stream`) already
inserts a `scope='user'` row via `store.insert_generated_widget` unconditionally on every
successful generation (and TASK-013's refine path updates that same row). So a freshly-generated
widget is already a durable, cross-device-visible `widget_instances` row from the moment
generation succeeds — there is no client-only, not-yet-persisted widget state left for "pin" to
promote to the server. Building the brief's endpoint literally would insert a second row from the
same spec, duplicating every generated widget on first pin (confirmed by tracing
`generate_widget_stream`'s success path before writing any test).

## Decision

`POST /api/dashboard/widgets/{widget_id}/pin` acts on an **existing** `scope='user'` row (owner-only,
404 otherwise, same IDOR-safe shape as the delete/patch routes):

- clears `suggested` if set (AC-6 "pinning clears Suggested state" — covers the starter/suggested
  tiles from TASK-010's `ensure_user_starters`)
- writes the `dashboard.widget.pinned` audit event in the same transaction (AC-2), distinct from
  `dashboard.widget.generated`/`.refined` (TASK-011/013) — pin is an explicit user action, not an
  automatic side effect of generation
- is a no-op on `column_span`/`spec`/`position` (already set at insert time; a later PATCH covers
  resize, `PATCH /widgets/order` covers reorder)

`DELETE /api/dashboard/widgets/{widget_id}` (unpin, existing TASK-010 route) gains the
`dashboard.widget.unpinned` audit write in the same transaction — TASK-010 shipped this route
without any audit call, which was the real gap for AC-2's "unpin" half.

## Consequences

- No duplicate-row bug: pin never re-inserts from `spec`.
- The `column_span`/`position` fields in the brief's `POST /widgets` body are not built as a
  creation path; `PATCH /widgets/{id}` (resize) and `PATCH /widgets/order` (reorder) already cover
  post-creation adjustment per the brief's own API Contracts section.
- A future "compose a widget with no generation step" flow (not in scope for M2) would need its own
  creation endpoint — not this one.
