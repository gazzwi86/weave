# ADR-019: Canonical role slugs join ROLE_RANK rather than replacing the legacy 4-tier vocabulary

## Status

Accepted -- engineer decision during PLAT-V1-TASK-030, no human course-correction received.
Flagged in TASK-026's progress summary as "027/030 blocked on workspace_admin role-slug
remediation" -- this ADR is that remediation, scoped to what TASK-030 actually needs.

## Context

`rbac.py`'s `check_role` treats any role string not present in `ROLE_RANK` as insufficient
for every gate, never a `KeyError` but never granted either. Before this task `ROLE_RANK` had
exactly four entries: `read` (0), `author` (1), `publish` (2), `admin` (3) -- the vocabulary
`workspace_members.role` has used since PLAT-TASK-003/004.

TASK-030 AC-3 requires the Settings -> Members invite form's role selector to offer the 10
canonical in-tenant roles from `weave-platform.md` ("Canonical human roles"): Company admin/
owner, Enterprise architect, Business analyst/SME, Data steward/data engineer, Brand/content
owner, Compliance/risk officer, Engineer/developer, Ops/SRE, Automation author, Viewer/
stakeholder. TASK-027 already established the slug convention for two of these
(`workspace_admin`, `compliance_officer`) in `can-suppress.ts`, with an explicit code comment
that the session-claims role field "still carries the pre-realignment admin/author pair" and
that its gate is "written against the spec's canonical role vocabulary so it needs no rework
when that lands."

Left alone, wiring AC-3's selector to submit these slugs as `InviteMemberRequest.role` would
have silently broken every workspace-scoped route for any member invited through the new UI --
`require_workspace_role` gates invite/revoke/settings/sparql/etc., and none of the 10 canonical
slugs were recognised by `check_role`.

## Decision

Add the 10 canonical slugs to `ROLE_RANK` as additional keys, ranked against the closest legacy
tier rather than inventing a finer-grained 10-level hierarchy:

| Canonical slug | Rank | Matches legacy tier |
|---|---|---|
| `viewer` | 0 | `read` |
| `automation_author`, `ops_sre`, `engineer`, `brand_content_owner`, `data_steward`, `business_analyst_sme` | 1 | `author` |
| `enterprise_architect`, `compliance_officer` | 2 | `publish` |
| `workspace_admin` | 3 | `admin` |

Both vocabularies coexist permanently in the same dict. A workspace created before this task
(rows still holding `"admin"`/`"author"`/`"read"`) and a member invited after it (rows holding
`"workspace_admin"`/`"engineer"`/...) both rank correctly against the same
`require_workspace_role(min_role)` gates, with no migration of existing `workspace_members` rows
and no branching in `check_role` itself.

## Alternatives considered

- **Migrate existing rows to canonical slugs.** Rejected: no deployed environments yet per
  `0001_tenancy.sql`'s own comment on a different table, but this task's brief scopes no schema
  migration (0089/0090 slots were reserved, unused -- no column changes needed), and a data
  migration is a bigger, riskier change than a dict addition for a "should have" task.
- **Give canonical roles their own finer-grained rank scale (0-9).** Rejected: no route in the
  codebase currently discriminates below the 4 legacy tiers (`read`/`author`/`publish`/`admin`
  are the only `min_role` values ever passed to `require_workspace_role`), so a finer scale adds
  complexity with no consuming gate -- YAGNI.
- **Keep the invite selector's stored value as one of the 4 legacy tiers, with the 10 canonical
  names as display-only labels.** Rejected: TASK-030 AC-6 (and TASK-027's `can-suppress.ts`) needs
  to distinguish `compliance_officer` from `enterprise_architect` for notification-preference
  defaults and the `audit.chain.invalid` non-suppressible gate -- both rank as `publish` (2) under
  the legacy scale, so collapsing them loses information both TASK-027 and TASK-030 already
  depend on.

## Consequences

- `require_workspace_role`/`enforce_workspace_role`/`check_role` need no changes -- purely a data
  addition to the dict they already read.
- Legacy-seeded workspaces (`_grant_creator_admin_membership`'s `role="admin"` bootstrap,
  `db/seed_demo.py`) are unaffected and keep working exactly as before.
- Future routes that need a canonical-slug-only gate (e.g. "compliance officers only") should
  compare `resolve_workspace_role(...)`'s return value against the specific slug set, not against
  `ROLE_RANK`'s numeric rank -- rank alone can't distinguish `compliance_officer` from
  `enterprise_architect` (both rank 2).
