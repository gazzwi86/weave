# ADR-019: token-spend binding (S3) uses M1's workspace-scoped billing summary, not PLAT-BILLING-1's group_by dimension

## Status

Accepted — engineer decision during PLAT-V1-TASK-016, no human course-correction received.

## Context

TASK-016's S3 story ("token-spend") binds the dashboard's spend widget to `PLAT-BILLING-1`.
`contracts.md`'s canonical text for that contract documents an optional `group_by=engine|user|project`
query parameter for the usage-summary endpoint. The M1 implementation
(`src/weave_backend/routers/billing.py` / `billing/usage.py::get_usage_summary()`) only supports a
workspace-scoped total — there is no `group_by` parameter anywhere in the M1 code.

TASK-016's brief scopes new upstream calls narrowly (CE-READ-1 coverage_gap, CE-VERSION-1 lag,
CloudWatch GetMetricData, `GET /api/billing/usage`, PLAT-SETTINGS-1/IDENTITY-1 RBAC, PLAT-AUDIT-1
S11 feed only) and does not list "extend the billing router with grouping" as in-scope work.

## Decision

`_token_spend` (`dashboard/bindings.py`) calls the existing `get_usage_summary()` as-is (workspace
total, no grouping) and renders that single scalar/series for S3. The `group_by` dimension from the
contract text is not implemented here.

## Consequences

**Positive:** no speculative billing-router surface added; the binding only calls code that already
exists and is already tested.

**Negative:** the dashboard's token-spend widget cannot currently break spend down by engine/user/
project, only show the workspace total — a narrower view than the contract's documented grouping
implies is available.

**Follow-up:** if a future story needs per-engine/per-user/per-project spend breakdown on the
dashboard, `routers/billing.py` needs a `group_by` parameter added first — that is new billing-engine
work, not a dashboard-binding change, and should be scoped as its own task.

## Alternatives considered

- **Add `group_by` support to `routers/billing.py` inline as part of this task** — rejected: out of
  TASK-016's stated scope (a binding-layer task, not a billing-router task), and risks duplicating
  or conflicting with however the billing engine's own roadmap eventually implements grouping.
