# ADR-012: Generative widgets stream over a custom SSE endpoint on the Platform API

## Status

Accepted — human decision 2026-07-08 (M2 plan approval, relayed by coordinator). Resolves OQ-02.

## Context

E1-S1/FR-002/FR-003 require: prompt → streamed widget, header/skeleton within 1 s (tunable),
fully rendered ≤ 5 s p95. Three candidate patterns were named in OQ-02: Vercel AI SDK
`streamUI` (RSC streaming), Next.js server actions, or a custom streaming endpoint.

Two facts decide it:

1. **The AI governance path already lives in the Python Platform API.** Budget hard-reject
   *before any AI API call* (FR-035, M1 TASK-008), per-token metering on a never-dropped queue
   (`PLAT-BILLING-1`), audit attribution (`PLAT-AUDIT-1`), and two-tier model routing (Model
   Router container, `architecture.md` L2) are all M1 platform machinery. Any pattern that
   issues LLM calls from the Next.js runtime creates a **second AI call site** that must
   re-implement all four — or silently bypass them.
2. **The pattern is declarative (FR-005).** Intent maps to exactly one component from a finite
   library; nothing free-form is ever rendered. What streams is therefore a widget *spec* +
   *data rows*, not UI. RSC component streaming solves a problem this design has already
   ruled out.

## Decision

1. **One endpoint:** `POST /api/dashboard/widgets/generate` on the FastAPI Platform API returns
   `text/event-stream` (SSE). Refine (FR-007) and change-visualisation (FR-006) reuse the same
   event grammar (refine passes the prior widget spec + delta prompt; change-viz is a pure
   client re-render, no call at all).
2. **Typed event grammar** (exact schemas in `tech-spec/m2-delta.md`):
   - `spec` — resolved intent: `component_type`, title, data-source contract ID(s), bindings,
     column span. Emitted first; the client renders the skeleton/header from it (the ≤ 1 s
     target is measured to this event).
   - `data` — result rows/points, chunked.
   - `done` — terminal success (carries token count for the metering receipt).
   - `error` — terminal, named state: `budget-cap` (E1-S1 mid-stream halt + rollback),
     `unsatisfiable` (FR-004), `provider-503` (FR-003), `source-unavailable`,
     `source-not-ga` (FR-015). Never a blank or fabricated result.
3. **LLM calls happen only in the Model Router**, invoked by the endpoint *after* the budget
   gate passes; tokens are metered on the existing `PLAT-BILLING-1` queue; the generation is
   audited with the acting principal.
4. **The SPA consumes SSE with one client hook** (native `EventSource`/fetch-stream — no new
   dependency) and renders from the component library. No Vercel AI SDK on the widget path; no
   RSC streaming; the dashboard remains a client component fed by the API.
5. **Deployment constraint (binding on arch-delivery):** SSE needs response streaming end to
   end. The dashboard-generate route runs on **Fargate behind ALB** (streams natively). If the
   Platform API otherwise deploys to Lambda, this route must use Lambda response streaming or
   be pinned to the Fargate deployable — buffered proxies that break SSE are a spec violation.

## Consequences

**Positive:** one AI governance path (budget/metering/audit/model-routing) — no duplication, no
bypass; component library stays wholly client-owned; the event grammar is contract-testable
(Plugin Law B) without a browser; works identically for generate and refine.

**Negative:** we own ~one SSE endpoint + one client hook (reconnect/abort handling) instead of
using a packaged SDK; streaming infra constraint above must be honoured at deploy time.

**Neutral:** if a future surface genuinely needs server-driven UI composition, `streamUI` can be
revisited *behind* the Platform API gate — nothing here forecloses it.

## Alternatives considered

- **Vercel AI SDK `streamUI` (RSC)** — rejected: LLM calls move into the Next.js server runtime
  → second AI call site re-implementing FR-035 budget gate, PLAT-BILLING-1, PLAT-AUDIT-1; and
  it streams React trees when the declarative pattern only needs spec + data.
- **Next.js server action proxying the Platform API** — rejected: adds a hop and an ownership
  seam for zero benefit; server-action streaming ergonomics are poor, and the action would be a
  thin pass-through that still terminates SSE awkwardly.
- **WebSocket** — rejected: bidirectional transport for a one-way stream; connection management
  on Lambda/Fargate + ALB is heavier than SSE for no functional gain.
