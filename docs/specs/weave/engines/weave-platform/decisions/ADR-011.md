# ADR-011: Ollama runs natively on the host, not as a Docker service

## Status

Accepted — human decision 2026-07-05 (gazzwi86).

## Context

The local-first dev stack ([`dev-environment.md`](../../../dev-environment.md)) routes small-model
work to **Ollama** (provider abstraction: Ollama / Bedrock / Anthropic). Until now `docker-compose.yml`
shipped an `ollama` container started by default with the rest of the stack (`make up`).

The primary development platform is Apple-Silicon macOS. **Docker Desktop on macOS runs Linux
containers inside a lightweight VM that has no access to the Apple-Silicon GPU (Metal).** There is
no Apple-GPU passthrough to containers, and none is on Docker's or Apple's roadmap — it is an
architectural limitation, not a bug to wait out. (NVIDIA passthrough via `nvidia-container-toolkit`
exists, but only on Linux hosts.)

Consequences of the in-container Ollama on Mac:

- Inference falls back to **CPU-only**, typically **5–20× slower** than native for a 7B+ model —
  unusable for interactive dev.
- It duplicates a daemon most Mac developers already run natively.

Two signals show the container was already vestigial:

- The harness drift-detector (`.claude/scripts/modules/stop.py:330`) already POSTs to
  **`http://localhost:11434`** — the *native host* daemon, never the container.
- No backend/frontend code wires the compose `ollama` service; the model-routing provider
  abstraction is specified but not yet built. The container was a placeholder.

## Decision

1. **Native host Ollama is the dev default.** Developers run Ollama natively (`brew install ollama`
   or the `.app`), which uses Metal for full Apple-Silicon GPU acceleration. Code and the harness
   reach it at `http://localhost:11434`; from inside other containers it is `host.docker.internal:11434`.
2. **The compose `ollama` service moves behind an opt-in `ollama` profile.** `make up`
   (`docker compose up -d`) no longer starts it. It remains available for Linux/CI hosts with no
   native Ollama via `docker compose --profile ollama up`.
3. **The default-stack integration check drops ollama.** `test_local_stack.py`'s `SERVICES` no
   longer lists `ollama`, matching what `docker compose up` actually starts. `conftest.py` and
   `test_ci_workflow.py` already excluded it.

The provider abstraction (Ollama / Bedrock / Anthropic) and tiered model routing are **unchanged** —
this ADR changes only *how the Ollama daemon is hosted in dev*, not the architecture that consumes it.

## Consequences

**Positive:** usable inference speed on Mac (Metal, not CPU); one fewer container in the default
`make up`; no duplicate daemon; behaviour now matches the harness's existing `localhost:11434`
assumption.

**Negative:** developers must install Ollama natively as a one-time step (documented in
`dev-environment.md`) — it is no longer bootstrapped by `make up`. Linux/CI users needing a
containerised Ollama must pass `--profile ollama` explicitly.

**Neutral / parity:** the cloud path (Bedrock) is unaffected; the local↔cloud provider-abstraction
parity the dev-environment doc requires still holds (native-local Ollama ↔ Bedrock).

## Alternatives considered

- **Keep the container, started by default** — rejected: CPU-only on Mac, unusably slow, and
  duplicates the native daemon developers already run.
- **Delete the `ollama` service entirely** — rejected: Linux/CI hosts without a native Ollama still
  benefit from a one-command containerised option; a profile keeps it at near-zero cost while
  removing it from the default path. (YAGNI cuts the other way here — the escape hatch is cheap and
  the spec keeps Ollama in the architecture.)
- **`host.docker.internal` with no container at all** — this *is* the default now; the profile-gated
  container is purely the Linux/CI fallback.
