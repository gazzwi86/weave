# ADR-002: Tenant-context stamping happens before/during the request, never after

## Status

Accepted

## Context

AC-5 requires every backend request to emit an OTel span carrying `tenant_id`,
`engine`, and `principal_iri`. The obvious implementation — a middleware that
calls `add_tenant_attributes(span, ...)` after `await call_next(request)`, once
the real (verified) principal is known — does not work, for three compounding
reasons discovered by running the test suite in its full, realistic ordering
(not just the new test file in isolation):

1. `FastAPIInstrumentor.instrument_app` wraps its own `OpenTelemetryMiddleware`
   outside every user-added middleware, and that middleware ends the span as
   soon as the response body's final chunk is sent — which happens deep
   inside the downstream call, before any "after `call_next`" code in an
   outer middleware runs.
2. Starlette's `@app.middleware("http")` / `BaseHTTPMiddleware` runs the
   downstream app in a separate `asyncio` task with a **copied** context, so
   `ContextVar` writes made by a dependency deep in that task (e.g.
   `get_current_principal`) are never visible back in the middleware after
   `call_next()` returns.
3. Starlette caches `app.middleware_stack` on first request and only rebuilds
   it if that attribute is `None`. If any request reaches the shared `app`
   singleton before `FastAPIInstrumentor.instrument_app` runs, the cached
   stack silently keeps serving the un-instrumented version even after
   instrumentation happens later — a real failure mode once more than one
   test module shares the same `app` object.

## Decision

- `TenantContextMiddleware` is a raw ASGI middleware class (not
  `BaseHTTPMiddleware`), so it runs in the same task/context as everything
  downstream. It sets default `tenant_id`/`principal_iri` `ContextVar`s and
  stamps the current span **before** calling the downstream app.
- `auth.dependencies.get_current_principal` re-stamps the span with the real,
  verified values immediately after token verification — still well before
  the response finishes streaming.
- `observability.tracing.setup_tracing()` resets `app.middleware_stack = None`
  after instrumenting, forcing Starlette to rebuild the (now correctly
  instrumented) stack on the next request.
- A missing `tenant_id` raises in test mode (`WEAVE_TESTING=1`) rather than
  silently defaulting to `"unknown"`, so a regression fails CI, not
  production.

## Consequences

- Span attribution is only ever "as good as known so far" — a real principal
  determined by code that runs *after* the span would have already closed by
  the time it publishes. There is no such code today (auth resolves before
  any handler body runs), but any future middleware/dependency that resolves
  identity late in the request lifecycle needs to re-stamp the span itself
  rather than relying on a generic "after call_next" pattern.
- Tests that build the ASGI app and send requests through it must not assume
  span instrumentation is active unless they called `setup_tracing(app,
  testing=True)` themselves — `test_otel_tracing.py`'s fixture does this per
  test.
