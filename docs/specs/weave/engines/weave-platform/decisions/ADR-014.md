# ADR-014: Dashboard widget state lives in Aurora Postgres under M1 RLS

## Status

Accepted — human decision 2026-07-08 via Stochastic Reasoning elicitation (user took the
recommendation). Resolves OQ-10.

## Context

FR-008/FR-011 fix widget state server-side: (tenant,user)-scoped pins, a tenant-scoped
published library, independent per-user copies, refinement history (default 10 steps), all
RBAC-scoped and audit-visible — never localStorage. ADR-013 adds a persisted `last_result`
per widget instance. OQ-10 asked which store holds this and what the render-cache strategy is.

The M1 platform already runs Aurora PostgreSQL Serverless v2 with row-level-security tenant
isolation proven end to end: `tenant_id` RLS policies, the non-superuser `weave_app` role
(ADR-003), pre-request tenant-context stamping (ADR-002), and the mandatory cross-tenant-read
test.

## Decision

1. **Widget state = new tables in the existing Aurora database**, exact DDL in
   `../tech-spec/m2-delta.md`: widget instances (pins + default/role-home tiles + per-user
   library copies), tenant widget library, refinement history. All carry `tenant_id`
   NOT NULL with the same RLS policy family as the M1 16-entity ERD; the existing
   cross-tenant-read test extends to cover them.
2. **Render cache = `last_result` JSONB + `fetched_at` + `status` on the widget-instance row**
   (ADR-013). No separate cache system for widget payloads.
3. **Not DynamoDB, not Redis** — no second store, no new isolation story, no new local-dev
   emulation, nothing new for the security review to re-prove.
4. **Recorded upgrade path (additive, not speculative):** if dashboard read fan-out later
   demands sub-100 ms state reads at high concurrency, add an ElastiCache read-through tier
   *in front of* these tables (the Hybrid option) — tables and RLS unchanged, cache is
   invalidate-on-write. Nothing in this decision forecloses it.

## Elicitation record (Stochastic Reasoning, 2026-07-08)

Options: **A** Aurora Postgres (existing DB + RLS) · **B** DynamoDB (partition-key + IAM
isolation) · **C** Hybrid (Aurora state + ElastiCache render cache).

| Criterion | Weight | A Aurora | B DynamoDB | C Hybrid |
|---|---|---|---|---|
| C1 Tenant-isolation safety (reuse proven RLS vs prove a new story) | 5 | 5 | 2 | 4 |
| C2 Fit to access pattern (library listing, per-user copies, history, transactional pin+audit) | 4 | 5 | 3 | 5 |
| C3 Operational simplicity / fewest moving parts | 4 | 5 | 3 | 3 |
| C4 Latency & scale headroom | 2 | 4 | 5 | 5 |
| C5 Cost | 2 | 4 | 4 | 3 |
| **Weighted (max 85)** | | **81** | **52** | **68** |
| Confidence | | high | medium | medium |

A wins C1 because RLS + `weave_app` + the cross-tenant test already exist and are proven
(ADR-002/003); B loses C1 because leading-key isolation is a new story the security review
must re-prove, and loses C2 because the library/copies/history access pattern is
relational-transactional; C loses C3 because two stores serve one feature and Redis eviction
needs its own answer to E1-S4's retained-last-render guarantee.

**Runner-up analysis:** C (Hybrid) becomes the right choice only under high-concurrency
sub-100 ms read fan-out — captured as the additive upgrade path in Decision 4.

**User decision:** Accepted recommendation A (relayed via coordinator, 2026-07-08).

## Consequences

**Positive:** zero new infrastructure; tenant isolation, migrations, backup, local dev
(Postgres container) all inherited from M1; widget writes join the same transaction as their
`PLAT-AUDIT-1` entries; one degradation/isolation test family extends instead of forking.

**Negative:** widget-state reads share Aurora capacity with the six PLAT-* services — accepted
at M2 volume (5-min refresh cadence, small JSONB aggregates); the upgrade path above is the
relief valve.

**Neutral:** JSONB payload size is bounded by the finite component library's data shapes;
refinement history is capped (10 steps default, tunable) so rows don't grow unbounded.

## Alternatives considered

Scored above. Summary: **DynamoDB** rejected on isolation-proof cost and relational access
pattern; **Hybrid** rejected now as a second moving part with an eviction-vs-E1-S4 conflict,
recorded as the additive scale upgrade instead.
