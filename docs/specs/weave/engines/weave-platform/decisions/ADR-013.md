# ADR-013: Widget data uses stale-while-revalidate over a persisted last-result

## Status

Accepted — human decision 2026-07-08 (M2 plan approval, relayed by coordinator). Resolves OQ-03.

## Context

OQ-03 asked: cache the last widget result for instant pre-refresh display, or always fetch
fresh? The requirements already half-answer it:

- E1-S4 (FR-009): on refresh/provider error the widget SHALL retain the **last successful
  render** with a stale-data badge + timestamp — so a durable last-result must be stored
  regardless of the caching choice.
- Widgets auto-refresh every 5 min (tunable); the dashboard SLA is ≤ 2 s p95 initial load.
- The dashboard is **not a consistency surface** — CE already caches `CE-METRICS-1` 60 s
  per tenant on the same rationale (CE `m2-delta.md` §7).
- CE-METRICS-1 `shacl_errors_by_severity` MAY return `{ "pending": true }` instead of counts
  (canonical note in `../../contracts.md`); honesty rules ban rendering zeros for it.

## Decision

1. **Persist the last successful payload server-side per widget instance** (pinned widget,
   fixed default tile, role-home tile): `last_result` JSONB + `fetched_at` + `status` columns
   on the widget-state row (store per ADR-014). This is the same record E1-S4's failure AC
   requires — one mechanism serves both.
2. **Read path = SWR:** widget-state reads return the stored payload + `fetched_at`
   immediately; the client paints it, then revalidates in the background (on mount, and on the
   5-min auto-refresh timer). Fresh data replaces payload + timestamp atomically.
3. **Failure path:** refresh failure never blanks or zeroes — `status` becomes `stale`,
   the client renders the retained payload with the stale badge + timestamp (E1-S4).
4. **Staleness bound:** a payload older than 2× the refresh interval (default 10 min, tunable
   via `PLAT-SETTINGS-1`) renders with the stale badge even if no refresh has failed —
   old data is never silently presented as current.
5. **Honest-state matrix** (normative; full rendering spec in `tech-spec/m2-delta.md`):
   `fresh` · `stale` (badge + timestamp) · `pending` (CE `{pending:true}` sub-fields — "counts
   pending", never zeros) · `unavailable` (no successful result yet + provider error) ·
   `source-not-ga`. Every widget resolves to exactly one.
6. **No extra cache tier.** No ElastiCache entry for widget payloads; the JSONB last-result IS
   the cache. Redis remains for what M1 already uses it for.

## Consequences

**Positive:** instant paint from first byte of state read; E1-S4's failure AC and the cache are
one mechanism (nothing to drift); zero new infrastructure; degradation states are enumerable
and testable in one sweep (epic-level AC "one degradation sweep verifies all").

**Negative:** users can see up-to-5-min-old (bounded ≤ 10 min) numbers before revalidation
completes — accepted, the dashboard is not a consistency surface and every payload carries its
timestamp.

**Neutral:** per-user pinned copies each hold their own last_result; storage cost is bounded by
widget count × payload size (KPI/chart aggregates, small).

## Alternatives considered

- **Always fetch fresh** — rejected: blank-until-loaded first paint works against the ≤ 2 s
  dashboard SLA, hammers CE-METRICS-1 on every dashboard open, and E1-S4 still forces storing
  the last result for the failure path — so "fresh-only" costs the same storage plus worse UX.
- **ElastiCache render cache** — rejected: a second moving part whose eviction would break the
  E1-S4 guarantee (the last successful render must survive; Redis eviction/restart loses it);
  5-min-refresh widgets don't need sub-millisecond reads.
- **Client-side SWR only (localStorage/memory)** — rejected outright by FR-008: widget state is
  server-side, (tenant,user)-scoped, cross-device, RBAC- and audit-visible.
