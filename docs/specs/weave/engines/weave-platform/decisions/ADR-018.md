# ADR-018: Dashboard-generate E2E wires a real Ollama provider; provider_503 stays integration+component-only

## Status

Accepted — engineer decision during PLAT-V1-TASK-012, no human course-correction received.

## Context

`playwright.config.ts`'s backend `webServer` command started `uvicorn` with no
`WEAVE_MODEL_PROVIDER` set, so it defaulted to `AnthropicProvider` (no API key in any dev/CI
sandbox) — every real classification call failed. Before TASK-012, `dashboard/intent.py::resolve()`
was itself a hard stub that always raised `ProviderUnavailable`, so this went unnoticed: the one
E2E test that existed (`prompt-bar.spec.ts`, "surfaces the named provider_503 state") happened to
pass either way.

TASK-012 makes `resolve()` real (JSON+Pydantic classification over `ai/providers.py`). Its own
brief requires a happy-path E2E ("skeleton fills to a real widget, backend row exists") that can
only pass if the shared e2e backend process has a genuinely reachable provider — and per ADR-011,
Ollama running natively on the host is the dev-default provider for exactly this reason.

A single `playwright.config.ts` `webServer` is one process shared by every test in the run. It
cannot be "working" for the happy-path test and "down" for the provider_503 test at the same time
without an unsafe, flaky mechanism (stopping/starting the host's Ollama daemon mid-suite, or
swapping env vars per-test against an already-running process).

## Decision

1. Add `WEAVE_MODEL_PROVIDER=ollama` + `OLLAMA_MODEL` (same default as `make dev`) to the backend
   `webServer` env in `playwright.config.ts`. This is the only spec file that calls a real
   AI-classifying endpoint over the network (verified: `ce-authoring.spec.ts` and friends mock
   `page.route` at the network layer, so they are unaffected).
2. Retire the E2E-level "surfaces the named provider_503 state" scenario. AC-4's requirement (a
   provider outage is a clean, named, retryable terminal state) stays covered at two levels that
   don't need a real network failure to exercise it:
   - `test_provider_503_named_state` (`test_dashboard_generate_api.py`) — integration test,
     `app.dependency_overrides[get_dashboard_agent_resolver]` injects a resolver that raises
     `ProviderUnavailable` deterministically.
   - `renders a named, retryable state for provider_503` (`prompt-bar.test.tsx`) — component test,
     asserts the "AI provider unavailable" copy + "Try again" button render from a mocked SSE
     stream.
3. The unit-level gap this surfaced (nothing proved `resolve()` itself translates a real connection
   error into `ProviderUnavailable`) is closed separately:
   `test_resolve_raises_provider_unavailable_on_connection_error` in `test_dashboard_intent.py`,
   backed by a `_call_model` exception boundary in `intent.py` mirroring `routers/authoring.py`'s
   pattern.

## Consequences

**Positive:** the happy-path E2E genuinely exercises the real classify → map → persist → render
flow end to end (Law B); AC-4 keeps deterministic, non-flaky coverage at the levels best suited to
it (no test relies on an accidental "provider happens to be unconfigured" environment quirk).

**Negative:** there is no longer an E2E test that proves a *real* provider network failure reaches
the UI as `provider_503` — only that the resolver raises the right exception (unit) and that the
UI renders the right state given that exception (component/integration). Judged acceptable: the
three tests together cover every seam in the chain, just not all three in one browser-driven run.

**Neutral:** CI/other dev sandboxes must have Ollama reachable with the `make dev` default model
for this E2E suite to pass — already true for `make dev` itself (ADR-011), so no new operational
requirement.

## Alternatives considered

- **Leave the backend webServer with no provider configured** — rejected: the happy-path E2E named
  in TASK-012's brief could never pass, defeating the point of un-`fixme`-ing it.
- **Stop/restart the Ollama daemon mid-suite for the provider_503 test** — rejected: flaky, requires
  host-level process control from a test, and races the other 55+ tests sharing the same worker.
- **Point `provider_503` at a deliberately-wrong `OLLAMA_URL` for just that spec** — not possible
  with a single shared `webServer` process; would require a second backend process (real cost/
  complexity for coverage already held elsewhere).
