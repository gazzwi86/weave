# ADR-001: Cognito token validity uses AWS's real minimums, not the brief's 60s

## Status

Accepted

## Context

PLAT-TASK-001's brief specified a binding design decision: Cognito access,
ID, and refresh token validity should all be 60 seconds, for the shortest
possible token lifetime.

`terraform plan` (run offline per Law F, with dummy credentials and the AWS
provider's `skip_*` flags) fails this value at plan time — not apply time —
because the AWS provider schema for `aws_cognito_user_pool_client` validates
`access_token_validity` / `id_token_validity` as a duration in the range
5 minutes–24 hours, and `refresh_token_validity` in the range 1 hour–8760
hours. 60 seconds is below both floors and is rejected before any network
call is made.

## Decision

Use AWS's own minimums as the shortest legal approximation of the brief's
intent:

- `access_id_token_validity_seconds` default `300` (5 minutes)
- `refresh_token_validity_seconds` default `3600` (1 hour)

Both are exposed as separate module variables (previously a single
`token_validity_seconds`) with `validation` blocks enforcing AWS's bounds, so
a future attempt to set an illegal value fails fast with a clear Terraform
error instead of a rejected plan.

## Consequences

- The module can never be configured with a token lifetime shorter than AWS
  allows — validation blocks make illegal values a Terraform-time error.
- Actual session/token lifetime is 5x (access/ID) and 60x (refresh) longer
  than the brief intended. If this matters for a specific compliance
  requirement, it needs an explicit follow-up decision — flagging for
  architect/QA review.
- No further action needed to make `terraform plan` pass; this was
  discovered and fixed during PLAT-TASK-001's offline plan test
  (`test_terraform_plan.py`).
