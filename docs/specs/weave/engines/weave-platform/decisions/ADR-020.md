# ADR-020: canonical in-tenant role slugs are additive to ROLE_RANK, not a replacement

## Status

Accepted ‚Äî engineer decision during PLAT-V1-TASK-030, no human course-correction received.

(Renumbered from a draft ADR-019 to ADR-020: `PLAT-V1-TASK-016` landed its own, unrelated
ADR-019 ‚Äî "token-spend binding uses M1's workspace-scoped billing summary" ‚Äî on `main` via PR #75
first. `docs/specs/weave/engines/weave-platform/decisions/` ADR numbers are assigned in landing
order, not task order, so this file takes the next free slot.)

## Context

`weave-platform.md`'s "Canonical human roles" section names 10 in-tenant role slugs (`viewer`,
`automation_author`, `ops_sre`, `engineer`, `brand_content_owner`, `data_steward`,
`business_analyst_sme`, `enterprise_architect`, `compliance_officer`, `workspace_admin`). TASK-030's
invite role selector (AC-3) must offer exactly this set, and the invited member's
`workspace_members.role` column is written verbatim from it.

`rbac.py`'s `ROLE_RANK` dict (the single authoritative role hierarchy `check_role` /
`require_workspace_role` gate every workspace-scoped route against) only defined the legacy 4-tier
M1 vocabulary: `read` (0), `author` (1), `publish` (2), `admin` (3) ‚Äî the ranks seeded workspaces'
existing members already carry. Inviting a new member with a canonical slug the dict has no entry
for would make `check_role` raise `KeyError` (an unhandled 500) the first time that member's role is
checked against any `require_workspace_role(min_role)` dependency.

## Decision

`ROLE_RANK` gains the 10 canonical slugs as additional keys, ranked by seniority against the same
0‚Äì3 scale the legacy vocabulary already uses (`viewer`‚Üí0, the six individual-contributor slugs‚Üí1,
`enterprise_architect`/`compliance_officer`‚Üí2, `workspace_admin`‚Üí3). No existing key's rank changes,
no data migration touches already-seeded `workspace_members` rows, and `check_role` /
`require_workspace_role` need no branching ‚Äî a workspace created before this task (still
`"admin"`/`"author"`/`"read"`) and one invited after it (`"workspace_admin"`/`"engineer"`/...) rank
correctly through the exact same gate.

## Consequences

**Positive:** zero-migration fix; both vocabularies' existing callers (invite, revoke, every
`require_workspace_role` route) keep working unchanged; `test_role_rank_order_is_authoritative`
(`test_rbac.py`) still pins the legacy 4 ranks exactly, now alongside two canonical-slug spot checks
and a total-key-count assertion so a future accidental collision (a new slug reusing an existing
key) fails loudly.

**Negative:** `ROLE_RANK` now carries two parallel vocabularies rather than one normative
enumeration ‚Äî a reader has to know both exist. Documented here rather than left implicit.

**Follow-up:** a future task could migrate seeded M1 workspaces' legacy role strings to their
canonical-slug equivalents and retire the 4-tier vocabulary entirely; out of TASK-030's scope (a
Settings-UI task, not an RBAC-migration task).

## Alternatives considered

- **Migrate existing `workspace_members.role` values to canonical slugs, drop the legacy 4-tier
  keys** ‚Äî rejected: a data migration is out of TASK-030's stated scope and risks breaking any other
  M1 route that still compares against the literal legacy strings.
- **Branch `check_role` on which vocabulary a role string belongs to** ‚Äî rejected: adds a conditional
  to the one function every workspace-scoped route depends on, for no behavioural gain over a flat
  additive dict.
