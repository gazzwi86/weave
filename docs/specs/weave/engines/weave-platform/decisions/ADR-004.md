# ADR-004: domain/project scope levels are addressable but unmodeled; the AC-5 guard only proves company-scope writes

## Status

Accepted

## Context

AC-4 requires a settings cascade resolved tightest-first across four scope levels:
`project` -> `workspace` -> `domain` -> `company`. AC-5 requires that setting a *looser*
scope's value is rejected with 422 when a *tighter* scope already has an explicit override
for the same key.

Only one of those four levels — `workspace` — is a real, queryable entity in this task
(PLAT-TASK-003 ships workspace creation; no `domain` or `project` entity, table, or
parent-link exists anywhere in the codebase yet). AC-1's own workspace IRI grammar
(`urn:weave:tenant:{tid}:ws:{wid}`) has no `domain` segment, so a workspace's ancestor
chain skips `domain` entirely and goes straight to `company` — `domain` is addressable (a
caller can write a setting scoped to a `urn:weave:tenant:{tid}:domain:{did}` IRI, and it
will resolve correctly if a workspace's IRI is later extended to include a domain segment)
but nothing in this task ever produces or looks up a domain-scoped IRI's *children*.

This matters specifically for AC-5's write guard: proving "does a tighter scope already
override this key" requires knowing which tighter-scope IRIs are descendants of the scope
being written to. That descendance is only provable today for `company`, because every
workspace deterministically belongs to exactly one tenant (`tenant_id` on the `workspaces`
row) — there is no equivalent provable link from a `domain` or `project` scope down to the
workspaces/projects nested under it, because those nestings aren't modeled yet.

## Decision

- `settings/scope.py`'s `ancestor_chain()` is purely IRI-grammar-driven (string parsing of
  the `urn:weave:tenant:...` segments), not a database lookup against a hierarchy table —
  it has nothing to query yet for domain/project parenthood.
- `settings/resolver.py`'s `set_setting()` only runs the "does a tighter scope already
  exist" guard when writing at `scope == "company"` (checking `scope_rank < company's
  rank` for the same tenant+key, which correctly captures every workspace since all
  workspaces belong to the tenant/company). Writes at `domain`, `workspace`, or `project`
  scope are accepted unconditionally in this task — the guard does not (yet) claim to
  reject a `project` write that is looser than some other `project` under the same
  workspace, because there is no way to prove that relationship without a real
  domain/project entity model.
- This is recorded here rather than silently under-implemented: AC-5's acceptance
  criterion is satisfied for the scope pair the brief's test scenarios actually exercise
  (company vs. a tighter override), and the gap for domain/project is explicit, not
  discovered later as a surprise.

## Consequences

- The moment a future task introduces real `domain` or `project` entities with a queryable
  parent link (e.g. a `projects` table with `workspace_id`), `set_setting()`'s guard must
  be extended to walk that link the same way it currently trusts `workspaces.tenant_id` —
  otherwise AC-5 will have a real, exploitable gap (a looser `workspace`-scope write could
  silently override an existing tighter `project`-scope value).
- QA/PLAT-TASK-004 should not assume the looser-override guard is complete for every scope
  pair — only company-vs-tighter is proven and tested
  (`test_settings_looser_override_rejected` in `tests/unit/test_settings_resolver.py`).
