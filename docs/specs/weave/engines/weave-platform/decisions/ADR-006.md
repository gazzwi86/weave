# ADR-006: principals.iri is not the table's primary key

## Status

Accepted

## Context

PLAT-TASK-004's brief specifies the exact IRI mint formulas:

- human: `urn:weave:principal:user:{sub}`
- agent: `urn:weave:principal:agent:{sha256(iam_role_arn)[:16]}`

Neither formula includes `tenant_id`. In production this is safe: a real
Cognito `sub` is a UUID unique across a shared user pool, and a real IAM
role ARN embeds a 12-digit AWS account ID, so two different tenants
essentially never produce the same `sub`/hash.

That assumption breaks in this dev/test environment two ways, both
surfaced by running the actual docker-integration suite (not caught by
unit tests, which mock the DB boundary):

1. LocalStack's community-edition STS `GetCallerIdentity` always resolves
   to the same fixed root ARN regardless of the session token presented --
   so every agent-token mint in one test run derives the identical
   `agent_principal_iri`, no matter which tenant/workspace is involved.
2. Test fixtures reuse simple, low-cardinality human `sub` strings
   (`"u-admin"`, `"u1"`) across multiple test functions, each with its own
   tenant -- the same collision, for the same reason: `iri` is a function
   of `sub` alone.

With `iri TEXT PRIMARY KEY`, the second tenant's insert hits a primary-key
violation that the `ON CONFLICT (tenant_id, sub) DO UPDATE` clause does not
catch (it isn't the constraint being violated), so the write fails outright
-- silently swallowed by the best-effort persistence wrapper in production
code paths, and a hard test failure in the docker-integration lane.

## Decision

`principals.id UUID PRIMARY KEY DEFAULT gen_random_uuid()` is the table's
real primary key. `iri` is a plain `NOT NULL` column (indexed via
`(tenant_id, iri)` for the `GET /api/principals/{iri}` lookup).
`UNIQUE (tenant_id, sub)` remains the only real uniqueness guarantee and the
upsert target -- unchanged from the original migration. The IRI mint
formulas themselves are untouched (still exactly as the brief specifies);
only the schema's uniqueness scope changed, from "globally unique iri" to
"unique per tenant".

## Consequences

- Two different tenants CAN now have rows with the literal same `iri`
  string. This is intentional: each row is still correctly isolated by
  `tenant_id` (RLS-enforced), and a lookup by `{iri}` is always additionally
  scoped by the caller's own `tenant_id` (`GET /api/principals/{iri}` never
  crosses tenants regardless).
- If a future requirement needs `iri` to be a genuine cross-tenant-unique
  resource identifier (e.g. a public, cross-tenant graph-linking use case),
  that requires embedding `tenant_id` into the mint formula itself -- a
  breaking change to the brief's literal formula, out of scope here and
  worth a fresh ADR if it comes up.
