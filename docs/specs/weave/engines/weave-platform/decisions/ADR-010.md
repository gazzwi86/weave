# ADR-010: PLAT-AUDIT-1 hash-chain implementation deviations

## Status

Accepted

## Context

PLAT-TASK-009's pseudocode (`db.serialisable_transaction(): ...`) and prose
describe four things the existing codebase either can't do literally or
already does differently:

1. **Per-tenant seq/prev_hash race.** The pseudocode wraps the read-latest +
   insert in a nested serialisable transaction. `tenant_connection()`
   already opens one transaction per request; asyncpg has no nested
   transaction primitive that maps onto "retry on serialization failure"
   without restructuring the whole request lifecycle.
2. **`ts`'s storage type.** Not specified. `verify_chain` must recompute the
   exact hash that was signed at emission time -- any lossy round trip
   (e.g. Postgres `TIMESTAMPTZ` normalising precision/offset) would make
   every entry fail verification.
3. **Security-event notification.** The pseudocode calls
   `background_task(dispatch_notification, ...)`, but `billing/gate.py`'s
   `_notify_workspace_admins` (the only existing precedent for "audit-like
   event triggers a notification") dispatches synchronously on the same
   connection/transaction, not via a detached background task.
4. **Append-only enforcement across roles.** The brief requires the
   `BEFORE UPDATE OR DELETE` trigger to block writes "in all test
   environments," calling out that migrations run as a superuser role that
   bypasses RLS. The integration tests (`test_audit_chain_api.py`) revealed
   that Postgres checks table-level GRANT privileges *before* a trigger
   ever fires, so `weave_app`'s UPDATE is rejected at the GRANT layer, not
   the trigger -- only a role with UPDATE/DELETE privilege that ignores
   RLS (a superuser) reaches the trigger at all.

## Decision

1. `HashChainAuditEmitter.emit` calls
   `SELECT pg_advisory_xact_lock(hashtext($1))` (keyed on `tenant_id`)
   before reading the latest `seq`/`hash`, inside the caller's already-open
   transaction. This serialises concurrent emits for the same tenant without
   a nested transaction: the lock is released automatically at
   commit/rollback, and two concurrent emits for different tenants never
   contend.
2. `ts` is stored as `TEXT` (a canonical ISO-8601 string, `datetime.now(UTC).isoformat()`),
   not `TIMESTAMPTZ`. `verify_chain` re-hashes the exact string fetched from
   the row, guaranteeing byte-for-byte reproducibility.
3. `notify_tenant_admins_of_security_event` runs synchronously, on the same
   connection, inside the same transaction as the audit insert -- matching
   `billing/gate.py`'s precedent, not a `BackgroundTasks`-style detached
   task. `dispatch_notification`'s never-raises guarantee covers only its
   Slack retry leg; its DB awaits (`insert_notification`, the re-entrant
   `default_audit_emitter.emit`, `get_user_prefs`) can still raise. The call
   is wrapped in `try/except Exception` in `emitter.py` (log a warning, do
   not re-raise) precisely because it cannot be trusted never to raise --
   the audit entry is the primary record and must not roll back because a
   notification failed.
4. Both layers stay: `GRANT SELECT, INSERT` only (no UPDATE/DELETE) on
   `audit_entries` for `weave_app`, *and* the `BEFORE UPDATE OR DELETE`
   trigger. `weave_app` is rejected at the GRANT layer
   (`InsufficientPrivilegeError`); the migration/superuser role -- which
   bypasses GRANTs -- is rejected by the trigger
   (`RaiseError: audit_entries is append-only`). Two independent
   enforcement layers, each proven by its own test
   (`test_audit_table_update_rejected_at_db`,
   `test_audit_table_update_rejected_for_superuser_migration_role`).

## Consequences

- Concurrent emits for the same tenant serialise on a Postgres advisory
  lock rather than a nested transaction; this is a coarser lock (whole
  tenant, held for the emit's duration) but scoped to one `INSERT` and
  released at commit, so contention is brief and per-tenant only.
- `ts` being `TEXT` means no `TIMESTAMPTZ` range/interval queries against
  it directly; if a future task needs time-range filtering, it should add a
  derived indexed column rather than parsing the text column at query time.
- The security notification runs inside the audit transaction, but guarded
  by a `try/except`: if `dispatch_notification` raises, the audit insert
  (already executed above it) is unaffected -- the exception is caught,
  logged as a warning, and swallowed. AC-6 delivery is therefore
  best-effort, matching the platform's in-app-first degradation
  philosophy: the audit entry (the primary record) must never be lost
  because a notification failed.
- Append-only enforcement has two distinct failure modes depending on
  caller role (`InsufficientPrivilegeError` for `weave_app`, `RaiseError`
  for the superuser/migration role) -- any future test or tooling that
  checks "was the write blocked" must not assume a single exception type.
- Known limitation (QA, PLAT-TASK-009 pass): a superuser can still `TRUNCATE
  audit_entries` or `ALTER TABLE ... DISABLE TRIGGER` -- row-level triggers do
  not fire for TRUNCATE and a superuser bypasses the GRANT layer. This is an
  inherent ceiling of DB-level enforcement; detecting it requires external
  chain anchoring (e.g. periodic head-hash export), a post-v1 concern.
