# ADR-023: G15 operator-console tenant management — platform scope, registry table, suspension enforcement

## Status

Accepted.

## Context

G15 (`docs/design/remediation-2-api-gaps.md`) asks for cross-tenant
operator-console endpoints (list/provision/suspend companies), super-admin
only. Four design points had no prior precedent in this codebase and are
recorded here per Law 10.

## Decisions

**1. Super-admin is a new `RoleGrant.scope = "platform"` value, not a new
claim shape.** The JWT `roles` claim (`auth/dependencies.py::RoleGrant`)
already carries `scope`/`role`; adding `"platform"` alongside the existing
`"tenant"/"domain"/"project"` reuses the already-hardened
`_parse_roles_claim` malformed-input handling (degrades to `[]`, never
over-grants) instead of inventing a second claim. `require_super_admin`
(`rbac.py`) checks `scope == "platform" and role == "super_admin"` purely
from the JWT — no DB lookup, unlike `require_tenant_admin` — because a
platform operator is not a member of any tenant's `workspace_members`.
A tenant-scope `admin`/`owner` grant (which overlays project/domain checks
elsewhere via `has_admin_grant`) does **not** satisfy `require_super_admin`:
tenant-admin of company A must never reach company B, which is the whole
point of G15's isolation requirement.

**2. `tenants` is a new, deliberately non-RLS'd registry table.** Every
other tenancy table (`workspaces`, `workspace_members`, `settings`,
`audit_events`) is `FORCE ROW LEVEL SECURITY`'d on `tenant_id`, because
those tables hold *per-tenant data*. `tenants` is different: it IS the
cross-tenant list, one row per company, and the one query G15 exists to
serve reads across all of them by design. Putting RLS on it and bypassing
via a `SECURITY DEFINER` function (the `list_pollable_tenants()` pattern,
migration 0084) would add ceremony for no isolation gain — the isolation
boundary here is `require_super_admin` at the app layer, not the row
policy. `weave_app` gets plain `SELECT/INSERT/UPDATE` on it.

**3. Suspension is enforced in `get_current_principal`, fail-open on an
unregistered tenant or DB error.** A `status` column nobody reads at
request time doesn't "freeze access" — it just sits there. The single
choke point every authenticated request already passes through is
`get_current_principal` (`auth/dependencies.py`), so the suspension check
lives there: one indexed PK read on `tenants` via `untenanted_connection`,
403 `{"error": "tenant_suspended"}` when `status = 'suspended'`. It fails
**open** (no block) when the tenant has no `tenants` row at all — every
tenant provisioned before G15 (demo/seed/test tenants, `seed_demo.py`,
`onboarding/sandbox.py`) has none, and fail-closed would turn "no operator
record" into "nobody can log in", which is wrong. It also fails open on any
DB error, matching the existing session-version check's degrade-silently
posture — Postgres must never become a second hard dependency for the
"am I allowed to authenticate at all" path. Cost: one extra indexed lookup
per authenticated request, app-wide. Upgrade path if that matters at scale:
cache the per-tenant status in Redis alongside `session_version`.

**4. Provisioning is synchronous (`201`, not `202`+async).** The brief
flagged "seed the 13 framework kinds" as the reason provisioning might need
to be async. It doesn't: the BPMO framework kinds are global SHACL shapes
(`operations/shacl.py::shapes_graph()`, `ontology/catalogue.py`), not
per-tenant instance data — `decision_ontology-bpmo.md` already establishes
BPMO as upper-framework grammar, not a populated client taxonomy. A new
company gets the framework for free the moment its named graph exists;
there is nothing to seed. Provisioning is therefore exactly as heavy as
`create_workspace_route` (insert row, mint workspace, invite first admin,
audit) — a single transaction, no async job needed.

## Consequences

- `require_super_admin` needs its own operator-identity story before this
  ships to real users (who mints a `platform`-scope JWT?) — out of scope for
  G15, which only had to build the endpoints; provisioning the first
  super-admin operator account is a follow-up.
- `company_summaries()` (cross-tenant aggregation) does one
  `tenant_connection` round-trip per tenant plus one Oxigraph
  `entity_count_by_kind` call per workspace — fine at operator scale
  (dozens of tenants), would need batching before hundreds.
