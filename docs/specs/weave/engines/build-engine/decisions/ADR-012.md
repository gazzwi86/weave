# ADR-012: project-role guard's admin overlay is tenant-scope only; the JWT `roles` claim is engineer-added

## Status

Accepted (TASK-011, EPIC-002, engineer-authored during implementation)

## Context

TASK-011's brief has `require_project_role` overlay a tenant/domain admin or owner grant over the
per-project `admin`/`editor` role (AC-4), reading that grant off `ctx.roles_claim` /
`has_admin_grant(ctx.roles_claim, domain=project.domain_iri)`.

Two things this presupposes did not exist in the codebase before this task:

1. **A `roles` claim on the authenticated principal at all.** `Principal` (`auth/dependencies.py`)
   had no `roles` field; JWT verification parsed no such claim; the mock-OIDC test-token issuer
   (`mock_oidc/tokens.py`) had no way to mint one.
2. **A `domain_iri` on a project**, needed to decide whether a domain-scope grant covers *this*
   project. `projects` (migration `0009_projects.sql`) has no such column, and no `domains` table
   exists anywhere in this schema.

## Decision

**1. Roles claim: added, following the existing claim-parsing precedent.** `Principal` gains a
`roles: list[RoleGrant]` field (default `[]`), parsed from the JWT's `roles` claim the same way
`session_version`/`principal_type` already default for tokens issued before those claims existed.
`RoleGrant` is a small pydantic model (`scope: "tenant" | "domain" | "project"`, `role: str`,
optional `domain_iri`/`project_iri`) — shape-validated, not cast, per Law 13 (the claim is untrusted
input). Malformed input degrades to "no grants" rather than 500ing the request, since the overlay is
additive-only: losing it only narrows the caller back to their per-project role, never over-grants.

`mock_oidc/tokens.py` is deliberately **not** extended to mint a `roles` claim — its `_claims()`
output is pinned byte-for-byte by an existing test (`test_claims_shape`), which exists specifically
because `session_version` is injected at sign-time rather than in `_claims()` for the same reason.
Integration tests needing a `roles`-bearing principal use `app.dependency_overrides[
get_current_principal]` instead (precedent: `test_identity_rbac.py` already overrides
`get_oidc_client` the same way) — still a real route, real Postgres, real guard logic; only the
principal is stubbed.

**2. Domain overlay: implemented as a pure function, not wired to a real column.**
`has_admin_grant(roles, *, domain)` takes an explicit `domain` parameter and is unit-tested against
both AC-4 domain scenarios (matching domain grant allows, different domain grant denies) directly.
At the route boundary (`enforce_project_role`), it is always called with `domain=None` — there is no
`project.domain_iri` to pass. A tenant-scope admin/owner grant is therefore the only overlay that is
*live* against a real project today; the domain branch is correct and tested, but dormant until a
`domain_iri` column exists.

## Consequences

- AC-4's "domain admin covering the project's domain" and "domain admin for a different domain"
  scenarios are proven at the unit level (`has_admin_grant` called directly with a domain argument),
  not at the integration/HTTP level against a real project row. This is the honest state of AC-4
  today, not a skipped test.
- The moment a `domain_iri` (or equivalent Company → Domain → Project cascade field, per
  `PLAT-SETTINGS-1`) lands on `projects`, `enforce_project_role` is a one-line change
  (`domain=project.domain_iri` instead of `domain=None`) to make the domain overlay live — no
  further design work needed, `has_admin_grant` already handles it.
- `Principal.roles` is new, real, load-bearing production surface (not a stub) — every route
  guarded by `require_project_role` reads it. It is currently populated only via real JWTs that
  carry a `roles` claim (none are minted by this repo's mock OIDC yet) or via test overrides. No
  production issuer in this codebase populates a `roles` claim today, so in practice `roles` is
  `[]` for every principal outside of tests until PLAT-IDENTITY-1's issuer-side work lands — the
  overlay is real code with no real callers yet, which is expected for a guard shipped ahead of the
  routes that will use it (TASK-011 ships no new endpoint).

## Alternatives considered

- **Escalate both gaps as a blocking ambiguity (Law 11).** Rejected for the same reason as
  ADR-011's precedent: AC-4's own text names the overlay roles (tenant/domain "admin"/"owner"), so
  no role vocabulary is being invented; the JWT-claim-defaulting pattern already exists twice in the
  same file (`session_version`, `principal_type`); and the domain_iri gap was pre-scoped by the task
  brief itself as "document, don't block." Evidence is one-sided in both cases — Law 10 territory.
- **Extend `mock_oidc/tokens.py` to mint a `roles` claim.** Rejected: breaks `test_claims_shape`'s
  exact-equality pin, which exists precisely to keep claim-shape changes deliberate and visible
  (`session_version`'s own sign-time injection is the established workaround). Overriding
  `get_current_principal` in integration tests achieves the same coverage without touching that
  pinned shape.
- **Invent a `domain_iri` column on `projects` to make AC-4 fully route-testable now.** Rejected:
  not this task's scope, no schema/migration authority was given, and `PLAT-SETTINGS-1` already
  specifies a Company → Domain → Project cascade — adding an ad hoc column here would likely
  conflict with that future shape rather than anticipate it correctly.
