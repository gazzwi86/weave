# ADR-002: Specs/tasks are an in-memory process-local store in M1; PLAT-AUDIT-1 health check is a local DB probe

## Status

Accepted

## Context

TASK-005's pseudocode reads/writes specs and tasks through `aurora.get_spec`,
`aurora.update_task`, etc., implying an Aurora-backed table. The task brief's
own migration guidance says an engineer implementing this task "should NOT
need one" and to stop and report rather than invent a migration number if
one seems necessary.

Investigation confirmed: `packages/backend/migrations/` has no `specs` or
`tasks` table anywhere (only `0001_tenancy.sql` .. `0005_audit_chain.sql`
and `0009_projects.sql`). Every specified unit-test scenario for this task
(AC-1, AC-2, AC-3, AC-5, AC-6, AC-7) is a Unit test, not an Integration
test -- only AC-4 and AC-8 are Integration, and both exercise the real
`PLAT-AUDIT-1` audit-chain persistence, not a specs/tasks table. This is
consistent with M1's "thin-proof" milestone scope: the FSM/retry/HITL
*mechanics* are the deliverable, not durable spec/task storage (that lands
with TASK-006/TASK-007, which this task `unlocks`).

Decision: `weave_backend/build/store.py` is a module-level, in-memory,
process-local dict keyed by `(tenant_id, id)` -- `reset_for_tests()` clears
it between tests. No migration is added. A real Aurora-backed store is
expected to replace this module wholesale in a later task once specs/tasks
need to survive a process restart or be queried across replicas; the public
functions (`create_task`, `get_task`, `update_task_status`, ...) are the
seam a later task swaps the backing implementation behind.

Separately, the brief's `fire_hitl_gate` pseudocode calls
`plat_audit_client.health_check()` as if `PLAT-AUDIT-1` were a separate HTTP
service with its own `GET /api/audit/health` endpoint. In this codebase
`PLAT-AUDIT-1` (`audit/emitter.py`) runs in-process against the same
Postgres the rest of the backend uses -- there is no separate service to
probe. `build/hitl.py`'s `default_audit_health_check()` therefore acquires
a pool connection and runs `SELECT 1` under an explicit 200ms timeout
(per the brief's own implementation hint on timeout value), treating
`OSError` / `asyncpg.PostgresError` / `TimeoutError` as "unreachable" --
functionally equivalent fail-closed behaviour to the brief's
`ConnectionError` catch, adapted to this codebase's actual topology.

## Consequences

- Specs/tasks state does not survive a process restart and is not shared
  across replicas in M1 -- acceptable for the thin-proof milestone; a real
  backing store is a follow-up task, not a gap in this one.
- `get_retry_ceiling` treats a task with no `project_iri` (the common case
  today, since no route creates a task with one yet) as "no
  `PLAT-SETTINGS-1` scope to resolve" and falls straight to
  `DEFAULT_RETRY_CEILINGS`, rather than raising -- this keeps the ceiling
  lookup a no-op until project-scoped tasks exist.
- The HITL gate's fail-closed semantics (AC-5) are real (a genuine
  Postgres probe, not a stub), so the audit-outage test path is exercised
  against actual connection-pool failure modes, not a fake flag.
