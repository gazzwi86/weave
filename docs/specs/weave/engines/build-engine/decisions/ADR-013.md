# ADR-013: Build budget/rate-card cascade is company-scoped until projects carry a domain

## Status

Accepted (TASK-013 / TASK-012, EPIC-002). Coordinator-authored to resolve the dangling docstring
references in `build/costs.py` and `build/cost.py`, and to record a decision surfaced by QA
(finding XT-BE013-1) that is above generation-tier authority.

## Context

`GET /api/projects/{id}/costs` resolves a cumulative-project-spend cap (`build.budget.cap_usd`) and
generation resolves a rate card (`build.rate_card`), both through the PLAT-SETTINGS-1 tighter-wins
cascade (`resolve_setting(context_iri=...)`). The brief (AC-3) asks for a Company→Domain→Project
cascade so a project- or domain-level override wins.

Two facts make the domain/project rungs unreachable in production:

1. **IRI grammar mismatch.** `settings/scope.py` parses only `urn:weave:tenant:{tid}:...` scope
   chains. A Build project's real IRI is `urn:weave:project:{tid}:{slug}`
   (`projects/model.py::build_project_iri`), which never parses — `resolve_setting` raises
   `InvalidScopeIri`.
2. **No domain on projects.** `migrations/0009_projects.sql` gives `projects` no `domain_id` /
   `workspace_id` column, so there is no schema-backed way to construct a grammar-conforming
   domain- or project-scope IRI for a Build project even if the grammar were extended.

Two related distinctions also needed recording:

- `build.budget.cap_usd` (cumulative project spend, this module) is **deliberately distinct** from
  `billing/caps.py`'s per-period, Redis-metered `ai.budget.per_period_usd` used by the invoicing
  gate. Different concept, different key — not a duplicate.

## Decision

- The cascade **catches `InvalidScopeIri` and retries at company scope** (`company_iri(tenant_id)`).
  The **company-wide default cap / rate card is the reachable ceiling**; domain/project overrides
  are inert until the schema gap is closed. `build/cost.py::resolve_rate_card` gets the identical
  company fallback (previously it had none — every real dispatch produced an empty card /
  `RateCardConfigError`; that was the XT-BE013-1 production bug, now fixed).
- **Fail-open on no cap** (ADR-009 posture): a project with no configured cap anywhere is unmetered,
  not halted.
- Money crosses the wire as **`float`** (matching `schemas/billing.py`), not `Decimal` (which
  Pydantic serialises as a JSON string); internal computation stays `Decimal` for precision, cast at
  the DTO boundary.

## Consequences

- **Positive:** the dead-cascade / rate-card-erroring-every-dispatch bug is fixed; company-level
  budget caps and rate cards work end to end; wire contract is consistent.
- **Deferred (phase-gate ratification):** domain/project-level budget caps and rate cards are not
  honoured. Closing this needs a `projects.domain_id` migration **plus** a settings-grammar
  extension for project/domain scope IRIs. This is the **same root cause as ADR-012** (TASK-011's
  dormant domain role overlay) — one missing `projects.domain_id` column produces three symptoms
  (role overlay, budget cap, rate card). A single migration + grammar extension closes all three.
- **Safety:** the limitation under-grants (only the company default applies; a would-be tighter
  project cap simply isn't seen) — it never over-grants spend, and tenancy isolation is unaffected.
