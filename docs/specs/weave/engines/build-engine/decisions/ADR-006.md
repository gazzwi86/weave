# ADR-006: BE-SDK-1 codegen — deterministic template pipeline over one typed IR

## Status

Proposed

## Context

FR-059 / contracts.md §BE-SDK-1: from a pinned `CE-VERSION-1` version, emit (a) a typed graph SDK
in TypeScript/npm and Python/pip and (b) a standalone OpenAPI 3.1 contract wrapping
`CE-READ-1`/`CE-WRITE-1` — including one typed method per `CE-FUNCTION-1` function (definition
surface only in M2; execution is **post-v1**, CE ADR-009, deferred 2026-07-08). Output is
client-owned, forkable, emitted as
portable source into the project repo. Generation must be atomic and regenerable on `CE-DIFF-1`
deltas. The pipeline shape (LLM vs deterministic, one IR vs per-language, registry vs repo) was
undecided.

## Decision

1. **Deterministic, zero-LLM emit path.** The SDK generator is ordinary Python code (a Fargate
   task in the same family as the preview deployer), not an agent. Same pinned inputs ⇒
   byte-identical output (modulo generation timestamp in the provenance header). Codegen output is
   contract-tested; an LLM in the emit path would make the M2 exit criterion unverifiable.
2. **Single typed IR, three emitters.** Fetch stage pulls, for the pinned version: SHACL node
   shapes (via `ce_client`/CE-READ-1), the function list + derived JSON-Schema projections
   (`GET /api/functions`, `GET /api/functions/{iri}`), and CE-BRAND-1 **closed-core** tokens
   (`GET /api/brand/tokens` — core fields only; `extensions` passed through untyped). Normalise
   into one Pydantic IR (`SdkModel`: classes ← node shapes, fields ← property constraints with
   datatype + cardinality, query methods ← named SPARQL SELECTs, function methods ← JSON-Schema
   signatures, theme constants ← core tokens). Three Jinja2 emitter sets consume the IR:
   `typescript/`, `python/`, `openapi/`. Mapping bugs get fixed once, in the IR.
3. **Atomic emit.** Generate into a staging directory; validate (tsc --noEmit, mypy, OpenAPI 3.1
   schema lint) — then one `ScmDriver.commit_workspace` into the project repo. Unreachable
   CE-READ-1, unresolvable shape, or a failed validator ⇒ nothing lands, error names the
   shape/version (FR-059 atomicity). No npm/PyPI publishing — portable source in the repo is the
   contract.
4. **Versioning + provenance.** SDK package version = `{ce_version_tag}+build.{n}` (n = per-project
   generation counter); every generated file carries the `BE-ARTEFACT-1` provenance header (spec
   ID, pinned CE version, referenced entity IRIs). Prior packages stay in git history — a client
   can pin an older SDK by ref.
5. **breaking:true refusal.** Before regeneration, the generator calls `CE-DIFF-1` for the span
   `last-generated pin → requested pin` and evaluates `any(versions[].breaking)` over the ordered
   version span the response carries. CE computes each version's `breaking` flag **at publish
   time** and it covers BOTH function-signature changes (CE ADR-009) AND shape/kind surface
   changes — so this one flag is the whole breakingness signal. Build MUST NOT re-derive
   breakingness: no parsing of the diff's `added`/`removed`/`modified` triples, no re-derivation
   from the `GET /api/functions` list. Any `breaking: true` version in the span ⇒ generation
   halts to the existing HITL gate machinery
   (fail-closed, no-self-approval, D9); the approval row (approver principal IRI + acknowledged
   breaking version IRIs) is persisted in `gate_results` and audited via PLAT-AUDIT-1. No ack, no
   regeneration — silent breakage of a client-owned SDK is the failure mode this buys out.

## Consequences

- M2 exit criterion 2 is mechanically checkable: generate against the seeded fixture graph,
  assert one typed method per registry function, run tsc/mypy over the output in CI.
- Two emitter template sets to maintain (TS + Python) — accepted; the IR keeps them from
  diverging semantically, and a golden-file test per emitter pins output drift.
- `sh:or` / complex SHACL constraint shapes map to the IR conservatively (union → `Union`/type
  alias; unmappable constraint ⇒ generation error naming the shape, not a silent `Any`).
- Function methods are **definition-typed but not executable** in M2 and v1.0 (CE ADR-009:
  execution is post-v1) — generated method bodies raise a typed `NotExecutableUntilPostV1` error
  carrying the `fn_iri`, so the surface is real, compilable, and honest.

## Alternatives Considered

- **LLM-generated SDK (agent writes the bindings)** — non-deterministic, unverifiable against the
  exit criterion, burns run budget per regeneration. Rejected.
- **Per-language independent pipelines (no shared IR)** — two places for every mapping rule;
  TS and Python drift is guaranteed. Rejected.
- **AST builders instead of templates** (ts-morph / Python `ast`) — heavier to maintain than
  Jinja2 for straight-line generated code; templates are diffable and reviewable. Rejected.
- **Publish to npm/PyPI registries** — contradicts client-owned/forkable portable-source
  contract; adds registry credentials and supply-chain surface. Rejected.
