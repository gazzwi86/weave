# ADR-021: TASK-003 implementation decisions

## Status

Accepted (TASK-003, EPIC-011).

## Context

ADR-005 pins the retrieval *strategy* (deterministic seed + weighted k-hop, 200-node cap,
investigator overflow) and TASK-003's brief supplies pseudocode, but three implementation-level
gaps had no answer in either document.

## Decisions

**1. `predicate_class` ships structural (explicit set) + annotation (fallback) only -- no
associative classification yet.** The brief's Implementation Hints name the eight structural BPMO
predicate IRIs explicitly and pin the fallback rule ("unknown predicate ⇒ annotation weight, logged
once"), but supply no associative IRI set. Hand-inventing one would violate
`ontology-standards.md`'s "never hand-copy the served kind/predicate list" rule -- there is no
authoritative associative list to copy from within this task's scope (Law 8: brief-only reading).
The `associative: 0.5` weight is still resolved from `PLAT-SETTINGS-1` (AC-4 is satisfied), it is
simply unrouted: no predicate currently classifies as `associative`. Every non-structural edge
degrades to `annotation` (0.1), which is a strictly *more* conservative ranking than a correctly
tuned associative weight would produce (under-ranks rather than over-ranks). **Upgrade path:**
thread `GET /api/ontology/types`' predicate-class field through once that endpoint carries one; a
v1.0/M2-follow-up concern, same framing as ADR-005's embedding-retrieval upgrade path.

**2. Investigator summaries reuse `dep_summaries` via its existing `content` JSONB column -- no
migration.** ADR-005's own text says "reuse `dep_summaries` ... no new table," but its pseudocode
sketch (`kind="investigation"`) reads as if `kind` were a queryable SQL column. No AC actually
requires `kind` to be filterable at the SQL level: AC-6 returns the summary from the in-memory
dispatch result, and AC-7's isolation is already enforced by the table's existing
`tenant_id`-scoped RLS policy (migration 0012). `build/investigator.py::save_investigation_summary`
therefore keys on `task_id = <investigation pointer>` (unique per investigation, so the table's
existing `(project_iri, task_id, tenant_id)` primary key still holds as an upsert target) and
stores `{"kind": "investigation", "pointer", "summary"}` inside `content`. If a future task needs
`kind` as an indexed/queryable column, that is a trivial additive migration then -- not warranted
by this task's AC surface. This keeps the migration block reserved for this lane (0031+) unused.

**3. The investigator's Sandbox-class principal is a lightweight in-process descriptor
(`InvestigatorPrincipal`), not a real AWS STS-backed `PLAT-IDENTITY-1` principal.**
`auth/agent.py`'s real agent principals authenticate via STS `GetCallerIdentity` against an IAM
role ARN; there is no real agent runtime in this task's scope for such a principal to
authenticate -- `agent_run_fn` is an injected, test-stubbed seam (Law F), same as
`orchestrator.py`'s `dispatch_pdac_fn`. What actually enforces "read-only, no write tools" today is
`READ_ONLY_TOOLS`, the explicit tool allowlist passed to `agent_run_fn` and asserted directly in the
spawn-guard unit test (Implementation Hints: "assert the allowlist ... not just the guard flag").
**Upgrade path:** once a real agent runtime task exists, wire `InvestigatorPrincipal.iri` to a real
STS-minted Sandbox-class credential the way `auth/agent.py` does for ordinary agent principals --
the descriptor's shape (`iri`, `tools`) does not need to change for that swap.

## Consequences

- No schema drift for this task: `dep_summaries`' shape and RLS policy are untouched.
- The predicate-classification ceiling is conservative (annotation-weighted, not mis-weighted) and
  documented, not silently wrong.
- The Sandbox principal is not yet a security boundary in the AWS-IAM sense -- the tool allowlist is
  the actual enforcement point today. A real agent runtime must not treat `InvestigatorPrincipal` as
  sufficient authorization on its own until the STS wiring described above lands.

## Alternatives Considered

- **Migrate `dep_summaries` to add a `kind` column now** (matches the brief's pseudocode literally).
  Rejected: no AC needs it queryable, and it is strictly additive later if a real need appears --
  smaller diff now, no schema drift to review for a column nothing reads.
- **Invent an associative predicate list from general BPMO knowledge.** Rejected: violates
  `ontology-standards.md`'s hand-copy prohibition; a mis-guessed IRI would silently mis-rank
  candidates rather than fail loudly.
