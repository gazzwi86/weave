# ADR-014: Grid derivation substitutions (owner, phase FSM, status/lifecycle-phase collapse)

## Status

Proposed

## Context

`v1-delta.md` §2 (B10) specifies the lifecycle-phase FSM as: `Speccing` (no approved
`build_specs` row yet) -> `Building` (state spine has one or more running/queued tasks) ->
`Live monitoring` (`demo_output_location_ref` is set and nothing is in flight), with `Archived`
overriding all three when `archived_at` is set. `build/store.py`'s own docstring records that
"M1 has no `specs`/`tasks` Aurora tables yet" — the spec store is in-process, not a
`project_iri`-keyed Postgres row, so "an approved `build_specs` row" cannot literally be queried
today. Separately, AC-1's EARS statement asks for "status/lifecycle-phase/owner filters" but
`projects` carries no `status` column and no `owner`/`created_by` column — those need a real
source or an explicit, documented substitution rather than a made-up column.

## Decision

1. **Speccing/Building boundary substitutes "a state-spine row exists" for "an approved
   `build_specs` row."** `state_spines` (migration 0012) is the one real, `project_iri`-keyed
   Postgres signal that dark-factory work has started on a project. No state-spine row -> a
   project that has never had a run -> `Speccing`. Any state-spine row -> `Building` (existence
   alone, not a per-task in-flight check on the `tasks` JSONB column — a run that finished all its
   tasks but hasn't published yet is still "past Speccing," and refining further needs the same
   real spec/task table this whole point is already substituting for). This is a proxy, not the
   literal spec-approval signal the tech-spec names, because that signal does not exist in
   Postgres yet. When the specs subsystem gains a real table (tracked separately, out of
   TASK-014's scope), this predicate should switch to the literal "approved build_specs row"
   check.
2. **`archived_at IS NOT NULL` overrides all three phases**, exactly as `v1-delta.md` states —
   no change needed here, `archived_at` already exists (migration 0019).
3. **`demo_output_location_ref IS NOT NULL` -> `Live monitoring`**, dropping the tech-spec's
   additional "and nothing is in flight" refinement for the same reason as point 1 (no per-task
   in-flight signal is queried) — once a project has published, it reads as `Live monitoring`
   until `archived_at` is set, even if a later re-run is briefly in flight. `demo_output_location_
   ref` already exists (migration 0016). Phase precedence:
   `archived_at` > `demo_output_location_ref` > has a state-spine row > else `Speccing`.
4. **"Owner" is the earliest-added `admin` contributor**, not a stored column — zero schema
   change, since `project_contributors.get_all` already returns oldest-added first. A freshly
   created project has no contributor rows yet (ADR-009 Decision #2: direct create "seeds
   nothing"), so its `owner` is honestly `null` until someone is added via
   `PUT .../contributors/{principal}` — this grid does not seed a contributor at create time to
   manufacture a non-null owner; that would be additional state beyond what ADR-009 already
   defines the create-shell as producing. Grid filtering by `owner=<principal_iri>` matches any
   project where that principal holds an `admin` contributor row.
   - **Separate, out-of-scope observation:** `POST /api/projects` carries no role guard today —
     any tenant member can create a project shell, but only a tenant-wide admin/owner JWT grant
     (`rbac.py::has_admin_grant`) can manage a freshly-created project's settings/contributors
     until an `admin` contributor row exists for it (a chicken-and-egg gap: nobody holds a
     per-project role on a project that has none yet). ADR-009 Decision #4 gestures at
     "create-rights should mirror auto-create's authority" without pinning down what that check
     is. This is a create-time authorization question, not a grid-display one — out of TASK-014's
     ACs, flagged here rather than silently patched via contributor-seeding.
5. **AC-1's "status" filter is the same axis as "lifecycle-phase," not a second one.** No
   `status` column exists on `projects`, and no second FSM is described anywhere in the cited
   specs. The grid accepts a single `lifecycle_phase` query parameter (values: `Speccing`,
   `Building`, `Live monitoring`, `Archived`); `status` is treated as EARS-statement phrasing for
   the same concept, not a distinct filter, and is not implemented as a second parameter.

## Consequences

- The phase-derivation predicate is a documented proxy, not the literal tech-spec signal. A
  project mid-generation-run and thus `Building` is derived correctly; a project with an approved
  spec but no run started yet would incorrectly show `Speccing` under this proxy where the
  literal FSM would show something spec-driven — this gap closes automatically once the specs
  subsystem persists to Postgres, no grid-code change needed then beyond swapping the predicate.
- A freshly created project's `owner` grid field reads `null` until an `admin` contributor is
  added — no contributor-seeding at create time, consistent with ADR-009 Decision #2's "direct
  create seeds nothing." The separate create-time-authorization gap this surfaces is flagged
  above, not fixed here.
- If a future task introduces a real `status` concept distinct from lifecycle phase, this
  decision's point 5 is the one to revisit — it is not encoded anywhere else.

## Alternatives Considered

- **Add an `owner_iri`/`created_by` column to `projects`** — rejected for now: the contributor
  table already carries this information once creator-seeding lands (point 4), so a redundant
  column would need to be kept in sync with the contributor table for no benefit.
- **Escalate the `build_specs`-table gap as a TASK-014 blocker** — rejected: `build/store.py`
  already documents this as a known, accepted M1 limitation (not something TASK-014 introduces),
  and a grid-display proxy with a documented, easily-swapped predicate is a normal fail-safe
  substitution, not a structural blocker on the order of ADR-013's IRI-grammar mismatch.
