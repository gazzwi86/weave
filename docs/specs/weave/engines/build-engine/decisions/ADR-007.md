# ADR-007: Standards catalogue — Aurora documents, Policy-linked, per-key override

## Status

Proposed

## Context

FR-063: company/workspace-level standards (coding standards, architecture patterns, preferred
stacks — the family of this repo's `docs/standards/`) are consumed by every build project in the
workspace at generation (E8-S1) and drive the generated stack (Build is stack-agnostic, decision
B8). FR-064: project-level standards extend/override company standards, tighter-wins, aligned to
the `PLAT-SETTINGS-1` cascade. Both are governed by `Policy` entities read via `CE-READ-1`
(`governedBy`) so agents can reason over them. Brand/voice stays `CE-BRAND-1`. Undecided: where
the documents live, what "override" means mechanically, and how much structure a standard has.

## Decision

1. **Storage: one Aurora table, standard Build patterns.** `standards_documents` —
   `(tenant_id, workspace_id, scope, project_id NULL unless scope='project', standard_key, title,
   body_md, stack_pins JSONB NULL, policy_iri, status draft|active|retired, created_by,
   updated_at)`, unique on `(tenant_id, workspace_id, scope, coalesce(project_id), standard_key)`.
   RLS + repo-layer base filter, exactly like every other Build table (D3). Standards are
   documents authored by humans for agents — Aurora rows with markdown bodies, not RDF individuals
   and not repo files (no PR machinery inside Weave-the-product exists for tenant docs).
2. **Graph linkage is the `policy_iri`, nothing more.** Each active standard names the CE `Policy`
   entity that governs it; Build validates the IRI resolves via `ce_client` at authoring time and
   surfaces the linked policy in prompts. The document body is not mirrored into the graph —
   the graph gets governance, agents get prose. (CE-READ-1 unavailable at *authoring* time ⇒
   the same graceful-degrade wording as FR-002/003; at *generation* time the stored IRI is used
   as-is.)
3. **Override = whole-document replacement per `standard_key`.** Effective set for a project =
   all active company-scope docs, overlaid by active project-scope docs with the same
   `standard_key` (project wins whole-key; no line-level merge — merge semantics inside prose are
   unenforceable). Keys are free-form but conventional (`stack.frontend`, `stack.backend`,
   `patterns.api`, `testing.pyramid`, ...). "Tighter-wins" is realised as "more-specific scope
   wins", the same direction as the `PLAT-SETTINGS-1` cascade.
4. **Consumption at E8-S1.** PLAN/DELEGATE prompt assembly loads the effective set (one repo-layer
   call) and injects it as the standards section of the generation context; the `stack_pins`
   JSONB (optional machine-readable pins, e.g. `{"frontend":"nextjs@15"}`) is what the generator
   actually keys stack selection off, with `body_md` as the reasoning context. Missing catalogue
   ⇒ M1 behaviour (demo-default stack) with a run-log warning — absence degrades, never halts.

## Consequences

- Zero new infrastructure; one migration + repo methods, testable with the existing two-tenant
  isolation fixture.
- Whole-key override is blunt: a project wanting to tweak one paragraph re-states the document
  under that key. Accepted — it keeps "which text applied" answerable in one row lookup, which the
  spec-coverage audit (FR-053) and ceremony evidence need.
- Effective-set resolution is trivially unit-testable (company ⊕ project fixtures).
- `stack_pins` schema is deliberately open JSONB in M2; if v1.0 wants validated pin vocabularies,
  a SHACL-backed `Policy` extension can constrain it without moving the documents.

## Alternatives Considered

- **Standards as RDF documents in the CE graph** — puts multi-page markdown prose into named
  graphs; CE-WRITE-1/SHACL adds nothing to prose, and every read costs a CE round-trip. The
  governance *link* belongs in the graph; the prose does not. Rejected.
- **Standards as files in each generated repo** — company-level reuse breaks (N copies drift);
  the catalogue must exist before any repo does. Rejected (generated repos *receive* a rendered
  snapshot; the catalogue is the source).
- **Line/section-level merge of project over company docs** — unenforceable semantics inside
  markdown; audit trail becomes "which merged text applied?". Rejected.
- **Reuse `PLAT-SETTINGS-1` itself as the store** — settings values are small typed keys, not
  multi-page documents; only the cascade *direction* is borrowed. Rejected.
