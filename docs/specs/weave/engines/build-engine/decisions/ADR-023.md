# ADR-023: TASK-017 implementation decisions

## Status

Accepted (BE-V1-TASK-017, EPIC-004).

## Context

The brief's Story section says the board/tree "reads the M1 state spine (`build_tasks`, FR-044)".
No `build_tasks` table exists anywhere in the codebase or migrations -- it is the tech-spec's
original, never-implemented table name (`tech-spec/data-model.md`). The only durable, tenant-RLS
store matching FR-044's schema (`{project_iri, phase, tasks[{id, status, blocked_by}]}`) is
`state_spines` (migration `0012`, `build/state_spine.py`), already the data source for the existing
`GET /api/state/{project_iri}` route. Same gap shape as ADR-012 (TASK-011): brief cites a spec name,
implementation uses the real table underneath it.

The Implementation Hints also say to "reuse the existing run-status pub/sub -> SSE channel the M1
Studio uses for live updates". No SSE mechanism exists anywhere in the frontend (grepped
`EventSource`/`event-stream` across `components/shell/`); the only comparable "live update"
mechanism in the codebase is 2-second interval polling (`app/build/projects/[id]/request/
use-request-status.ts`), and a sibling comment in `use-notifications.ts` states explicitly "no
polling/SSE infra for M1".

## Decisions

**1. `state_spines` is the board/tree data source, not a `build_tasks` table.** `build/board.py`'s
`build_board`/`build_task_tree` take a `StateSpine` (loaded via the existing `load_state_spine`),
mirroring `routers/runs.py`'s `get_state_route` exactly (same `tenant_connection` +
`get_current_principal` pattern, same `project_iri` path-param convention).

**2. No live-refresh mechanism is built.** None of the six ACs require push updates -- AC-1's
budget is initial-load and filter-switch latency, not staleness. Since the brief's cited SSE channel
does not exist, and no AC calls for one, the board is a plain fetch-on-mount; filter changes are
pure client-side recomputation over the already-fetched payload (this is also what makes the
"filter switch <=100ms" budget trivial to hit -- no network round-trip on filter change). Adding a
poller here would be unrequested scaffolding (Law 2/YAGNI) for a requirement that isn't in this
task's ACs.

**3. Six-lane mapping (`LANE_OF_STATUS`, `build/board.py`) -- total function, not a partial one.**
`Queued -> Backlog`, `Ready -> Ready`, `Blocked -> Review` (HITL-escalation ceiling hit --
`orchestrator._dispatch_one` -- reads as "needs a human to look at it"), `revision -> QA` (self-verify
found a violated rule, TASK-006 AC-5 -- reads as "failed the compliance check, needs rework"),
`Done -> Done`. Any unrecognised status falls back to `"Backlog"` (`lane_for_status`'s default)
rather than being dropped -- same "flag, don't drop" spirit AC-3 applies to tree orphans. The
"In Progress" lane is structurally always empty in M1: no `TaskState.status` value represents a
persisted mid-dispatch state (this task adds no new run state).

**4. Retry-chip data (AC-2) is a best-effort join against `build/store.py`'s ephemeral,
process-local `TaskRecord.retry_counts`** -- the same non-durable, single-process limitation
`typed_result.py`'s retry bookkeeping already carries (no Aurora `tasks` table exists in M1). The
failure class with the highest recorded count is shown as "current"; the ceiling is read from the
static `DEFAULT_RETRY_CEILINGS` table, not resolved per-project via `PLAT-SETTINGS-1` (that needs an
async DB call this read-mostly endpoint doesn't make -- AC-2's specified test is a component-level
unit test asserting the card renders `failure_class`/ceiling from props, not that the backend sources
them from a particular durable store, so this is scoped to what the AC actually tests).

**5. "This phase" filter (fifth of FR-017's five fixed filters) approximates against the one
per-task "phase-relevant" signal that actually exists: a task with `hold_reason` set, or
`status == "Blocked"`, is "why the run is currently in its present phase" (a halt is caused by
specific tasks). No AC's named test discriminates this filter's exact contents (only that a valid
filter resolves and an invalid one hits the AC-4 empty state), so a healthy `phase == "running"`
board with nothing held legitimately resolves this filter to zero tasks -- which correctly, honestly
exercises AC-4's empty-state path rather than being a bug.

**6. "Self-improvement-flagged" filter (fourth of the five) always resolves to zero tasks in M1** --
no such field exists anywhere in the state spine or task store. This is not a stub to silently drop;
it is the existing system's honest state, and selecting it exercises AC-4's empty-state path exactly
as designed. Add the real field/join when self-improvement flagging is built (not scoped to this
task's brief).

## Alternatives Considered

- **Add a `build_tasks` migration to match the brief literally.** Rejected: `state_spines` already
  carries the exact FR-044 shape and is the schema every other M1 build-loop route already reads --
  adding a second, poorly-synchronised table for the same data is duplication, not fidelity to the
  brief's intent.
- **Build an SSE channel or a poller for "liveness".** Rejected: no AC requires it, no existing SSE
  infra exists to reuse (the hint was inaccurate), and `use-request-status.ts`'s poller exists for a
  materially different problem (watching one in-flight draft request, not a whole board). Revisit if
  a future task's AC explicitly requires sub-page-load staleness bounds.
- **Resolve per-project retry-ceiling overrides (`PLAT-SETTINGS-1`) on the board endpoint.** Rejected
  for this task: no AC or specified test exercises the override path on the board specifically;
  `typed_result.get_retry_ceiling` remains the source of truth for the retry decision itself, this
  endpoint only displays the default ceiling as a hint.
