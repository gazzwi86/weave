# ADR-022: TASK-005 implementation decisions

## Status

Accepted (TASK-005, EPIC-008).

## Context

TASK-005's brief (Design Decisions table) states: "Reuse `generation_runs` with `run_kind='sdk'` --
no new runs table; status enum reused (`breaking_hold` maps to `hitl_escalated` family)."

Checked against the actual schema and codebase, both premises are false:

1. `generation_runs.status` (migration `0015_generation_runs.sql`) carries a hard
   `CHECK (status IN ('passed', 'failed'))` -- it cannot hold `queued`, `running`, or
   `breaking_hold` without a migration.
2. `generation_runs` has no `run_kind` column at all.
3. `hitl_escalated` does not exist anywhere in the codebase -- no status family, no HITL action,
   no constant. The brief's own claim that `breaking_hold` "maps to" it has nothing to map to.

This is a genuine brief-vs-reality gap (Engineer Law 11), escalated to and resolved by the team
lead/coordinator before implementation began.

## Decisions

**1. Widen `generation_runs` via an additive migration (0031), not a new table.** The brief's
"reuse, no new table" intent stands -- only its "status enum reused as-is" premise was wrong.
Migration 0031: drops `generation_runs_status_check` (status becomes an open TEXT enum, mirroring
`gate_results.result`'s own precedent comment -- "stays an open TEXT enum ... so both vocabularies
coexist without a migration conflict later"), adds nullable `run_kind TEXT` (`'sdk'` for this
task's rows, `NULL` for existing app-gen rows, unbackfilled), and adds `payload JSONB NOT NULL
DEFAULT '{}'::jsonb` (this task's addition, not in the coordinator's literal two-op list, needed
to persist `package_version` / `breaking_version_iris` / `failure_cause` -- AC-7's response shape
requires all three and no existing column carries them). Existing `insert_generation_run`
(hard-coded `'passed'`, no `run_kind`) keeps working unchanged.

**2. `breaking_hold` is a status value on the widened enum, released via the `feature_dispatch_held`
pattern -- not a new state machine.** No `hitl_escalated` family is introduced (there was nothing to
map to). `run_sdk_generation` sets `status = 'breaking_hold'` and fires the existing M1 HITL gate
with `task_id = f"sdk_generation:{run_id}"`. `build/hitl.py::handle_hitl_response` gets a sibling
branch to its existing `env_verification:` one: a `sdk_generation:` prefix routes `approve` actions
to `approve_sdk_breaking_ack`, which mirrors `rich_scaffold.py::approve_env_verification` 1:1 --
same D9 non-human-approver-rejected check, same deferred-import-to-avoid-circularity shape, same
"one column release, not a new FSM state" philosophy (here: one status transition, not a new table).

**3. Ack recorded via the existing `record_gate`/`gate_results` extension seam (`build/gates.py`),
reused for the third time.** No new audit/persistence pairing -- `gates.py`'s own docstring already
names this as a public, reusable seam (BE-TASK-007's `qa_suite` was the second reuser).

## Alternatives Considered

- **New `sdk_generation_runs` table + genuinely new `hitl_escalated` status family** (the brief's
  literal words, taken at face value). Rejected: contradicts the brief's own explicit intent
  ("no new table", "no new approval flow") the moment the two premises supporting that intent turn
  out to be wrong -- building the thing the brief said NOT to build, on the technicality that its
  reasoning for avoiding it happened to be inaccurate, serves nobody.
- **Leave the CHECK constraint narrowed to `passed|failed` and encode `queued`/`running`/
  `breaking_hold` some other way (e.g. a separate boolean/enum column).** Rejected: splits a single
  lifecycle across two columns for no benefit: the `gate_results` precedent for exactly this
  situation (open TEXT enum, no CHECK) already exists in this schema and is cited directly above --
  reusing it costs nothing and adds one column, not two.
