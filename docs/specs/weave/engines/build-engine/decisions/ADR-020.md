# ADR-020: Release-plan field placement + deferred population

## Status

Proposed

## Context

FR-034 (AC-5): "WHEN run reaches deploy readiness, THE SYSTEM SHALL generate
release/rollback-plan artefact ... rollout sequence, feature-flag rollback
path, approvers, target date." Pseudocode reads all four off `run`/`project`
attributes that do not exist on either model. Two brief sources disagree on
where `target_date` lives:

- Pseudocode: `target_date=run.target_date` (on the generation run).
- Design Decisions table: "Release plan = repo artefact, not DB entity —
  Markdown in repo; approvers/date project record — no new table."

The Design Decisions table is the brief's explicit resolution layer over
illustrative pseudocode (task brief authoring convention elsewhere in this
document too — Design Decisions rows are the citable rule, pseudocode is
shorthand). Taking it as authoritative: `signoff_roles` (approvers) and
`target_date` go on `projects`. `deploy_sequence` and `feature_flags` are not
mentioned in the Design Decisions row at all — by elimination, and because
they are per-deploy-run values (a rollout sequence and its flag set are
naturally scoped to one generation run, not the whole project), they go on
`generation_runs`, matching the pseudocode's `run.deploy_sequence` /
`run.feature_flags`.

**Population path is a genuine brief gap.** No API Contract in the task
brief adds an endpoint or field to write any of these four columns (the API
Contracts section explicitly says release-plan is a repo artefact with "no
new endpoints"). Per Law 11 this would normally be an escalation; the
resolution taken here (confirmed via advisor consult, not a unilateral
assumption) is: add the columns nullable now (this task's reserved migration
block), leave population to a future task, and make `render_release_plan`
honest about the gap — every section renders, but an unset field prints
`_TBD -- not yet set_` rather than a fabricated value. This keeps AC-5's
"artefact has required sections" true without inventing a write API this
brief never asked for.

## Decision

- `projects.signoff_roles JSONB NULL`, `projects.target_date DATE NULL` —
  migration `0021_release_plan_fields.sql`.
- `generation_runs.deploy_sequence JSONB NULL`, `generation_runs.feature_flags JSONB NULL`
  — same migration.
- `render_release_plan` renders all four sections unconditionally; a `None`
  value renders as `_TBD -- not yet set_`, never omitted and never
  fabricated as empty-but-present data.
- No new endpoint. Whichever future task defines how an operator sets these
  (likely a project-settings or deploy-config surface) writes the columns
  directly; this ADR's contract is the column shape, not the write path.

## Consequences

- FR-034's artefact ships on schedule (this task) even though the
  operator-facing "set your rollout sequence" UX does not exist yet — the
  plan will show `_TBD_` sections until that follow-up task lands.
- The follow-up task has a fixed target: these four columns, no schema
  negotiation needed.
- Renderer must never fall back to inventing a plausible-looking default
  (e.g. `"main"` for rollout, today's date for target_date) — that would
  violate the same honesty principle as FR-036's staleness `"unknown"`.

## Alternatives Considered

- **Escalate and block on the missing write path** — rejected: the column
  shape itself is fully determined by the Design Decisions table plus
  ordinary schema judgement (run-scoped vs project-scoped); only the write
  *API* is undetermined, and AC-5 only requires the artefact structure to
  exist, not a UI to populate it.
- **Follow pseudocode literally, put `target_date` on `generation_runs`** —
  rejected: contradicts the Design Decisions table, which this brief treats
  as the authoritative resolution layer over pseudocode elsewhere.
- **Add a minimal `PATCH` endpoint to set these fields now** — rejected:
  API Contracts section explicitly rules out new endpoints for this
  feature; scope creep beyond FR-034's stated surface.
