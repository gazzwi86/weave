# ADR-018: self-verify is a structural no-op until an M1 rule registry exists

## Status

Accepted (TASK-006, EPIC-011).

## Context

`self_verify.py` (AC-4/AC-5) checks an agent's `self_verification` block against the set of rules
applicable to its dispatch, and stops the task for revision (not Done) if any applicable rule is
violated or unaddressed. That check needs a source of "which rules apply to this agent role" --
`default_applicable_rules()` is that source.

No M1 agent-role rule registry exists yet: there is no table or config mapping an agent role (or
task type) to the list of rules it must self-report against. Building one is out of scope for
TASK-006 (per "touch only what you must") -- this task wires the self-verify *mechanism* (AC-4/AC-5),
not the *content* of what gets verified.

## Decision

`default_applicable_rules()` returns `[]` unconditionally. With zero applicable rules,
`self_verify()` always reports `compliant=True` -- every dispatch passes self-verify, but only
because nothing was asked of it, not because compliance was actually demonstrated. This is a
structural no-op, not a security or quality check, until the registry lands.

## Consequences

- **Positive:** the self-verify mechanism (data model, stop-for-revision wiring, dep-summary
  attachment) is in place and exercised by tests now, ready to enforce real rules the moment a
  registry exists -- no rework needed at that point, only populating the registry.
- **Deferred:** self-verify currently enforces nothing. It must not be read as an active compliance
  gate until the registry ships.
- **Upgrade path:** add an M1 agent-role rule registry (role/task-type -> list of rule IDs) and
  have `default_applicable_rules()` look up the dispatch's role against it instead of returning
  `[]`. Tracked as follow-up work, not part of TASK-006.
