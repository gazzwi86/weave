# ADR-011: v1 PM tables use `tenant_id TEXT` / `project_iri TEXT`, not the spec's literal UUID types

## Status
Accepted (TASK-010, EPIC-002, engineer-authored during implementation)

## Context
`v1-delta.md` Section 4 pins DDL for four new tables (`project_contributors`,
`external_bindings`, `cost_events`, `project_prompts`) using `tenant_id UUID`
and `project_id UUID` on three of the four, while `cost_events` in the same
DDL block already uses `project_iri TEXT` -- an internal inconsistency in the
spec text itself.

The brief's own instruction is to "copy the RLS policy + migration idiom from
`0013_gate_results.sql`" verbatim. That idiom is:

```sql
CREATE POLICY tenant_isolation ON <table>
    USING (tenant_id = current_setting('app.tenant_id', true));
```

`current_setting` returns `text`. A `UUID` column would force a `::uuid` cast
in the policy, which is a different policy shape than 0013's -- so "verbatim
UUID types" and "copy 0013's idiom verbatim" cannot both be satisfied.

Further evidence against the literal UUID types:

- Every `tenant_id` value in the existing test suite is a non-UUID string
  (e.g. `f"tenant-a-{uuid.uuid4().hex[:8]}"`), set via `set_config` as a text
  parameter. A UUID column rejects every value the existing tests use.
- `0010_task_briefs.sql` documents this exact convention explicitly:
  "`tenant_id` is TEXT (not UUID) to match every other tenancy-scoped table
  in this schema ... Cognito's `tenant_id` claim is a string, not necessarily
  a UUID literal."
- `0014_fix_layout_tenant_text.sql` is a prior migration that made this exact
  UUID -> TEXT fix once already, after `0008` shipped with `tenant_id UUID`.
- There is no `projects.project_id` surrogate key anywhere in this schema --
  `projects` (migration `0009`) has `project_iri TEXT PRIMARY KEY`. A
  `project_id UUID` column in the new tables would have no table to
  reference and no way to populate from any existing caller.

## Decision
The four new tables use `tenant_id TEXT NOT NULL CHECK (tenant_id <> '')`
(matching `0013`'s idiom exactly) and `project_iri TEXT NOT NULL` (renamed
from the spec's `project_id`, matching `cost_events`'s own column in the same
spec block and every other project-scoped table in this schema). All other
structure -- primary keys, CHECK constraints, UNIQUE constraints, the
`idx_cost_events_rollup` index -- is copied verbatim from `v1-delta.md`
Section 4.

`projects.description` / `projects.archived_at` ALTERs from the same spec
block are out of scope for this task: `description` already exists
(`0009_projects.sql` line 17, would fail without `IF NOT EXISTS`), and the
brief names only "four tables + two `generation_runs` columns".

## Consequences
- Migration `0018_v1_pm_tables.sql` is not a byte-for-byte copy of the spec
  DDL. Anyone diffing against `v1-delta.md` literally will see a type/column-
  name mismatch; this ADR is the record of why.
- `v1-delta.md` Section 4 should be corrected to `TEXT`/`project_iri` at the
  next spec-maintenance pass, so future tasks reading it don't re-hit the
  same contradiction.
- No behavioural risk: every repo-layer function and every test in this
  codebase already treats `tenant_id`/`project_iri` as opaque strings, so
  this brings the new tables in line with 100% of the existing schema rather
  than introducing a second convention.

## Alternatives considered
- **Follow the spec literally (UUID types).** Rejected: cannot satisfy
  simultaneously with the brief's other instruction to copy 0013's RLS idiom
  verbatim; would reject every existing tenant_id/project_iri value in the
  test suite; contradicts `cost_events`'s own column in the same DDL block.
- **Escalate as a blocking ambiguity (Law 11).** Rejected: the evidence is
  one-sided and already documented in-repo (0010's header comment, 0014's
  prior fix) -- this is Law 10 (undocumented decision -> ADR) territory, not
  a genuine unresolved gap requiring human input.
