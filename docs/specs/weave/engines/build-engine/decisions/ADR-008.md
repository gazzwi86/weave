# ADR-008: Per-task cost attribution — Build-local rollup, Platform invoicing unchanged

## Status

Proposed

## Context

> *Amended 2026-07-08:* PLAT-BILLING-1 has since gained a read surface
> (`GET /api/billing/usage` — contracts.md). The "no read API" premise below is therefore
> stale, but the decision stands on the remaining ground: the billing pipeline is
> eventually-consistent (`as_of` lag semantics), structurally late for the FR-008 synchronous
> checkpoint breach check. Local rollup for halting + display, Platform for invoicing truth —
> unchanged. Also updated: FR-008's cascade is the three-level Company→Domain→Project
> (workspace level dropped 2026-07-08).

OQ-05 (per-task cost attribution across provider tiers) was deferred at M2 because no M2 exit
criterion consumed it. v1.0 consumes it three ways: the Project Dashboard budget and forecast
tiles (FR-013), the mid-run budget-cap breach halt at the next safe checkpoint (FR-008,
Company→Domain→Project tighter-wins cascade), and forecast recalibration against the
per-task token estimates every brief already carries (FR-018 / Arch Law 4). The M1 data model has
no cost column anywhere (`generation_runs`, `gate_results`, state spine — none carry usage).
PLAT-BILLING-1 defines per-token + per-run metering as the billing pipeline but exposes **no read
API in the contract**, and Platform v1 is being specced in parallel — a read-side dependency would
couple Build v1 to an unshipped surface. Breach-halting also needs synchronous accounting; an
async billing pipeline is structurally late for a checkpoint decision.

## Decision

1. **Attribution at the dispatch site.** The orchestrator already scopes every agent dispatch by
   `{tenant_id, project_iri, task_id, run_id, agent_role, model}`. On every typed agent result it
   persists one row to a new Build-owned Aurora table `cost_events`:
   `(tenant_id, project_iri, task_id, run_id NULL for non-run work, agent_role, model,
   tokens_in, tokens_out, cost_estimate_usd, recorded_at)` — RLS + repo-layer base filter like
   every Build table (D3). Token counts come from the Agent SDK response usage block; no new
   instrumentation layer.
2. **`cost_estimate_usd` is computed from a PLAT-SETTINGS-1-resolved rate card** (per confirmed
   model ID), never a hardcoded price. Every surface that shows it labels it **"estimated"** —
   Build does not assert invoice truth.
3. **PLAT-BILLING-1 emission unchanged, plus tags.** The orchestrator keeps emitting the same
   per-token metering events; they now carry `task_id` and `run_id` as additive metadata
   (non-breaking; proposed to Platform via the coordinator, not edited into contracts.md here).
   Invoicing truth and reconciliation remain Platform's domain — drift between Build's estimate
   and billed actuals is acceptable for display and is not silently reconciled by Build.
4. **Read side: one aggregate endpoint.** `GET /api/projects/{id}/costs` returns
   `{total_estimate_usd, by_task: [{task_id, tokens_in, tokens_out, cost_estimate_usd,
   brief_estimate_tokens}], burn_rate}`. Dashboard forecast = mean actual per-task cost over
   completed tasks × remaining backlog count, corrected by the brief estimates for un-started
   tasks (stated formula — the tile never shows an unexplained number).
5. **FR-008 breach check reads the local rollup synchronously** at each safe checkpoint (task
   boundary / CODIFY): resolved cascade cap vs `SUM(cost_estimate_usd)` for the binding scope.
   Breach ⇒ halt at that checkpoint + `PLAT-NOTIFY-1` budget event (existing E2 AC wording).

## Consequences

- One migration + repo methods + one endpoint; testable with the existing two-tenant fixture and
  a seeded cost_events fixture. No new infrastructure, no Platform read-API dependency.
- Two cost numbers exist (Build estimate vs Platform invoice). Accepted and labelled; the
  alternative — blocking checkpoints on an async billing pipeline — trades a display nuance for
  an operational stall.
- Rate card staleness makes estimates drift; mitigated by settings-resolution (update the rate,
  no deploy) and the "estimated" label.
- Per-agent-role granularity comes free from the dispatch context and feeds the M2 ceremony
  report's per-run aggregates without a second pipeline.

## Alternatives Considered

- **Query PLAT-BILLING-1 read-side** — no read API exists in the contract; couples Build v1 to
  the parallel Platform v1 spec; async lag breaks checkpoint halting. Rejected.
- **Columns on `generation_runs`** — a PDAC task dispatches multiple agents across retries and
  non-run work (spec drafting, replans) has no run row; a rollup column can't attribute
  agent_role/model splits. Rejected in favour of an event table aggregated read-side.
- **Project-level accounting only (status quo)** — fails OQ-05's point: forecast can't
  recalibrate against per-task brief estimates, and a runaway single task is invisible until the
  project cap trips. Rejected.
- **Build-side reconciliation against billed actuals** — makes Build a second billing system;
  reconciliation is Platform's job (single system of record principle, PLAT-AUDIT-1 analogue).
  Rejected.
