# ADR-001: BE-TASK-003 Request Studio implementation decisions

## Status

Accepted

## Context

BE-TASK-003 ("Request Studio: Intake Form & AI Spec Drafting") specifies no
new migration, and its design-decision table names Redis pub/sub for SSE
delivery with the note "No ADR yet -- first decision made here." Four
choices had to be made during implementation that the brief left open:

1. **Where to persist a `request` record.** The brief's pseudocode calls
   `aurora.execute(...)`, but the coordinator's task instructions are
   explicit: no migration in this task, and inventing one risks a
   cross-lane migration-number collision (other lanes are adding migrations
   in parallel).
2. **How to deliver SSE events reliably.** The brief's table names Redis
   pub/sub.
3. **How to call the (synchronous) Anthropic/Bedrock SDKs from an `async`
   pipeline** without blocking the event loop.
4. **How to ground the draft in CE-READ-1** when the task's own DoR
   checklist lists "tech-spec diagrams created" as an unmet blocker, and the
   existing CE SPARQL endpoint (`routers/sparql.py`) is tenant/workspace
   scoped -- not a fit for a global BPMO grounding query run before a
   project/workspace exists.

## Decision

**1. Redis-backed `RequestRecord`, not a new Postgres table.** A Request
Studio draft is provisional scratch state by the task brief's own user
story ("evaluate scope... before committing resources to a project"). A
24h-TTL'd Redis hash (`requests/store.py`) matches that lifecycle exactly:
no migration, no cross-lane migration-number collision, and `redis` (redis-py
8.0.1) was already a project dependency with no prior caller. Promote to
Postgres if/when a request ever needs to survive past its TTL, or be
queried/joined the way `projects` rows are.

**2. A Redis list, not pub/sub, for SSE delivery.** A pub/sub subscriber
only ever sees messages published *after* it subscribes. `GET
/api/requests/{id}/stream` cannot subscribe before it knows the
server-generated `request_id`, which only exists once `POST /api/requests`
has already returned and its background task has already started -- and,
for a fast (or test-mocked) drafting pipeline, the whole draft may already
have *finished* by the time a client asks for the stream. A Redis list
(`RPUSH` + `LRANGE`) makes every event replayable in order at any time, at
the cost of a short poll interval (150ms) instead of a push -- well within
AC-4's 5s first-token budget. This satisfies the same delivery guarantee
the brief's pub/sub note was reaching for, without the race.

**3. `asyncio.to_thread` for LLM calls.** Both `AnthropicProvider.complete`
and `BedrockProvider.complete` (`ai/providers.py`) are synchronous SDK
calls. `run_drafting_pipeline` is `async` and drafts three sections
sequentially; without offloading, a slow LLM call would block the event
loop from serving any other request (including this same request's own SSE
stream) for the call's duration. `asyncio.to_thread` is stdlib, needs no
new dependency, and is the standard pattern for wrapping a blocking call in
an async function.

**4. CE-READ-1 grounding scoped to CE-VERSION-1 pinned-version lookup
only.** The task's DoR explicitly lists the tech-spec sequence diagrams for
CE-READ-1 as an unmet blocker, and the existing SPARQL query surface is
tenant/workspace-scoped -- there is no global "read the BPMO ontology before
a project exists" endpoint to call yet. `_ground_in_ce_read` therefore
reuses `projects.ce_version_client.get_pinned_latest_version` (the same
CE-VERSION-1 client BE-TASK-001 proved out) to resolve and pin an ontology
version, records a static (not executed) `GROUNDING_QUERY` constant in the
PLAT-AUDIT-1 payload for provenance, and degrades to `graph_context:
"unavailable"` if CE is unreachable -- satisfying AC-2's audit requirement
and AC-7's degrade-gracefully requirement without inventing a new
cross-engine contract ahead of its own tech-spec.

Additionally, AC-4's wording is internally inconsistent (timeout paragraph
says "mark timed_out" but also "status: partial"); this implementation
follows the explicit unit-test name (`test_stream_times_out_at_60s_and_
fires_notify`) and sets `status="timed_out"` while still preserving the
partial `draft_content` collected before the timeout fired.

## Consequences

- A request record's data is available for at most 24h; nothing downstream
  should assume durability past that window.
- If Request Studio drafts later need to be audited/joined at the SQL
  level, this decision must be revisited (promote to Postgres, define a
  migration through the normal per-lane process).
- Once CE-READ-1's tech-spec diagrams land and a global (pre-project)
  ontology-read contract exists, `_ground_in_ce_read` should be revisited
  to actually execute `GROUNDING_QUERY` against it, rather than only
  pinning a version.
