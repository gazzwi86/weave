# ADR-003: BE-TASK-004 blast-radius/cost/sign-off implementation decisions

## Status

Accepted

## Context

BE-TASK-004 ("Request Studio: Blast-Radius, Cost Gate & Stakeholder
Sign-Off") lists CE-READ-1's "touches domain", "touches service", and "has
authority" queries among its Diagram References, but those diagrams are an
acknowledged unmet DoR item (same shape of gap as BE-TASK-003 / ADR-001).
Five choices had to be made during implementation that the brief left
open:

1. **Concrete BPMO predicate IRIs** for blast-radius (domain/service) and
   stakeholder-authority queries -- none are specced yet.
2. **Self-approval comparison direction.** The brief's hint says "compare
   [the resolved stakeholder_iri] against the submitting principal's IRI --
   fail if they match", which reads ambiguously: is it "the approver
   equals a resolved-required-stakeholder" (always true when the approver
   is a valid stakeholder, which would block all approvals) or "the
   approver equals the original submitter"?
3. **SPARQL query scope.** A `RequestRecord` has no domain/project IRI
   to scope a `run_query`-style call by, but the existing Oxigraph client
   only exposed `named_graph_iri`-scoped queries.
4. **Cost-cap resolution scope.** `PLAT-SETTINGS-1`'s cascade (Company ->
   Domain -> Project, tighter-wins; *workspace level dropped 2026-07-08*)
   needs an ancestor chain to walk, but `RequestRecord` has none.
5. **Cost estimate heuristic** -- no pricing model is specced for a
   not-yet-drafted spec's token cost.

## Decision

**1. Placeholder predicate IRIs, real SPARQL.** `requests/ce_read.py`
hard-codes `urn:weave:bpmo:touchesDomain`, `urn:weave:bpmo:touchesService`,
and `urn:weave:bpmo:hasAuthority` as the predicates for blast-radius and
authority queries. The queries themselves (VALUES-clause, single round
trip per the brief's own N+1-avoidance hint) are real and run today against
Oxigraph; swapping in BPMO's final predicate names is a one-line change
per predicate, following the same precedent as ADR-001's
`GROUNDING_QUERY`. Direction convention: the *toucher* is always the
subject (`?e <touchesDomain> ?domain`) and the *authority holder* is always
the subject (`?stakeholder <hasAuthority> ?e`) -- "the touched/authorised-
over IRI is the object" is the one convention used consistently across all
three predicates, so a future BPMO swap only needs a name change, not a
direction change.

**2. Self-approval = submitter-vs-current-approver, not
stakeholder-vs-itself.** `_check_not_self_approval` compares the current
request's `record.created_by_iri` (captured at `POST /api/requests` time
from `principal.principal_iri`) against the signing-off principal's own
`principal.principal_iri` -- i.e. you cannot approve a request you
yourself submitted. The alternative reading (stakeholder approving their
own resolved-authority record) doesn't parse: a stakeholder *is* expected
to match an entry in their own resolved-authority set, that's what makes
them a valid approver, so "reject if they match" under that reading would
reject every legitimate approval. The check applies to `approve` only, not
`reject` -- rejecting your own request bypasses no governance gate.

**3. `run_query_unscoped` via Oxigraph's `union-default-graph=true`.**
Rather than requiring every blast-radius/authority query to carry a known
`named_graph_iri`, `rdf/oxigraph_client.py` gained
`run_query_unscoped(query)`, which passes Oxigraph's non-standard SPARQL-
protocol query parameter `union-default-graph=true` so the query's default
graph becomes the union of all named graphs. This is read-only and scoped
by the query's own `VALUES ?e { ... }` clause (the caller-supplied entity
IRIs), so it cannot leak data outside the entities the caller already
named.

**4. Cost-cap resolution is company-scope-only in M1.**
`resolve_cost_cap` calls `resolve_setting(conn, tenant_id=, key=
"spec_cost_cap_usd", context_iri=company_iri(tenant_id))` -- the cascade's
root scope only, since there is no domain/project IRI on a
`RequestRecord` to start a narrower lookup from. `SettingNotFound` falls
back to `(DEFAULT_COST_CAP_USD, "company")` = `(25.0, "company")`. If a
future task attaches a request to a domain/project before sign-off,
this should be revisited to walk the full Company->Domain->Project cascade.

**5. Token-count heuristic for cost estimate.** `estimate_spec_cost`
computes `total_chars / 4 / 1000 * 0.01` (rounded to 4dp) across the
draft's sections -- a widely-used chars-per-token approximation (~4
chars/token) times a flat per-1k-token rate, with no model-specific
pricing table (none is specced). This is a placeholder heuristic, not a
billing-accurate estimate; swap for real per-model pricing once
`PLAT-BILLING-1` (or equivalent) specifies one.

## Consequences

- Blast-radius and authority queries return empty results against any
  graph that doesn't yet use these placeholder predicates -- callers must
  not assume a non-empty result means "no touches/no stakeholders"; it may
  mean "BPMO hasn't shipped these predicates yet". Revisit once BPMO's
  tech-spec fixes the real predicate names.
- `CeReadUnavailable` raised from `resolve_required_stakeholders` during
  the approval flow (as opposed to the blast-radius endpoint, which
  degrades to AC-2's `"unavailable"` shape) is **not caught** -- it
  propagates as an unhandled 500. No AC in the brief covers "CE goes down
  mid-approval"; this is a deliberate, documented gap, not a silent
  swallow. Add a matching degrade-gracefully path if this proves to be a
  real operational scenario.
- The cost cap only ever resolves at company scope until requests gain a
  domain/project binding; a domain- or project-level override set via
  `PLAT-SETTINGS-1` will not be honoured for sign-off cost gating until
  then.
