# ADR-010: Brand gate recording uses its own transaction, not the caller's

## Status

Accepted

## Context

TASK-002's brief specifies the CE-BRAND-1 conformance gate as a 6th safety gate that HALTS
(not warns) generation on nonconformance — same fail-closed shape as the other 5 M1 gates
(secret-scan, SAST, type-check, package-existence, delta-scoped mutation). It also requires the
gate's pass/fail/CE-unavailable outcome to be durably recorded (AC-1, AC-4) so an operator can
see why a generation was blocked.

The brief's pseudocode does not specify transaction semantics for that recording step. During
implementation this surfaced as a real problem, not a hypothetical one: `db/pool.py`'s
`tenant_connection` context manager wraps every connection acquire in `conn.transaction()`. A
`GateFailure` raised while the brand gate is using the caller's `conn` propagates up through
`generate_app`'s `except GateFailure` block exactly as intended — but because that whole call
runs inside the caller's `tenant_connection`, the exception unwind rolls back the **entire**
request transaction, including any gate_results/audit row the recording step just inserted on
that same `conn`. The gate would then fail correctly, but leave no record of why.

## Decision

`_default_record_brand_gate` (the default `record_brand_gate` implementation wired into
`GenerationDeps`) opens its **own** `tenant_connection`, independent of the `conn` the gate
itself uses to fetch the pass_bar setting. It writes one audit event
(`event_type="gate_result_brand"`) and one row to the existing `gate_results` table
(`build/gate_store.py`, migration 0013 — no new table), then returns. That connection commits on
its own `async with` exit, before `run_brand_gate` raises `GateFailure` and unwinds the caller's
transaction.

`record_brand_gate` stays an injected seam (`RecordBrandGate` type alias) on `GenerationDeps`
precisely so tests do not need a real independently-committing connection — they fake it with an
`AsyncMock`/closure and assert it was awaited with the right status/payload.

## Consequences

- The brand gate's result is durable even when it fails: an operator querying `gate_results`
  sees `gate="brand", result="failed"` with the score/critical-failures payload, regardless of
  the request transaction's outcome. This matches the other M1 gates' expectation that gate
  results are inspectable after a blocked generation.
- Two connections are held briefly during a failing brand-gate path (the caller's, and the
  recorder's own). This is the same shape already used elsewhere for audit durability (no new
  pattern introduced) and is bounded — the recorder's connection is released as soon as its
  `async with` block exits.
- If the independent commit itself fails (e.g. pool exhaustion), that failure is not swallowed —
  it propagates instead of the original `GateFailure`, which is an acceptable trade-off: a
  gate that cannot even record its own failure is a louder, more actionable failure than a
  silently unrecorded one.

## Alternatives Considered

- **Record after the caller's transaction commits/rolls back** (e.g. defer the insert to a
  post-response hook) — rejected: adds a new orchestration mechanism for one gate, and reopens a
  window where a crash between "gate failed" and "recorded" loses the record entirely.
- **Give `generate_app` a savepoint around each gate** so a single gate's failure only rolls back
  to the savepoint, not the whole request — rejected as out of scope for this task: it would
  change transaction semantics for all 6 gates, not just the brand gate, and the brief scopes
  TASK-002 to "adds one gate callable, not pipeline logic."
- **Do nothing (accept that failed-gate records are lost)** — rejected: directly contradicts
  AC-4 (nothing commits on a blocked generation, but the reason it was blocked must still be
  visible), which is the entire point of recording gate results in the first place.
