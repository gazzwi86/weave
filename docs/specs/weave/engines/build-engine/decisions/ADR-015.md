# ADR-015: `qa_suite` takes a `progress_cb` sink; it owns no run-log storage

## Status

Proposed

## Context

TASK-007's brief (E12-S3, FR-054) requires an integration test named
`should stream long-lane progress to run log` and states in the API contract: "long lanes
(mutation, Playwright) stream progress to the run log." No run-log table, file, or streaming
primitive exists anywhere in `packages/backend` today — that machinery is implicitly the
ceremony's (TASK-008, not yet built, `unlocks: [TASK-008]` on this task) responsibility, since the
ceremony is what owns the ≤ 10 min p95 ex-human budget this streaming exists to make legible.

Building real run-log persistence inside this task would mean inventing storage for a consumer
that doesn't exist yet, guessing at its shape, and likely rebuilding it once TASK-008 defines what
the ceremony actually needs.

## Decision

`run_full_qa_suite`'s invocation context (`QARunContext`) carries an optional
`progress_cb: Callable[[str, str], None]`. The suite calls it with `(category_name, "starting")`
immediately before running any category in `_LONG_LANES = {"delta_mutation", "a11y",
"browser_backend"}` — the categories backed by mutmut or Playwright. The suite has no run-log
storage, no file, no table of its own; `progress_cb` is a plain synchronous callback, not an
async generator or queue, so the caller decides whether "streaming" means a websocket push, a log
line, or a DB row.

## Consequences

- TASK-008's ceremony supplies the actual sink (e.g. append to a run log table, push over a
  websocket) when it wires this suite in. This ADR's contract for that wiring is exactly the
  `progress_cb` signature above — TASK-008 should not need to change `qa_suite`'s public API to
  connect it.
- Until a real sink is wired, `progress_cb=None` (the default) is a legitimate, tested
  configuration — the suite runs identically, just without progress signals. No category's
  verdict depends on whether a sink is attached.
- Only the three long-lane categories fire the callback; the six fast categories (ac_test_mapping,
  coverage, complexity, lint, perf, edge_case_extension) do not, keeping the signal meaningful
  rather than noisy.

## Alternatives Considered

- **Suite writes directly to a new `qa_run_log` table** — rejected: invents a schema/consumer
  contract that isn't this task's to define; TASK-008 would likely need a different shape once the
  ceremony's actual run-log design exists, forcing a rework.
- **No progress signal at all until TASK-008 lands** — rejected: the brief names this test
  explicitly (`should stream long-lane progress to run log`), so the suite's *interface* for it is
  in scope now, even though the storage behind it isn't.
- **`async` progress callback / queue** — rejected as unnecessary complexity for this task: nothing
  in the brief requires progress delivery to be non-blocking relative to the category run itself;
  a synchronous callback is the minimum surface that satisfies the requirement.
