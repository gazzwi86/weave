# ADR-005: BPMO subgraph retrieval under the 200-node cap

## Status

Proposed

## Context

Build grounds spec drafting (E1-S1), blast-radius (E1-S2), and PLAN prompt assembly in the tenant's
BPMO graph via `CE-READ-1`. The 200-node context cap (OQ-11) is a documented constraint; the PRD
names "retrieval/ranking strategy" as a required tech-spec task and FR-051 (isolated investigator
runs, M2) as a partial mitigation. The strategy must be deterministic enough to test (Plugin Law B)
and must not add M2 infrastructure Build does not otherwise need.

## Decision

Three-stage, fully deterministic pipeline inside the existing prompt-assembly component; all reads
via `ce_client` (ADR-001 — no raw SPARQL):

1. **Seed.** Entity IRIs explicitly referenced by the spec/task brief (request-studio grounding
   refs, task-brief `entity_iris`, write-back targets). Seeds always survive truncation.
2. **Expand.** Breadth-first k-hop expansion from the seed set through `CE-READ-1` paginated
   reads, default `k = 2` (config `retrieval.max_hops`, `PLAT-SETTINGS-1` cascade). Each candidate
   node scores `weight(predicate_class) / (1 + hops)` — predicate classes and weights come from one
   shared config table: structural BPMO predicates (`partOf`, `dependsOn`, `governedBy`,
   `realises`) weight 1.0; associative predicates weight 0.5; annotation predicates weight 0.1.
   Score of a node reachable by multiple paths = max.
3. **Truncate.** Sort by score desc, tie-break by IRI lexicographic (stable, reproducible), cut at
   200 including seeds. Emit `retrieval_truncated: true` + dropped-count into the run log and the
   prompt preamble so the agent knows the slice is partial.

**Overflow path = FR-051 investigator runs.** When an agent needs context beyond the slice, the
orchestrator dispatches a read-only investigator (own `PLAT-IDENTITY-1` Sandbox-class principal,
no write tools, cannot spawn sub-investigators). The investigator returns a pointer + ≤ 500-token
summary persisted to the tenant-scoped `dep_summaries` family store; the main agent gets the
summary, never the raw subgraph. This bounds worst-case context instead of raising the cap.

## Consequences

- Deterministic and unit-testable: same graph + same seeds ⇒ same 200 nodes; conformance test
  seeds a 300-node fixture and asserts stable selection and seed survival.
- No new infrastructure: no S3 Vectors, no embedding pipeline, no re-rank model calls in M2.
- Known ceiling: purely structural ranking can miss semantically relevant but graph-distant nodes.
  The investigator path covers this at agent discretion; if M2 telemetry shows frequent
  investigator dispatch for the same miss pattern, v1.0 revisits with S3 Vectors retrieval
  (upgrade path, not M2 scope).
- Predicate weights are a tuning knob, not code: shipped as config with the defaults above.

## Alternatives Considered

- **Embedding retrieval (S3 Vectors) now** — highest recall, but adds an indexing pipeline, a
  freshness problem (re-embed per graph version), and non-determinism in tests. Rejected for M2;
  named as the v1.0 upgrade path.
- **LLM re-ranker over candidates** — adds per-run model calls inside the 6,000-call budget and
  makes retrieval non-reproducible. Rejected.
- **Raise the cap instead of ranking** — pushes token cost up linearly and only defers the
  problem; the cap is the constraint, not the bug. Rejected.
