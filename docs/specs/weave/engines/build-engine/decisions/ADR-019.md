# ADR-019: BE-SDK-1 fetches raw Turtle shapes, not the /api/ontology/types projection

## Status

Proposed

## Context

TASK-004's mapping table (task brief "IR core") requires distinguishing `sh:datatype`
(and which XSD datatype), `sh:class`/`sh:node` (reference types), `sh:or` (unions),
`sh:in` (literal unions), and `sh:pattern` (regex-validated strings) for every SHACL
property constraint on a node shape. The task brief's pseudocode calls
`ce_client.shapes(version=pin.version_iri)` and says to cite CE-READ-1 "shapes" without
specifying the wire format (Law 8: task briefs are self-contained; no spec file beyond
the brief was read to resolve this).

The only shape-introspection surface that exists in this repository today is
`GET /api/ontology/types` (`routers/ontology.py` -> `ontology/catalogue.py::list_kinds`),
which returns `OntologyTypesResponse { kinds: [{ iri, label, properties: [{ path, name,
is_relationship, min_count, max_count, severity }] } ] }`. This is a **deliberately lossy**
projection built for one consumer (the Explorer UI's kind-card renderer) — it has no
`datatype` field and no representation of `sh:or`/`sh:in`/`sh:pattern`/`sh:class` at all.
Building BE-SDK-1's mapper against it would silently degrade every constrained field to
an untyped fallback, which AC-3 explicitly forbids ("never emit a silent `Any`/`unknown`
fallback").

## Decision

`CeClient.shapes(version_iri)` (`weave_backend/sdkgen/ce_client.py`) fetches
`GET /api/ontology/shapes?version={version_iri}` and treats the response body as the raw
SHACL shapes graph in Turtle — the same serialisation `operations/shacl.py` already loads
from `framework.shacl.ttl` on the CE side, just served over HTTP instead of read from a
local file. `ir.py::map_shapes` parses it with `rdflib.Graph.parse(format="turtle")` and
walks it exactly the way `ontology/catalogue.py` already walks the in-process shapes graph
— same library, same traversal idiom, richer output (every constraint kind, not just the
five fields the UI needs).

This endpoint does not exist yet in this codebase. It is stubbed in every test
(`httpx.MockTransport`, per the task brief's "External prerequisites... stubbed in all
tests"). When CE (a different engine, out of this task's scope) implements the real
route, whoever wires TASK-005's trigger API against a live CE instance reconciles the
exact path/query-param shape against this ADR — the contract this ADR fixes is "raw
Turtle for the pinned version graph", not the specific URL string, which is a two-line
change in `ce_client.py` if it differs.

## Consequences

- BE-SDK-1's mapper gets full SHACL fidelity without CE having to grow a second,
  richer JSON projection that would just duplicate what the shapes graph already
  expresses natively in Turtle.
- `/api/ontology/types` stays exactly as-is for the Explorer UI — no scope creep into
  a CE task from this Build-engine task.
- A future CE task that ships the real `/api/ontology/shapes` endpoint has one
  reconciliation point (this ADR + `CeClient.shapes`), not scattered assumptions across
  the generator.

## Alternatives Considered

- **Reuse `/api/ontology/types` and accept the fidelity loss** — rejected: directly
  contradicts AC-3 (no silent fallback) and AC-2 (datatype + cardinality from the
  shape's actual property constraints, not a five-field summary).
- **Have `CeClient.shapes` request the JSON projection and richen it client-side by
  re-fetching `/api/ontology/resource/{iri}` per property** — rejected: N+1 fetch
  pattern for what is one file read on the CE side; also can't recover `sh:or`/`sh:in`/
  `sh:pattern`, which don't appear in `ResourceResponse` either.
