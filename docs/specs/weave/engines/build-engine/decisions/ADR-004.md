# ADR-004: gate_results merge + pre-scaffold cascade store

## Status

Accepted

## Context

The tech-spec's `gate_results` table (`data-model.md#gate-results-table`) models the 5 M1
*safety* gates (SAST/type_check/mutation/pkg_existence/secret_scan): `run_id` FK ->
`generation_runs`, `result` a `passed|failed` enum, `command_executed` singular. TASK-007's three
gates are shaped differently: DoR/DoD key on `task_id` and return `READY|NOT_READY`/`PASS|FAIL`
plus a `failing_checks[]`/`commands[]` array; pre-scaffold keys on `project_iri` and returns
`PROCEED` plus a `findings[]` array.

Building a second, parallel table for these three gates would fork "gate result" into two
unrelated schemas for what is conceptually one concept (a gate evaluation record). Migration
0013 instead adds one `gate_results` table: `task_id`/`project_iri`/`run_id` are all nullable (a
row populates whichever key its gate uses), the variable per-gate payload
(`failing_checks`/`commands`/`findings`) lands in a single `payload JSONB` column, and `result`
stays an open `TEXT` enum (no `CHECK ... IN (...)`) so `READY|NOT_READY|PASS|FAIL|PROCEED`
coexists with the safety gates' `passed|failed` without a future migration conflict. RLS uses
`tenant_id TEXT` + `FORCE ROW LEVEL SECURITY`, matching every other tenancy-scoped table added
since 0010 (the tech-spec's UUID `tenant_id` was never followed in practice -- Cognito's claim is
a string).

Separately, the pre-scaffold gate's pseudocode reads `aurora.get_project_spec(project_iri,
tenant=...)` as if a `project_specs` (brief/PRD/roadmap/tech-spec/impl-ready presence) table
existed. No such table exists in `packages/backend/migrations/` and the task brief's own Data
Model reference only names `gate_results`, not a project-spec-presence table. This is the same
situation ADR-002 already resolved for specs/tasks: `build/store.py` gains a `ProjectSpecRecord`
module-level, in-memory, process-local dict (keyed `(tenant_id, project_iri)`), following the same
`reset_for_tests()`/swap-later-behind-the-seam pattern as `SpecRecord`/`TaskRecord`.

## Consequences

- A query joining "all gate results for a run" across the two gate families now reads one table
  instead of two -- simpler for the (not-yet-built) gate-history UI/audit views.
- `gate_results.result` has no DB-level enum enforcement; a typo'd result string is only caught at
  the Python layer (`gates.py`'s literal `"READY"`/`"NOT_READY"`/... constants), not by Postgres.
  Acceptable at M1 scope -- only this task's code writes to the table.
- Pre-scaffold's cascade-presence state does not survive a process restart and is not shared
  across replicas in M1, same as ADR-002's specs/tasks limitation -- acceptable for a pass-through
  stub gate that never blocks. A real Aurora-backed `project_specs` table (populated by the
  brief/PRD/roadmap/tech-spec generation tasks themselves) is the natural follow-up once M2
  activates cascade-blocking (FR-055).
