# ADR-009: Direct project creation is a second create path, not a relaxation of governance

## Status

Proposed

## Context

FR-007 and EPIC-002's founding acceptance criterion establish that a Build project is never
allowed to exist ungoverned: at create time, THE SYSTEM resolves the Companyâ†’Domainâ†’Project
settings cascade (tighter-wins, `PLAT-SETTINGS-1`) and pins the newest published Constitution
Engine graph version (`CE-VERSION-1`), atomically with the insert. Until now the only create
path was Request Studio's approved-request auto-create (EPIC-001 â†’ FR-007).

The user ruled (2026-07-09) that requiring a full request/blast-radius/cost-gate/sign-off flow
just to create an **empty project shell** â€” before any spec exists to request work against â€” is
an unnecessary barrier. The Registry's "New project" action should open a minimal modal
(name + description only) and create the project directly.

The risk this ADR exists to close off: it would be easy to implement "direct create" as a
second, parallel code path that skips or re-implements the governance/pin step, silently
reopening the "ungoverned window" EPIC-002 was written to prevent. That must not happen.

## Decision

1. **One shared create function, two callers.** Both the request-approval auto-create (FR-007)
   and the new direct-create endpoint (`POST /api/projects`, FR-066) call the same internal
   function. That function, and only that function, is responsible for: pinning the newest
   published CE version and resolving the Companyâ†’Domainâ†’Project governance cascade, atomically
   with the project row insert. Neither caller is permitted to insert a project row any other
   way.
2. **The two paths differ only in what seeds the backlog.** Request-approval auto-create
   also carries the approved request's spec/scope into the new project. Direct create seeds
   nothing â€” the project starts with no spec, no tasks, no repo bound, in the `Speccing`
   lifecycle phase (a derived read-time value, not a stored field â€” see decision B10's
   phase-derivation note in `build-engine.md` and `tech-spec/v1-delta.md` Â§2).
3. **Request Studio is unchanged.** It remains the only way to request *work against an
   existing project* (EPIC-001, unchanged flow, unchanged ACs). The Registry's "New request"
   action is removed â€” replaced by "New project" â€” because a request without a project to
   attach it to was the false requirement being removed, not the Request Studio flow itself.
4. **Permissions on direct create mirror the existing project-create authority.** Whoever may
   trigger `POST /api/projects` is governed by the same company/domain create-rights check
   already implied by FR-007's auto-create â€” no new role is introduced.

## Consequences

- Governance resolution and CE-version pinning cannot drift between the two create paths,
  because there is exactly one code path that performs them. A future change to the cascade
  logic (e.g. a new governance key) automatically applies to both.
- A directly-created project is a legitimate, first-class state: governed and version-pinned,
  but empty. Every v1.0 surface that reads project state (Dashboard, Kanban, Decision Log) must
  render an honest "not started" empty state for it, not an error or a stalled loading state
  (`v1-delta.md` Â§5, surface honesty rules).
- The `projects` table gains two nullable columns (`description`, `archived_at`) and no
  lifecycle-phase column â€” phase stays derived, so this ADR does not reopen the phase-storage
  question addressed separately under decision B10.
- Registry copy changes ("New project" replaces "New request") but Request Studio's own screens,
  routes, and ACs are untouched â€” this ADR does not touch EPIC-001.

## Alternatives Considered

- **A second, independent create function for the direct path** â€” rejected: this is exactly the
  shape of bug that reopens an "ungoverned window," because the two implementations could drift
  (e.g. one forgets the CE pin, or resolves the cascade at the wrong scope). A shared function
  makes that class of bug structurally impossible rather than something to catch in review.
- **Require a minimal stub request behind every direct create** (auto-generate a throwaway
  request record so the existing FR-007 path is the only insert path) â€” rejected: this keeps a
  fake request-approval artifact around for every project, which is confusing in the Decision
  Log/audit trail (an approval event that nobody actually approved) and adds a table write for
  no governance benefit over calling the shared function directly.
- **Defer governance resolution to first spec/first task** (create the shell first, resolve
  cascade + pin later, when work actually starts) â€” rejected: this is precisely the ungoverned
  window EPIC-002 was written to prevent; an unpinned, ungoverned project is a real state a user
  could observe (e.g. via the API) between create and first-work, even if briefly.
