# ADR-004: Flow-canvas framework — React Flow (@xyflow/react)

## Status

Proposed

## Context

OQ-01 asked React Flow vs a custom SVG renderer for the visual flow canvas (E3). Requirements:
DAG rendering of six node types (Trigger, Condition, Action, HITL Gate, Error Handler, End), node
inspector, zoom/pan/minimap/fit-to-view, keyboard navigation (Tab cycles nodes, Enter opens
inspector, Escape closes), aria-labels on controls, PNG export, ≤ 500 ms initial render at ≤ 20
nodes, desktop-first, and re-projection from the canonical definition within ~500 ms of an AI edit.

## Decision

Build the canvas on **React Flow (`@xyflow/react`)** inside the shared Next.js 15 SPA:

- **Typed custom nodes** — one React component per node type, rendering from (and writing back
  to, via the projection layer) the canonical definition; node data is typed against the
  definition schema (ADR-003), never a parallel canvas model.
- **Built-ins used, not rebuilt:** minimap, zoom/pan, fit-view controls, and edge routing come
  from the library; PNG export via `html-to-image` over the flow viewport.
- **Projection, not state:** React Flow's node/edge state is derived from the canonical definition
  (`definition_rev`-keyed); a canvas edit dispatches a definition transaction and re-derives —
  the canvas never holds authoritative flow state (FR-011).
- **Accessibility:** the WCAG 2.1 AA / axe gate applies to the surrounding Builder surfaces;
  canvas keyboard navigation (Tab/Enter/Escape) is implemented over React Flow's focusable nodes
  per the NFR. Exact input bindings and node-rendering detail remain **deferred to the design
  spec (OQ-08)** — this ADR fixes only the framework.
- Note: Graph Explorer's `GE-CANVAS-1` (Cytoscape force canvas) is a different job — org-wide
  graph analytics vs flow editing — so no component is shared beyond design tokens; the boundary
  is already drawn in the brief (out-of-scope: `GE-CANVAS-1`).

## Consequences

- n8n-grade interactions arrive from a maintained MIT library (n8n's own editor is built on it);
  engineering effort concentrates on the projection/transaction layer, which is the actual product
  risk (FR-011 LWW, cycle detection, diff summaries).
- One new frontend dependency — mainstream, typed, actively maintained; acceptable under Law A
  (common-stack tool equivalent, no bus-factor exotic).
- Cycle/disconnection validation (FR-010) runs on the canonical definition (shared with backend
  activation validation), not on React Flow state — one implementation, imported by both sides.

## Alternatives Considered

- **Custom SVG renderer** — full control, but months of interaction engineering (pan/zoom/hit
  testing/minimap/edge routing) for zero product differentiation; violates complexity budget
  (Law E) and minimum-code.
- **Cytoscape.js (as used by Graph Explorer)** — graph-analytics layouting, poor fit for
  hand-arranged editable DAG flows with typed node inspectors; sharing it would couple two
  engines' UI roadmaps for a superficial reuse win.
