# ADR-002: Agent-run runtime binding — Bedrock AgentCore Runtime + registry-resolved artefacts

## Status

Proposed

## Context

`EA-AUTOMATION-1`'s complex tier (Agent Run actions, E5-S3) was declared **resolve-before-build**:
the PRD blocks the story until the AgentCore Runtime vs ECS Fargate binding and the
artefact-resolution contract are decided (OQ-09). OQ-10 additionally asks whether per-automation
service principals scale on Cognito. The confirmed Weave stack already names "Anthropic Agent SDK
on Bedrock AgentCore" and "ECS Fargate (long-running agents)" as the two sanctioned substrates.

## Decision

1. **Runtime binding: AWS Bedrock AgentCore Runtime** executes complex-tier agent actions. The
   Anthropic Agent SDK agent is deployed as an AgentCore runtime artefact; the Events interpreter
   invokes it via the AWS SDK. ECS Fargate is the recorded fallback if an agent class exceeds
   AgentCore execution limits (revisit trigger: agent wall-clock need > the AgentCore session
   ceiling or custom-container requirements).
2. **Artefact-resolution contract (v1):** an Agent Run node references an
   `agent_artefact` registry row (Aurora, tenant-scoped RLS):
   `{artefact_id, name, semver, agentcore_runtime_arn, entrypoint, model_id, default_timeout_s}`.
   The interpreter resolves the node's `artefact_ref` → newest compatible semver at activation and
   pins it in the published definition (same pin-at-activation semantic as the ontology version
   pin). Phase-2 exported artefacts (`pip`-installable, OQ-04) resolve through this same registry —
   the registry is the contract, the packaging is a fast-follow. Model IDs are the confirmed
   two-tier set only (`claude-sonnet-5` for agent volume; `claude-fable-5` reserved for authoring
   in the Builder, not run-time actions).
3. **Invocation is asynchronous** (consequence of ADR-001 §4): the interpreter records an
   `awaiting_agent` step state, acks the SQS message, and dispatches the AgentCore invocation; the
   completion callback (or a timeout sweep at the node's `timeout_s`, default 60 s tunable)
   re-enqueues a resume message. The SQS visibility timeout is therefore decoupled from model
   latency. A timed-out or unreachable runtime records a terminal failure; the step's completion
   marker gates re-execution on redelivery (FR-020/E5-S3).
4. **Principals (OQ-10):** each automation runs under a **per-automation least-privilege service
   principal** minted by `PLAT-IDENTITY-1` at activation; its scope derives from the grounded
   step's role/authority. The principal IRI appears in every `PLAT-AUDIT-1` event and PROV-O
   attribution as `prov:SoftwareAgent`. **Pooled-with-scoped-claims** is the recorded fallback if
   per-automation principals hit Cognito user-pool scaling limits — the switch is a
   `PLAT-IDENTITY-1` implementation concern and changes no Events-side interface (the engine only
   ever handles a principal IRI).
5. **Metering:** AgentCore usage metadata (input/output tokens) is forwarded to `PLAT-BILLING-1`
   on the **per-token** dimension, alongside the run's **per-run** event — the two dimensions
   co-exist (FR-031).

## Consequences

- E5-S3 is unblocked: the resolve-before-build condition is closed by this ADR (subject to HITL
  approval of this spec).
- The engine never embeds agent code in the definition — only an `artefact_ref` — so definitions
  stay declarative JSON in both tiers and the export path stays clean.
- AgentCore quotas/limits become a monitored operational boundary; the Fargate fallback is a
  documented migration, not a redesign (the artefact row gains a `runtime: agentcore|fargate`
  discriminator if ever exercised).

## Alternatives Considered

- **ECS Fargate task per agent run** — full container control, but the engine would own image
  builds, task lifecycle, and scaling for what AgentCore provides natively; violates
  minimum-code and Lambda-primary stack guidance for the common case.
- **Synchronous invocation held by the interpreter Lambda** — couples SQS visibility timeout and
  Lambda duration to model latency; a slow agent run would starve queue throughput (rejected with
  ADR-001 §4).
- **Per-node inline agent config (no registry)** — no version lineage, nothing for Phase-2 export
  to resolve against, and agent definitions would leak into every automation definition.
