# ADR-001: Run-engine execution model — SQS + Lambda interpreter + Postgres run state

## Status

Proposed

## Context

The PRD mandates at-least-once delivery, `run_id` dedupe, per-step idempotency markers with
replay-from-last-incomplete-step (FR-029), and HITL-gated runs persisted as durable paused-run
records that are acked off the queue — never held as in-flight SQS messages (FR-029b). OQ-09 left
the runtime/state-machine model open, and the PRD refused to commit a trigger→dispatch latency
target until it was resolved. A paused run can wait business-days, far beyond any SQS visibility
timeout. Candidates: AWS Step Functions (native `waitForTaskToken` pause), a custom interpreter on
Lambda with state in Postgres, or a long-running Fargate worker.

## Decision

1. **Transport:** one SQS **standard** queue per environment for run events (`tenant_id` +
   `workspace_id` in the message envelope). Ordering is NOT guaranteed and is not relied on.
2. **Execution:** a **Lambda-hosted step interpreter** consumes the queue and executes the
   canonical definition step-by-step (interpreter-first per `EA-AUTOMATION-1`). No Step Functions:
   the interpreter over the canonical JSON definition IS the product's execution semantics, and it
   must stay portable for the Phase-2 Agent-SDK artefact export — encoding flow logic into ASL
   would create a second, non-exportable definition of truth.
3. **State:** `run` and `run_step` rows in Aurora Postgres (RLS on `tenant_id`) are the durable
   state machine. A step writes its **idempotency completion marker** (`run_step.completed_at` +
   `step_config_hash`) in the same transaction that records its outcome; a redelivered message
   replays from the first step without a marker.
4. **Waiting states ack off the queue:** a HITL pause AND an in-flight agent run (ADR-002) persist
   a waiting `run` row (`paused` / `awaiting_agent`), the SQS message is deleted, and resumption
   re-enqueues a resume message. Consequence: the SQS **visibility timeout only has to cover
   synchronous step work** — pinned by rule `visibility_timeout ≥ (max synchronous step timeout ×
   (1 + in-process retries)) + interpreter overhead`; default **300 s** (API-call step default 30 s,
   3 retries, backoff 2/4/8 s), tunable via `PLAT-SETTINGS-1`.
5. **DLQ:** exhausted-retry events persist as **Postgres `dlq_event` rows** (workspace-scoped, RLS,
   default retention 14 days, never auto-reprocessed; "Retry from DLQ" re-enqueues). A physical SQS
   DLQ backstops only poison messages that crash the consumer before it can write state. Rationale:
   the PRD's per-workspace DLQ depth metric, typed rejection reasons, and inspectable run history
   are relational queries; one physical SQS queue per workspace does not scale operationally.
   Per-workspace depth is emitted as a CloudWatch custom metric with an alarm at depth > 0.
6. **Re-derived latency budget (closes the OQ-07/OQ-09 hold):** ingest ack p95 ≤ 150 ms (ADR-005);
   trigger receipt → first action dispatched p95 ≤ **5 s default, tunable** (excludes paused/HITL
   and agent-run model latency; SQS-standard dwell is non-deterministic, so this is a measured
   budget with a perf test, not a contractual guarantee).

## Consequences

- Pause/resume, dedupe, and replay are all one mechanism (state rows + markers) — one test surface.
- The interpreter is plain Python over the canonical definition: Phase-2 export can code-generate
  from the same definition without unwinding an ASL translation.
- We own retry/backoff logic that Step Functions would have given us; it is bounded (per-step
  in-process retries + SQS redelivery) and release-gate tested.
- A Lambda crash mid-step is safe: no completion marker ⇒ the step re-executes on redelivery;
  side-effecting steps must write their marker transactionally with the side-effect record
  (best-effort steps where dedupe is impossible are documented per action type).

## Alternatives Considered

- **AWS Step Functions (`waitForTaskToken`)** — native pause/resume, but flow logic would live in
  generated ASL: a second execution model to keep in sync with the canonical definition, hostile to
  the Phase-2 portable export, and per-transition pricing on high-volume simple-tier runs.
- **Long-running Fargate worker** — persistent consumer, but adds always-on infra the simple tier
  does not need; Lambda + SQS scales to zero and matches the confirmed stack's Lambda-primary rule.
- **Holding paused runs as in-flight SQS messages** — rejected outright: visibility timeout caps at
  12 h; business-day pauses would be lost (PRD FR-029b names this failure).
