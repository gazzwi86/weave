# ADR-005: Webhook ingestion — API Gateway + Lambda → SQS, Redis rate limits

## Status

Proposed

## Context

OQ-07 asked API Gateway + Lambda vs a dedicated high-throughput ingestion service, and held the
trigger-latency NFR open until the transport was chosen. The webhook endpoint is the engine's only
unauthenticated public surface, so the security requirements are strict (E4-S1): opaque
server-side tenant+automation token in the path (tenant NEVER inferred from the body),
HMAC-SHA256 required for write/external automations (secret in AWS Secrets Manager), size cap
(default 256 KB, tunable), schema check against the inferred/declared event schema, per-endpoint
rate limiting (default 100 req/min, tunable), and typed DLQ routing for every rejection class.

## Decision

1. **Transport: API Gateway (HTTP API) + a thin ingest Lambda → SQS** (ADR-001 queue). The
   Lambda does, in order: token → tenant/automation resolution (opaque token lookup, indexed,
   fail-closed 404 on unknown token with no existence oracle), size cap (reject before body
   parse where Content-Length declares oversize), HMAC-SHA256 verification against the Secrets
   Manager secret (constant-time compare) where required, per-endpoint rate limit
   (**ElastiCache Redis counter**, fixed 60 s window — Redis is already in the stack; API Gateway
   usage plans cannot key on a dynamic path token), schema match, then enqueue. Rejections write a
   typed `dlq_event` row (`signature_invalid` / `unknown_endpoint` / `payload_too_large` /
   `schema_mismatch` / `rate_limited`).
2. **Ack contract:** the endpoint returns `202 {run_id}` on enqueue — it never executes the
   automation inline. Ingest ack target: **p95 ≤ 150 ms** (token lookup + HMAC + enqueue only).
3. **Re-derived end-to-end latency (with ADR-001):** trigger receipt → first action dispatched
   **p95 ≤ 5 s (default, tunable)**, excluding paused/HITL runs and agent-run model latency.
   SQS-standard dwell is non-deterministic; this is a perf-tested budget, not a delivery
   guarantee — exactly the honesty the PRD demanded when it refused a hard 2 s target.
4. **Scale posture:** at mid-market volumes (≤ ~200 sustained rps per environment) this
   serverless path needs no dedicated service. Revisit trigger recorded: sustained ingress
   > 500 rps or p99 ack > 400 ms ⇒ evaluate a Fargate ingestion tier; the SQS boundary means the
   interpreter side is unchanged by any such swap.

## Consequences

- Zero always-on ingestion infrastructure; the security checks live in one auditable Lambda
  (single choke point for the webhook trust boundary — mirrors CE's single-rewriter philosophy).
- Rate limiting is best-effort-accurate (Redis fixed window) rather than API-Gateway-enforced;
  acceptable because the limit is an abuse guard, not a billing meter.
- Cron triggers reuse the same envelope: EventBridge Scheduler → the same SQS queue, so scheduled
  and webhook runs share the entire downstream spine.

## Alternatives Considered

- **Dedicated ingestion service (Fargate)** — justified only at throughput this engine's target
  segment does not generate; standing infra + deploy surface for no current requirement (YAGNI,
  Law E). Kept as the recorded scale escape hatch.
- **API Gateway usage plans / WAF rate limiting** — cannot rate-limit per dynamic endpoint token
  (per-automation), which is the granularity the PRD specifies; WAF remains available as an
  additional coarse layer at deploy time.
- **Lambda function URLs (no API Gateway)** — fewer moving parts but loses managed throttling,
  access logging, and future custom-domain/WAF attachment; API Gateway HTTP API is the cheaper
  managed middle ground.
