# ADR-003: Automation storage + tenant isolation — Aurora Postgres JSONB with RLS

## Status

Proposed

## Context

OQ-02 asked whether the canonical automation definition is a JSON document (DynamoDB) or
structured SQL (PostgreSQL), given compliance-query needs. OQ-11 asked for the final multi-tenant
isolation mechanism (store-per-tenant vs named-graph + query-rewriting), with the cross-tenant-read
expectation and test already fixed in PRD §2.2. The canonical definition is a single JSON document
(FR-011 — canvas and chat are projections of it), but the registry, run history, and compliance
report all need relational filters (status, grounded IRI, pinned version, actor class, date range).

## Decision

1. **Aurora PostgreSQL Serverless v2** (already in the confirmed stack — no new datastore) holds
   all engine state. The canonical definition is a **JSONB column** on the `automation` table;
   registry- and compliance-critical fields (`status`, `grounding_iri`, `grounding_kind`,
   `pinned_version_iri`, `tier`, `trigger_types[]`) are **extracted relational columns** maintained
   in the same transaction as the JSONB write, so list/filter/report queries never parse JSON at
   query time. Activation publishes an immutable `automation_version` snapshot row (edit-after-
   activate creates a new draft version; runs reference the exact snapshot they executed).
2. **Optimistic concurrency:** the definition row carries a monotonically increasing
   `definition_rev`; every write (AI edit or canvas edit) is compare-and-set on it — the losing
   concurrent writer gets a 409 with the winning diff (FR-011's LWW-with-diff, enforced in one
   place at the store).
3. **Tenant isolation mechanism (OQ-11 closed): Postgres row-level security** on `tenant_id` on
   every engine table, using the same fail-closed pattern as CE's Aurora layer
   (`tenant_id = current_setting('weave.tenant_id')`; missing setting ⇒ NULL predicate ⇒ zero
   rows). The store-per-tenant vs named-graph question dissolves for this engine: **Events owns no
   RDF store** — all graph access goes through CE contracts (`CE-READ-1`/`CE-WRITE-1`), where CE's
   single fail-closed query rewriter (CE ADR-001) already enforces named-graph isolation. Webhook
   tenancy is resolved from the opaque server-side token before any tenant-scoped resource is
   touched (ADR-005). The PRD's mandatory cross-tenant-read test runs against RLS + the CE
   contract boundary.

## Consequences

- One database engine for definitions, runs, DLQ, paused runs, idempotency markers, templates, and
  buffers — one backup/RLS/migration story.
- Compliance report filters (actor class, process IRI, date range) are indexed SQL over the
  run/audit-view projections; no JSON scanning, no cross-store joins.
- Defence in depth is two independent layers: application-level tenant scoping AND database RLS —
  an application bug alone cannot leak rows.
- JSONB definitions are schema-validated in the application layer (Pydantic v2 definition schema),
  not by the database; the definition schema version rides inside the document
  (`definition_schema: 1`) for forward migration.

## Alternatives Considered

- **DynamoDB document store** — natural JSON fit, but not in the confirmed stack (Law A), weak for
  ad-hoc compliance queries (the PRD's stated concern), and would split engine state across two
  databases since run history still wants SQL.
- **Store-per-tenant (database or schema per tenant)** — strongest physical isolation but an
  operational blowup at mid-market tenant counts (migrations × tenants, connection pools ×
  tenants); CE already rejected it program-wide for the graph layer (CE ADR-001), and Events
  inherits that posture for parity.
- **Fully normalised relational definition (no JSONB)** — the canvas/chat projection model wants
  one canonical document with atomic replace-on-edit; decomposing to rows makes the
  LWW-with-version-token semantic and the Phase-2 export harder for zero query benefit beyond the
  extracted columns.
