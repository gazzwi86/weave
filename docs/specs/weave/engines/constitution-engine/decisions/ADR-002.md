# ADR-002: Provenance/version-lifecycle design choices (CE-TASK-002)

## Status

Proposed -- shipped and in production use; recorded here per Law 10 (create an ADR for any
undocumented design decision) and the task brief's own instruction to document these four choices
rather than leave them implicit in code.

## Context

CE-TASK-002 (E9-S1/S2/S3) extends CE-TASK-001's mutation pipeline with PROV-O provenance, a
draft/published version lifecycle, reliable audit delivery, and a version-diff endpoint. The task
brief left four implementation details to engineering judgement rather than pinning them down
literally:

1. How audit delivery survives a down/slow sink without blocking or rolling back the mutation that
   triggered it (AC-002-04).
2. How a PROV-O activity models a human vs. an AI-agent-initiated mutation.
3. The mechanics of a draft-to-published transition on an otherwise append-only table.
4. Whether the diff endpoint's response shape must match `contracts.md`'s CE-DIFF-1 literal
   `Node|Edge`/`{ref, kind, before, after}` shape, given the brief's simpler framing.

## Decisions

### 1. Outbox pattern for audit delivery

`operations/outbox.py` inserts a durable `audit_outbox` row (migration `0007`) in the *same*
Postgres transaction as the mutation (`enqueue`, called from `pipeline._commit`), then delivers it
via the real `HashChainAuditEmitter` in a **separate**, later transaction (`flush_pending`, called
from `routers/operations.py::apply_operations_route` right after the mutation's own
`tenant_connection` block closes).

- A per-row `conn.transaction()` savepoint isolates one row's delivery failure from the rest of the
  batch ŌĆö a broken/slow audit sink retries next flush call, never blocks or drops the row, and
  never rolls back the mutation that already committed.
- The RBAC-denial path (`_enforce_write_access`'s 403) is deliberately **not** routed through the
  outbox ŌĆö it still calls `default_audit_emitter.emit` directly, in the same transaction as the
  denial. That path has nothing to protect from rollback (there's no mutation to lose), so the
  extra durability isn't needed there, and keeping it synchronous means a denial's audit entry is
  visible immediately, not on the next flush.

### 2. PROV-O actor modelling: human vs. agent

`operations/provenance.py`'s `write_activity` takes an `actor_type: ActorType` (`Literal["human",
"agent"]`) and models the two differently in PROV-O terms: a human actor is `prov:wasAssociatedWith`
a `prov:Person` and additionally `prov:wasStartedBy` that same person (self-referential); an agent
actor is `prov:wasAssociatedWith` a `prov:SoftwareAgent` with no `wasStartedBy` triple. *(Corrected
2026-07-05 ŌĆö this paragraph previously described the opposite wiring; QA caught the doc/code
mismatch.)* **Known open question (QA, CE-TASK-002 pass):** AC-002-01's model intends
`wasStartedBy` to name a *distinct approving human* for agent-initiated activities ŌĆö the field
exists to record human oversight of agent actions, and a self-referential human `wasStartedBy` is
semantically questionable. No agent-initiated flow exists yet (single principal per request, no
separate approver identity), so the shipped wiring is unreachable either way; **revisit when
TASK-004/006 introduces agent flows** ŌĆö likely: `wasStartedBy` only when a real distinct approver
IRI exists, omitted otherwise.
`routers/operations.py` narrows the JWT's untyped `principal_type: str` claim to this literal
explicitly (`"agent" if principal.principal_type == "agent" else "human"`), never trusting an
unrecognised claim value as anything other than "human".

### 3. Draft -> published lifecycle mechanics

`graph_versions` (append-only per migration `0006`) gains a `status` column (`draft` default) and a
relaxed trigger (`0007`) that permits exactly one legitimate `UPDATE`: `draft -> published`, setting
`published_at`, with every other column and every other status transition still rejected.
`versioning.publish_version`'s `UPDATE ... WHERE status = 'draft'` is itself the concurrency guard ŌĆö
two racing publish calls can only ever have one match a row, so no advisory lock is needed the way
`mint_version`'s "read latest, bump" sequence needs one. A zero-row result is disambiguated by a
follow-up `get_version` call: no row at all is a 404 (`VersionNotFound`), an existing-but-published
row is a 405 (`VersionAlreadyPublished`, AC-002-09's exact wording: `"version is published and
immutable"`). Publishing is gated by RBAC's `"publish"` role rank (`rbac.py`'s previously-defined,
until now unused, `ROLE_RANK` tier) ŌĆö reused rather than inventing a new permission.

### 4. Diff response shape: unified triple-set, not `contracts.md`'s literal shape

`contracts.md`'s CE-DIFF-1 specifies `{added: [Node|Edge], removed: [Node|Edge], modified: [{ref,
kind, before, after}]}` ŌĆö a shape that distinguishes node vs. edge changes. `operations/diff.py`
instead diffs two `rdflib.Graph`s as flat triple sets (`Graph.__sub__` both ways) and returns
`{added: [Triple], removed: [Triple], modified: [Modification]}`, where a `Triple` is
`(subject, predicate, object)` and a `Modification` is the single-value-replacement case of a
predicate whose object set changed by exactly one value (falling back to a plain added/removed
pair otherwise). There is no `Node`/`Edge` distinction ŌĆö a node's property change and an edge's
addition are both just a changed/added/removed triple.

**Why:** the task brief authorised simplifying the diff model and documenting the deviation here.
RDF has no first-class node/edge type distinction at the triple level ŌĆö recovering one would mean
re-deriving BPMO kind/shape metadata for every changed subject just to label it "node" or "edge",
which no consumer named in the task brief's acceptance criteria (AC-002-12/-13/-14) actually needs.
The unified shape is strictly less code and is what the diff *is* at the RDF layer; a
richer node/edge-labelled shape can be layered on top later if a real consumer (Graph Explorer's
diff overlay, per `contracts.md`'s own consumer note) needs it.

## Consequences

- `routers/ontology.py`'s `GET /api/ontology/diff` returns the unified triple-set shape, not
  `contracts.md`'s literal one. Any future consumer (Graph Explorer, Build) integrating against
  CE-DIFF-1 must code against the *shipped* shape (`schemas/ontology.py::DiffResponse`), not the
  contract text verbatim, until `contracts.md` itself is amended to match or a follow-up task adds
  the node/edge distinction on top.
- The outbox's per-row savepoint means a permanently-broken row (e.g. a malformed payload that will
  never deliver) retries forever rather than being dropped or dead-lettered ŌĆö no dead-letter queue
  exists yet. Acceptable for M1 volume; a future task should add one if outbox growth becomes an
  operational concern.
- The `"publish"` RBAC tier, defined but unused since the platform's initial RBAC task, now has its
  first real caller. Any future task needing a similarly "elevated but not admin" permission should
  check `ROLE_RANK` for an existing fit before adding a new tier.
