# ADR-010: v1 ingest is materialised-copy import through CE-WRITE-1, not query-time federation

## Status

Accepted

## Context

EPIC-012 (cold-start ingest) absorbs models clients already hold — documents, ArchiMate/BPMN
exports, diagrams, relational/CSV/JSON data. Two architectures could serve the structured-data
case (E12-S4):

1. **Materialised copy** — run an R2RML/RML mapping once, commit the resulting RDF through
   CE-WRITE-1, done. The graph owns the data from that point.
2. **Query-time federation** — a virtual graph that rewrites SPARQL to SQL against the live
   source at read time (Ontop-style OBDA).

The PRD already leans materialised-copy (FR-041, Non-Goal #11) but left OQ-17 "pending ADR".
Separately, OQ-14 asked whether OCEL 2.0 event logs need a dedicated ingest mapping.

## Decision

**All EPIC-012 ingest paths produce a materialised RDF copy committed through CE-WRITE-1** —
per-row/per-element SHACL-validated, PROV-O-attributed (`prov:used` names the source artefact),
versioned with the graph like any other authored content. No query-time SPARQL→SQL federation
ships in v1; no ingest path holds a live connection to a client source after the import run
ends.

**OCEL 2.0 (OQ-14) rides the same path**: an OCEL 2.0 event log is JSON — it ingests through the
E12-S4 RML mapping layer with a Weave-supplied reference RML mapping that lands OCEL objects as
BPMO individuals (`ocel:Event` → `Event`, object types → their nearest BPMO kind, unmapped →
`Concept` flagged, event↔object relationships → `consumes`/`produces`/`performedBy` where the
qualifier matches, else flagged). This is data modelling only — no process-discovery, conformance
checking, or PQL is built on top (Non-Goal #10). No separate event-ingest subsystem exists.

## Consequences

- Positive: one mutation path (the PRD §10 risk mitigation stays intact — CI asserts no second
  path); imported data is versioned, diffable (CE-DIFF-1), and SHACL-governed exactly like
  hand-authored data; no live credentials to client systems held by CE (that is
  PLAT-CONNECTOR-1's problem, deferred).
- Negative: imported data goes stale the moment the source changes — freshness is re-import, not
  sync. Acceptable for v1: the epic's purpose is cold-start seeding, not liveness, and the PRD
  explicitly never promises zero-effort liveness (§10 risk table).
- OCEL ingest quality is bounded by the reference mapping; messy real-world logs surface as
  flagged/skipped rows with reasons (FR-041 skip-and-report), not silent drops.
- The Non-Goal #11 boundary is now crisp: anything query-time-live is out of CE entirely.

## Alternatives Considered

- **Query-time federation (Ontop-style OBDA)** — rejected for v1: introduces a second read path
  that bypasses versioning/PROV/SHACL, requires holding live client credentials, and its
  query-rewriting layer would need its own tenant-isolation proof (ADR-001 rank). Revisit
  post-v1 only if a client contract demands live freshness; it would land as a Platform
  connector concern, not a CE mutation path.
- **Dedicated OCEL ingest subsystem** — rejected: OCEL is JSON; RML already covers it. A bespoke
  parser would duplicate the mapping layer for one format with no additional guarantee.
- **Hybrid (materialise + scheduled re-sync)** — rejected for v1 as scope creep toward
  PLAT-CONNECTOR-1; re-import is manual and user-triggered in v1.
