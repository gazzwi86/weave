# ADR-004: CE core performance benchmark (CE-TASK-008 spike) findings

## Status

Accepted (measurement + escalation) -- the 100k write-path result below is
**escalated to the coordinator/human for a target decision, not silently
rescoped**. This ADR records what was measured; it does not decide the
remediation.

## Context

CE-TASK-008 is a spike: benchmark the CE core hot paths built by
TASK-001..003 (`POST /api/operations/apply`, `GET /api/sparql`,
`GET /api/ontology/types`, `GET /api/ontology/resource`) at 10k/100k/500k
triple corpus scale, against the brief's go/no-go thresholds. Harness:
`scripts/benchmarks/ce-perf/run_benchmark.py` (pytest-free httpx timing
loop, no k6 in this repo; in-process ASGI transport against the real
FastAPI app, real Postgres/Redis/Oxigraph/LocalStack via this worktree's
own docker stack). CE-003 QA's Category-8 (API perf) gap is what this
spike closes -- no response-time target existed before this.

Only the **100k corpus gates M1**; 500k is recorded for the later
Neptune-vs-Fuseki production-store decision and never fails the build (see
`GATING_CORPUS_SIZE` in the harness).

## Measured numbers

| Corpus | Write p50 | Write p95 | Write p99 | Write throughput | Read p95 (worst of 5 patterns) | vs. target |
|---|---|---|---|---|---|---|
| 10k (informational) | 540 ms | 641 ms | 780 ms | 1.85 ops/s | 111 ms | pass (n/a, not gating) |
| 100k (**gates M1**) | -- | -- | -- | -- | -- | **CRASH** (see below) |
| 500k (informational) | -- | -- | -- | -- | -- | **CRASH** (see below) |

- **10k passes both targets with margin** (write p95 641 ms vs. an 800 ms
  target if it were gating; read p95 111 ms vs. 300 ms).
- **100k does not produce a write p95 number at all -- it crashes.**
  `scripts/benchmarks/ce-perf/reports/report-100000.json`:
  `{"corpus_size": 100000, "gated": true, "pass": false, "crashed": true,
  "error": "ReadTimeout: "}`. The apply pipeline's whole-graph
  fetch/validate/serialize/PUT round trip at true 100k scale exceeds
  `oxigraph_client.py`'s hardcoded `_TIMEOUT_SECONDS = 5.0` client timeout
  on essentially every write call. This is a harder failure than "misses
  the 800 ms target" -- **the write path cannot complete at all** at this
  scale under the current client timeout. See the hotspot analysis below
  for why.
- **500k also crashes** (`WriteTimeout`), consistent with the same root
  cause at greater scale -- informational only, doesn't gate M1.
- **AC-008-04 (NL-query benchmark) skipped, not silently dropped**:
  `POST /api/query/nl` does not exist anywhere in this codebase (confirmed
  via full-repo grep) -- no CE task has built NL-to-SPARQL translation yet.
  This spike's own brief says it does not implement new CE features, so
  the harness does not invent the endpoint to benchmark it. Every report
  carries `"nl_query_p95_ms": null` and `"nl_query_skipped_reason": "POST
  /api/query/nl does not exist yet (not built by CE-001..CE-003)"`.

### A prior run of this spike produced different (wrong) numbers

An earlier pass of this harness measured a 100k write p95 of ~1.4s (a
"miss," not a crash) and a 500k run that also crashed. Those numbers are
**invalid and superseded by this ADR**: the corpus seeder's edge-generation
loop (`_seed_turtle`) had a bug that made rdflib's triple-SET silently
dedup thousands of "unique" edges away, so a nominal "100k" corpus was in
fact only a small fraction of that size (confirmed: a nominal 10k corpus
parsed back to ~1650 real triples). The seeder now derives each edge's
(subject, object) pair from a base-`n_actors` digit decomposition of the
loop index, guaranteeing uniqueness up to `n_actors**2` triples --
`packages/backend/tests/unit/test_ce_perf_benchmark.py`'s
`test_seed_turtle_produces_exactly_corpus_size_triples` is the regression
guard. This run is the corrected, trustworthy one.

## Methodology notes

**A latency confound was found and isolated, not hidden.** The first raw
run showed every write call taking ~7.7 s regardless of corpus size.
Root-caused to `operations/metrics.emit_mutation_outcome_metric`
(`pipeline.py` awaits it inline on every apply) retrying CloudWatch's
`PutMetricData` 4x against this repo's LocalStack build, which returns a
protocol-mismatch 500 ("Missing Action in request for query-protocol
service ServiceModel(cloudwatch)") -- LocalStack 3.8's CloudWatch emulation
doesn't speak the query-protocol shape this boto3 version sends. The
metric call is already best-effort (caught, logged, never fails the
mutation) -- but nothing bounded its *latency*, so failed retries silently
dominated every measurement. The harness sets `AWS_MAX_ATTEMPTS=1` (env
var, zero app-code changes) so each doomed attempt fails in ~40 ms instead
of retrying for ~7.7 s, isolating the CE pipeline's own latency, which is
what this spike measures.

**A mock-token TTL confound was also found and fixed, harness-side.** The
mock OIDC access token expires after 300 s (`ACCESS_TOKEN_TTL_SECONDS`,
mirrors real Cognito's minimum validity per ADR-001). Seeding + the
100-batch write loop at 100k/500k scale can individually approach or
exceed that TTL, which surfaced as a `401 Unauthorized` mid-run on an
earlier attempt -- not a real auth bug, an artefact of one token being
reused for the whole benchmark. Fixed by re-minting a fresh token before
each slow phase (`_mint_headers`); workspace-switch state lives in the
session store keyed by `(tenant_id, sub)`, not in the JWT, so a fresh
token needs no re-switch call.

## Hotspot analysis (true 100k corpus, isolated diagnostic)

The in-process benchmark itself can't produce phase timings once it hits
the 5 s client timeout, so this is a standalone diagnostic that raises
that timeout for itself only (not the app, not the harness) and profiles
one full apply-equivalent cycle against a genuinely-100k-triple graph:

| Phase | Time |
|---|---|
| `fetch_graph_turtle` (GET Oxigraph) | 310 ms |
| rdflib parse turtle -> `Graph` | 658 ms |
| `validate_graph` (pyshacl SHACL validation) | 29 ms |
| `Graph.serialize` -> turtle | 1,293 ms |
| `load_graph` (PUT Oxigraph, replaces the named graph) | **11,915 ms** |
| **Total** | **~14,205 ms** |

**`load_graph`'s PUT dominates at 84% of total time.** SHACL validation is
fast (29 ms) and confirmed **not** the bottleneck, ruling out "whole-graph
re-validation instead of delta" as the dominant cost (even though
`pipeline.py` does re-validate the whole draft graph on every apply,
confirmed by reading `_fetch_scratch_graph`/`_run_apply`).

A secondary observation: the *initial* seed load into an empty named graph
took only ~3.1 s for the same ~100k triples, vs. ~11.9 s to replace an
already-populated named graph with an equal-sized graph. This suggests
Oxigraph's named-graph replace semantics (clear + reinsert) cost
meaningfully more than a plain insert into empty -- worth confirming before
assuming a flat "PUT cost scales with corpus size" model.

## Recommended follow-ups (not actioned by this spike -- spike scope is
measurement only)

1. **Escalate the 100k write-path crash** to the coordinator/human: the
   apply pipeline cannot complete a single write at true 100k triple scale
   under the current whole-graph replace design and 5 s client timeout.
   Options include (a) raising `oxigraph_client.py`'s client timeout as a
   stopgap, (b) moving to a delta/patch-based commit instead of
   whole-graph fetch/serialize/replace, or (c) revisiting whether Oxigraph
   is the right store for this write pattern at this scale ahead of the
   already-planned Neptune/Fuseki decision. This ADR does not choose
   between them.
2. Make `metrics.emit_mutation_outcome_metric`'s CloudWatch call
   fire-and-forget (`asyncio.create_task`, not awaited inline) so a slow or
   broken metrics backend can never inflate mutation latency in any
   environment, local or real AWS.
3. Investigate the named-graph-replace cost asymmetry noted above --
   understand why replacing an existing ~100k-triple graph costs ~4x an
   initial load of the same size before treating either number as
   representative of production behaviour.

## Decision

Record these measurements as the CE-EPIC-010 closing spike's output.
**Do not rescope the 800 ms write target, and do not silently treat a
crash as a soft miss** -- the 100k write path is escalated to the
coordinator/human per Law 11, mirroring GE-002's AC-8 pattern. This ADR
does not choose between the follow-up options above; that choice belongs
to whoever owns the target and the store decision.

## Consequences

- CE-EPIC-010 closes with an honest, escalated gap rather than a silently
  adjusted target or a hidden failure.
- The harness (`scripts/benchmarks/ce-perf/run_benchmark.py`) is
  repeatable and crash-tolerant: a hard failure at any single corpus size
  is caught and recorded in that size's own report without aborting the
  other sizes, so re-running after any remediation lands will confirm the
  fix using the same `AWS_MAX_ATTEMPTS=1` / fresh-token methodology to keep
  known confounds isolated out of the signal.
- 500k remains an open, non-blocking informational gap, consistent with
  the same root cause as the 100k crash.
