# ADR-004: Authoring endpoints (TASK-004) — dispatch reuse, single-batch import, collision resolution, shared BPMO guard, full SHACL kind coverage

## Status

Accepted

## Context

TASK-004 adds three new HTTP surfaces (`/api/ontology/authoring/nl`, `/restrictions`,
`/import`) that all need to end up going through CE-WRITE-1's RBAC, workspace resolution,
spike-write-back guard (ADR-003), and SHACL-violation handling. Three design gaps in the
brief needed a call during implementation:

1. **Dispatch reuse.** `routers/operations.py::_run_apply` already implements every
   CE-WRITE-1 invariant. The brief doesn't say whether the new router should reimplement
   that logic or reuse it.
2. **Import batching.** The brief's implementation hint says "batch per BPMO-kind group,
   never per triple" for Turtle import, to respect idempotency-key scope. Building the
   integration tests surfaced that CE-WRITE-1 validates SHACL shapes over the whole graph
   on every commit — so a `Process` node committed before its `performedBy` `Actor`/edge
   exist always fails `ProcessShape`'s `minCount` check. Per-kind batching has no valid
   intermediate state whenever one BPMO kind requires a relationship to another (which is
   the common case: Process→Actor, Process→Activity, etc).
3. **Import collision resolution.** AC-004-11's prose names three resolutions
   (skip/merge/overwrite); the DoD's test requirements table only exercises two.
4. **BPMO-kind guard bypass (QA finding).** `validate_kind()` was only called from
   `authoring/nl_parser.py` and `authoring/imports.py` — the pre-existing raw
   `POST /api/operations/apply` (CE-WRITE-1 itself) skipped BPMO taxonomy enforcement
   entirely, so any `add_node` naming a non-BPMO `kind` committed regardless of
   authoring surface.
5. **SHACL shape coverage (AC-004-05 dependency).** `framework.shacl.ttl` only had
   `sh:NodeShape`/`sh:targetClass` for 3 of the 14 BPMO kinds — `catalogue.list_kinds()`
   (CE-READ-1's guided-form options) couldn't surface the rest.

## Decision

1. **Reuse `_run_apply` directly.** The new `routers/authoring.py` imports
   `routers.operations._run_apply` and calls it after building its own `ApplyRequest`
   (from NL-parsed ops, restriction ops, or an import plan). No RBAC/guard/violation logic
   is duplicated. `_run_apply` stays private to `operations.py`; if a third router needs it
   this pattern repeats until there's a reason to promote it to a shared module.
2. **One `ApplyRequest` per import, not one per BPMO kind.** `imports.py::plan_import`
   builds a single flat `operations` list (every new node, in any kind order, then every
   edge) dispatched as one batch. This is both simpler and provably correct given SHACL's
   whole-graph validation; it still satisfies the brief's actual constraint ("not per
   triple" — an import is now one request regardless of triple count). The stated reason
   for per-kind batching (idempotency-key scope) doesn't apply here anyway: these
   authoring endpoints never set `ApplyRequest.idempotency_key` (it defaults to `None`).
3. **`on_collision` is `skip | overwrite` only** — `ImportRequest` narrows to the two
   resolutions the DoD actually tests. `overwrite` is implemented as `update_node` (never
   a fresh `add_node`), so an existing resource's other triples survive untouched.
   "Merge" (field-level union of an existing resource with the imported one) is left
   unimplemented pending a concrete need — it wasn't in the test requirements table and
   its semantics (which triples win on a real conflict, not just an absent field) aren't
   specified anywhere available to this task brief.
4. **Guard moved to the shared pipeline, not each caller.** `validate_kind()` is now
   called once, inside `operations/pipeline.py::_apply_uncached`, for every `AddNodeOp`
   in the request — the one place every authoring surface (NL, form, import, and any
   raw `POST /api/operations/apply` caller) routes through, checked before any DB/graph
   I/O. An absolute-IRI `kind` (e.g. `owl:Restriction`, used by the min-cardinality
   restriction authoring in AC-004-06/-07) is exempted from the guard — it's OWL/RDFS
   vocabulary, not a BPMO business kind, mirroring `graph_ops.py::_expand()`'s existing
   passthrough convention for any `kind`/predicate string containing `"://"`.
   `InvalidBpmoKindError` maps to `HTTPException(400, {"error": "invalid_bpmo_kind"})`
   in `routers/operations.py::_run_apply`.
5. **All 14 BPMO kinds now have a SHACL shape.** `framework.shacl.ttl` gained 11
   label-only shapes (Event, DataAsset, Field, System, Service, BusinessCapability,
   BusinessDomain, Policy, Actor, Concept, Class), matching the existing 3 shapes'
   structure. No second (relationship/warning/info) property was added for any of
   them — nothing in this task's scope needs one, and a speculative one would be an
   untested guess at each kind's real shape. Note: `authoring/bpmo.py::BPMO_KINDS` has
   14 members (Event included), not 13 as earlier prose in this task suggested —
   the enumerated frozenset is authoritative.

## Consequences

- Adding a fourth authoring surface repeats the `_dispatch`-then-`_run_apply` call
  pattern in `routers/authoring.py`; if that keeps happening, extracting a shared
  `dispatch_apply_request(principal, operations)` helper (already close to what
  `_dispatch` in this file is) becomes worth it. Not done now — one call site duplicated
  twice doesn't justify the abstraction yet.
- If a future brief calls for `on_collision: "merge"`, it needs its own field-level
  conflict-resolution spec (this ADR explicitly does not resolve that) before
  implementation.
- Import performance: dispatching one large `ApplyRequest` for a big Turtle file means
  one SHACL validation pass over a larger graph delta than per-kind batching would have,
  but no partial-import state can ever exist (all-or-nothing commit), which matches
  AC-004-10's "validated and committed" framing better than partial per-kind commits
  would have.

## Addendum (TASK-005): a second consumer of single-batch atomicity

TASK-005's `POST /api/instances` hit the exact same SHACL-whole-graph constraint as
decision 2 above, for the same reason: a BPMO kind with a hard relationship requirement
(e.g. `ProcessShape`'s `performedBy`, `sh:minCount 1`) can never be created alone and
linked in a later request — SHACL re-validates the whole graph on every commit, so the
node would fail validation until its required edge exists in the same commit.

`AddInstanceRequest` gained a `relationships` field (predicate name -> an already-existing
target IRI), and `routers/instances.py::_dispatch` now takes a list of ops instead of one,
so instance creation builds `[add_node, *add_edges]` and dispatches it as a single
`ApplyRequest` batch through `_run_apply` — the same single-batch pattern decision 2
established for import, applied here to interactive instance creation. This is not new
architecture: no second write path, no new validation rule, just a second caller relying
on the batch being atomic (the whole scratch graph is validated as one unit; there is no
partial commit if the SHACL check fails).
