# ADR-012: R2RML/RML mapping layer — morph-kgc execution, mappings are corpus artefacts, no visual editor in v1

## Status

Accepted

## Context

E12-S4 (FR-041) imports relational/CMDB data via W3C R2RML and CSV/JSON/XML via RML,
materialised through CE-WRITE-1 (ADR-010). Three sub-decisions were deferred to the tech spec:
which execution engine runs the mappings, where mapping documents live, and how users author
them.

## Decision

1. **Execution engine: `morph-kgc`.** Python (matches the FastAPI backend — Law A common
   stack), Apache-2.0, actively maintained, and the only mature Python engine covering **both**
   R2RML and RML (plus RML-star) in one dependency. It runs in-process inside the async ingest
   worker — no JVM sidecar, no new container.
2. **Mapping documents are corpus artefacts.** A mapping is a user-supplied Turtle file; it is
   uploaded through the same `POST /api/ingest/artefacts` surface as any document, stored in the
   S3 corpus under the tenant prefix (ADR-001), and named by the ingest activity's `prov:used`
   alongside the source dataset. No new storage subsystem; re-running an import re-uses the
   stored mapping.
3. **Authoring is upload-a-file in v1.** R2RML/RML authoring is a power-user affordance
   (mirroring the op-list-not-Turtle principle: the default persona never sees it). Weave ships
   reference mappings (including the OCEL 2.0 mapping, ADR-010) as documented examples. A visual
   mapping editor is deferred — build it only if real usage shows non-experts must author
   mappings.
4. **Execution semantics** (restating FR-041 as pipeline behaviour): mapping parse failure ⟹
   job fails before any commit, store untouched; per-row output validated by SHACL, failing rows
   skipped with per-row reason; committed-vs-skipped summary consistent with the bulk-CSV flow
   (FR-030); datatype inference samples ≥ 20 rows (tunable via PLAT-SETTINGS-1). All commits go
   through CE-WRITE-1 in batches with one PROV-O activity per import run.

## Consequences

- Positive: one pip dependency covers R2RML + RML + CSV/JSON/XML sources; mapping provenance is
  free (corpus + `prov:used`); no new authoring UI to build, test, or make accessible in v1.
- Negative: morph-kgc executes with database credentials for relational sources — v1 restricts
  R2RML sources to **uploaded dumps/exports** (SQLite file, CSV extract), never live connection
  strings; live-source credentials are PLAT-CONNECTOR-1 territory (deferred). This keeps the
  no-live-credentials property of ADR-010.
- Negative: no visual editor means the E12-S4 persona is technical in v1 — accepted; FR-041 is
  P2/Should and the epic's Must story (E12-S1) serves the non-technical persona.

## Alternatives Considered

- **Ontop** — OBDA/federation-first, JVM, wrong architecture per ADR-010. Rejected.
- **RMLMapper (Java)** — reference implementation but JVM sidecar container + process-boundary
  plumbing for no functional gain over morph-kgc. Rejected on Law E (complexity budget).
- **Hand-rolled mapping DSL** — reinvents a W3C standard, breaks the "mapping basis as
  reference" note in the epic, and every client mapping becomes proprietary. Rejected.
- **Visual mapping editor in v1** — speculative; no evidence non-experts author mappings at
  cold-start (they upload documents — E12-S1). Deferred, not rejected.
