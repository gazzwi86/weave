# ADR-003: Vendor a local copy of the spike-mode write-back guard

## Status

Accepted

## Context

QA cross-task finding XT-002 requires CE-WRITE-1's `POST /api/operations/apply`
(TASK-003) to reject writes from a Build Engine run tagged `run_mode: "spike"` --
a spike run must never write back to a protected branch/graph. The canonical
guard (`assert_not_spike_write_back`) was built on `feature/BE-EPIC-006`
(BE-TASK-005) as an intentionally unwired, reusable function, with a docstring
noting CE-WRITE-1 didn't exist yet for it to gate.

At the time TASK-003 needed to wire it in, `feature/BE-EPIC-006` was still
unmerged and not an ancestor of `feature/CE-EPIC-010` -- pulling it in would
mean merging unrelated, potentially-in-flight BE work into a CE feature
branch, which is out of scope for this task and risks conflicts neither lane
owns resolving.

## Decision

Vendor a byte-for-byte copy of the guard (`SpikeWriteBackForbidden`,
`assert_not_spike_write_back`) into
`weave_backend/operations/guards.py`, wired directly into
`routers/operations.py::_run_apply`. `ApplyRequest.run_mode` (new field,
default `"normal"`) carries the caller's run mode; a `"spike"` value maps to
`403 {"error": "spike_write_back_forbidden"}`.

## Consequences

- Two copies of the same ~10-line function exist until both branches land on
  `main`. This is a known, tracked duplication -- not silent drift.
- **Follow-up**: once `feature/BE-EPIC-006` merges, delete
  `weave_backend/operations/guards.py` and import the Build Engine's copy
  (or promote the guard to a shared module both engines import, if a
  natural shared location exists by then). Log this as a QA cross-task
  finding at merge time so it isn't forgotten.
- No behavioural drift risk in the meantime: both copies implement the exact
  same one-branch check, so the two engines can't disagree on what
  `run_mode: "spike"` means while both copies exist.
