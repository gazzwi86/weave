# ADR-011: Document corpus store implementation — chunking, embedding model, retrieval ranking, citation format

## Status

Accepted (2026-07-08, higher-power reviewed). Pins 1a/2a and the revisit trigger are part of
the decision, not commentary.

## Context

Program-level [ADR-003](../../../decisions/ADR-003-document-corpus.md) decided *that* every
EPIC-012 artefact is retained (original in S3, passage embeddings in S3 Vectors, tenant-prefixed
per ADR-001, linked via `prov:used`, strictly read-side) and explicitly deferred four
implementation choices to this tech spec: chunking strategy, embedding model, retrieval
ranking, and the citation format in NL-query responses (FR-043).

Constraints: stack pins S3 Vectors as the vector store; Bedrock is the model gateway; Anthropic
models do not provide embeddings, so the embedding model is a second vendor model by necessity;
cross-tenant isolation of the vector index must be testable at ADR-001 rank.

## Decision (recommended defaults)

1. **Chunking: structure-aware, ~512-token target, 15% overlap.** Per-format **simple**
   splitters only: XML notations (ArchiMate/BPMN) split per element/process **with parent
   context prepended**; prose formats split heading-first, then paragraphs; merge/split to a
   ~512-token target with ~15% overlap. Extraction already parses document structure for the
   extractor prompt (E12-S1), so the splitter reuses that parse — no second parser. Each
   passage keeps a locator (page number or heading path + char range) for citation.
   - **1a (pin, scope guard):** a `fixed-window-with-overlap` fallback splitter is
     **mandatory** for any unrecognised format — every upload chunks, none errors on format.
     **No ML layout parsing, no PDF table reconstruction** in v1 — v-later, behind
     retrieval-quality evals. <!-- ponytail: simple splitters + dumb fallback; ceiling is PDF
     table fidelity; upgrade path is a layout parser IF evals demand it -->
2. **Embedding model: Amazon Titan Text Embeddings v2 (`amazon.titan-embed-text-v2:0`),
   1024-dim, cosine.** Bedrock-native (no new vendor contract — already the AgentCore
   gateway), supports 256/512/1024 output dims (the S3 Vectors storage-cost lever — re-embed
   cheaply at a smaller dim if the index needs shrinking), and is the low-operational-risk
   default at this corpus scale.
   - **2a (pin, reversibility):** every tenant index records `embedding_model_id` +
     `dimensions` as index metadata. A model swap is a **re-embed/re-index job, never a schema
     migration**; models are **never mixed within one tenant index**. This is what keeps
     Titan-vs-Cohere a two-way door.
   - **Revisit trigger:** a multilingual-corpus requirement landing in any PRD flips the
     default to Cohere Embed Multilingual. (Confirmed at review: no such requirement exists
     today — Onboarding i18n is interface copy, not customer corpora.)
3. **Retrieval: filtered cosine top-k, no re-ranker in v1.** Query embedding → S3 Vectors
   cosine top-k (default k=8, tunable) **always** filtered by tenant prefix (fail-closed —
   filter injected by the same request-context mechanism as ADR-001 query rewriting, never
   caller-supplied), with optional metadata filters (artefact kind, source system from FR-044
   context, date-of-truth). Consumers: the E12-S1 extractor (find related passages during
   proposal) and NL query (FR-043 citations). No cross-encoder re-ranker in v1 — corpus sizes
   at cold-start are small; add one only if precision evals fail.
4. **Citation format: additive `citations` array on the NL-query response.**
   `POST /api/query/nl` (CE-READ-1) gains an optional, additive field:

   ```json
   "citations": [
     {
       "entity_iri": "<graph IRI the answer rests on>",
       "artefact_iri": "<prov:used source artefact>",
       "passage_id": "<corpus passage>",
       "locator": { "page": 12, "heading_path": "3.2 Claims intake", "char_range": [104, 512] },
       "snippet": "<= 300 chars of the passage text>"
     }
   ]
   ```

   Every citation pairs a **graph IRI** with the **source passage** (FR-043: answers cite
   both). Field is additive-only — the base CE-READ-1 response shape is unchanged, so no
   contract-breaking delta (any contracts.md wording touch-up is proposed to the coordinator,
   not edited here).

## Consequences

- Embedding lifecycle rides ingest: embed on ingest commit, re-embed on re-ingest, delete on
  tenant deletion (ADR-003). One Bedrock model dependency added, config-pinned.
- The cross-tenant vector-isolation test (tenant-A passage must never surface in tenant-B
  retrieval) becomes release-gating, same rank as ADR-001's graph tests.
- Read-side-only is CI-asserted: no corpus module may import the mutation pipeline; the
  corpus/retrieval code path has no route to `POST /api/operations/apply`.
- Cost profile: Titan v2 is priced per input token for embedding; cold-start corpora (tens to
  hundreds of documents per tenant) keep this marginal. If corpus scale changes the economics,
  the config-pinned model + re-embed path is the escape hatch.

## Alternatives Considered

- **Cohere Embed v3 (on Bedrock)** — stronger multilingual retrieval benchmarks; higher cost
  and a third model vendor in the stack. Rejected as default for v1 (English-first cold-start
  corpora); the revisit trigger in §2a governs the flip — swap is re-embed/re-index per pin 2a.
- **Fixed-window chunking as the primary strategy** — splits mid-clause, degrading extractor
  grounding and citation quality; locator quality (heading path) is load-bearing for the trust
  story. Rejected as primary; **kept as the mandatory fallback** for unrecognised formats (pin
  1a).
- **ML layout parsing / PDF table reconstruction** — cut at review (scope guard, pin 1a);
  v-later behind retrieval-quality evals.
- **Cross-encoder re-ranking in v1** — precision uplift not yet evidenced at cold-start corpus
  sizes; adds latency to every NL query. Deferred behind retrieval-quality evals.
- **Embedding citations into the answer text (inline markers)** — brittle to parse, breaks the
  additive-contract rule. Rejected in favour of the structured array.
