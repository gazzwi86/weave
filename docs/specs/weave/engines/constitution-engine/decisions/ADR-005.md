# ADR-005: NL query + SPARQL editor implementation choices (CE-TASK-007)

## Status

Proposed -- shipped and in production use; recorded here per Law 10 (undocumented design decision)
and Law 11 (surfacing gaps rather than silently assuming).

## Context

CE-TASK-007 (NL->SELECT query + SPARQL editor) supplied a reference SPARQL query for
`coverage_gap(process)` and named the ACs/response fields, but left four implementation details
underspecified relative to what CE-TASK-003/CE-TASK-001 actually shipped.

## Decisions

1. **`coverage_gap_process`'s step label predicate is `weave:label`, not `rdfs:label`.** The brief's
   reference query reads `?step_iri rdfs:label ?step_label`, but `operations/graph_ops.py` writes
   every node's label as `weave:label` (`WEAVE.label`, confirmed by `test_ontology_catalogue.py`) --
   there is no `rdfs:label` triple anywhere in a Weave graph. Using the brief's literal predicate
   would silently return zero rows against every real graph. The shipped query uses `weave:label`
   to match the actual data model; `rdfs:label` is still imported as a prefix for future
   interoperability but unused by this pattern.
2. **`coverage_gap_process` is wrapped in `GRAPH ?g { ... }`**, like every other stored/generated
   query in this codebase (`search/sparql_search.py`, NL-generated SPARQL). `rdf/query_rewriter.py`'s
   `validate_query` rejects any SELECT without a `GRAPH` clause (AC-003 dataset-scoping invariant) --
   the brief's reference text has no such wrapper, so it would 400 `unscoped_query_rejected` as
   written.
3. **`GET /api/sparql?pattern=<name>` returns `{rows, column_names, message?}`**, a different shape
   from the existing `query=` path's `{version_iri, page, head, results}` (CE-TASK-003). The two
   response shapes coexist on one route rather than migrating `query=` callers, since AC-007-13's
   exact `{rows:[], message:"No coverage gaps found"}` shape has no `head`/`results` nesting and
   changing the `query=` shape would break CE-TASK-003's shipped contract and its passing tests.
4. **LLM boundary in Playwright E2E**: the one NL-question E2E scenario (AC-007-01/E2E) mocks
   `POST /api/query/nl` at the browser network layer (`page.route()`), the same pattern
   `tests/e2e/billing.spec.ts` already uses for its AI-touching "Simulate AI call" button. The
   `coverage_gap` and copy-to-editor E2E scenarios need no LLM at all and run against the real
   backend + Oxigraph. Python-level `ModelProvider` injection (Law F) covers the NL translation
   path at the unit/integration tier, where tests run in-process (ASGI transport, not a live
   server) and can monkeypatch the provider directly.

## Consequences

- A future pattern added to `NAMED_PATTERNS` must remember the `GRAPH ?g` wrapper or it will 400.
- If `query=` and `pattern=` response shapes are ever unified, that is a breaking change to
  CE-TASK-003's contract and needs its own task/ADR, not a silent tweak here.
