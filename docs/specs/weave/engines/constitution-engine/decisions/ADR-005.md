# ADR-005: Chat panel and guided forms (TASK-006) — preview flag, actor derivation, local history, templated explanations, undo scope

## Status

Accepted

## Context

TASK-006 builds three authoring surfaces (persistent chat panel, SHACL-driven guided
forms, AI-explains-AI) on top of TASK-003/004/005's CE-READ-1/CE-WRITE-1 contracts. Five
gaps needed a call during implementation.

## Decisions

### 1. `preview` flag on `/api/ontology/authoring/nl`, not a new endpoint

AC-006-02/-03 require propose-then-confirm: the chat panel must show the parsed operation
batch before it is dispatched. TASK-004's `/nl` route committed immediately. Rather than
add a second endpoint, a `preview: bool = false` field was added to `NlAuthoringRequest`;
`preview=true` returns the parsed `Op[]` batch (`NlPreviewResponse`) without dispatching,
`preview=false` (default) preserves TASK-004's original behaviour for any existing caller.
**Alternative rejected:** a separate `/nl/preview` endpoint — more surface area for what is
one boolean-gated code path in the same handler.

### 2. Actor derivation from the session, not a client-supplied field

CE-WRITE-1's `ApplyRequest.actor` is attribution-only (never an authz check — see ADR-003).
The Next.js proxy's `applyRequestSchema` (zod) deliberately excludes `actor` from client
input and injects `session.user?.email ?? "unknown-actor"` server-side before forwarding.
This keeps `principal_iri` off the wire to the browser entirely, at the cost of using the
session's email rather than a `principal_iri` (no route currently maps session → principal
IRI on the frontend). **Flag for QA:** if audit trails need the principal IRI specifically,
a `GET /api/whoami`-style lookup (already used by `dashboard/page.tsx`) would need to be
threaded through this proxy too — not done here to avoid an extra round trip per mutation.

### 3. localStorage schema: `weave:ce:chat:v1`

AC-006-05 requires chat history to survive a page reload via in-browser (not server-side)
persistence. A single versioned key (`weave:ce:chat:v1`) holds the full `ChatMessage[]`
array as JSON; a version suffix lets a future schema change invalidate old data by reading
a new key rather than migrating. Corrupt/missing JSON falls back to an empty history
(never throws).

### 4. AI explanations are deterministic/templated, not a second LLM call

AC-006-12/-13/-14 require a plain-language explanation to accompany every proposal and
answer "why"/"consequences" follow-ups. These are built from the operation batch's own
structured fields (kind, label, properties) and the cached SHACL shape's `min_count`/
`max_count` bounds via string templates (`explain.ts`), not a second round-trip to
claude-sonnet-5. **Rationale:** the brief's hint describes compressing a 3-hop
neighbourhood into a prompt for a real LLM explanation call; the templated version is
simpler, has zero added latency/cost, and is fully deterministic for testing. **Ceiling:**
explanations reference the shape's static bounds, not live instance counts (e.g. "this
kind allows at most 3 Owners" not "this Process already has 2 of its 3 allowed Owners").
**Upgrade path:** swap `explain.ts`'s template functions for a real LLM call with 3-hop
context if business users need dynamic, instance-aware reasoning.

### 5. Undo scope: `add_node`/`add_edge` inversion only

AC-006-04 requires "undo" to propose the inverse of the most recent applied batch.
`invertOperations` (`invert.ts`) inverts `add_node → delete_node` and `add_edge →
delete_edge` (in reverse order) using the `ref_map` returned by the original apply. It
returns `null` (dropped from the inverse batch) for `update_node`/`delete_node`, which
would require a pre-edit snapshot this task does not capture. **Rationale:** only the
add-then-undo path is exercised by the DoD's required test ("undo tested: inverse batch
correctly reverts most recent session mutation") and by the E2E suite; building a
partially-correct snapshot mechanism for the untested paths would add complexity without
verified behaviour. **Flag for QA/future work:** editing or deleting an entity via chat and
then saying "undo" currently proposes an empty batch (a no-op), not a silent wrong revert —
this is a documented gap, not a silent bug, but should be prioritised before EPIC-011 ships
edit-via-chat as a first-class flow.

## Consequences

- QA should verify the `preview` flag's default (`false`) doesn't change any TASK-004
  caller's behaviour.
- A future task threading `principal_iri` through the frontend session would let the
  actor-derivation decision (2) be tightened from email to IRI without changing the wire
  contract's shape (still a `string`).
- The undo-scope gap (5) is the main open item for EPIC-011's Phase 2 (server-side history,
  full undo/redo).
