# ADR-009: CE-FUNCTION-1 signature typing model — RDF/SHACL source of truth with derived JSON-Schema projection

## Status

Accepted

## Context

CE-FUNCTION-1 is the single registry of ontology-bound functions
(`reorderStock(System, DataAsset) → Activity`). The contract stub (contracts.md §1)
pins ownership and two read endpoints but not the typing model. Three consumers put
conflicting pressure on that model:

- **Build** codegens one typed SDK method per function (`BE-SDK-1`) — the signature
  must map mechanically to Pydantic/TypeScript types.
- **Agents** (Build/Events) invoke functions as tools — Anthropic tool schemas are
  JSON Schema.
- **CE itself** is semantic-web-native (ontology-standards rule): the registry must
  live in the graph, be SPARQL-queryable, and carry PROV-O like everything else.

A JSON-first model serves the first two and betrays the third (a second typing
system beside SHACL — the exact divergence this contract exists to prevent); a pure
RDF model serves the third and pushes a SHACL→types converter into every consumer.
Decided by stochastic-reasoning elicitation over four candidates (SHACL-referencing /
bespoke OWL vocabulary / JSON-Schema-first / SHACL + derived projection), weighted
matrix scores 93 / 73 / 84 / **106**, plus a versioning-semantics question round.
Human-confirmed outcome: **D, b, a, a, a**.

## Decisions

### 1. Typing model (D): RDF/SHACL is the single source of truth; JSON Schema is a derived projection

- A function is a `weave:Function` **individual in the tenant's graph** (grounded in
  obpm `mi-catalog`). Each parameter and the return reference a **BPMO kind IRI**
  (served by `GET /api/ontology/types`, CE-READ-1 — never a hand-copied list) and
  optionally a `sh:NodeShape` constraining accepted nodes. Written via CE-WRITE-1
  (single mutation entry point), so SHACL validation and PROV-O attribution apply
  for free.
- CE owns **one** converter that derives a JSON-Schema view of each signature
  (signature subset of SHACL only — not a general SHACL→JSON-Schema translator).
  `GET /api/functions/{iri}` returns both the RDF-level signature (kind/shape IRIs)
  and the derived JSON Schema. This is the CE-BRAND-1 pattern reused: RDF source of
  truth, JSON projection derived on read/commit, **never hand-edited, never allowed
  to diverge**.
- Build codegen and agent tool registration consume the JSON-Schema projection;
  SPARQL/graph consumers query the RDF. Neither Build nor Events writes its own
  SHACL→type mapping.
- **Drift mitigation (mandatory test):** a projection **round-trip contract test** —
  for every registered function, the derived JSON Schema must validate exactly the
  node set its SHACL shape accepts (checked over seeded fixtures). Converter bugs
  fail CI, not downstream SDKs.

### 2. Versioning (Q1 = b): functions version WITH the graph

No per-function semver lineage. Registry individuals live in the graph, so a
function revision lands as a **new ontology version** via the existing publish
pipeline; the contract's `version_iri` **is** the CE-VERSION-1 version IRI. Build
already pins generated artefacts to a CE version (BE-ARTEFACT-1 provenance header),
so SDK pinning falls out with zero new machinery.

**Accepted assumption (on record, user-acknowledged):** this assumes function
revisions are *slower or equal* in cadence to ontology publishes (catalogue-like,
per obpm `mi-catalog` grounding). If function churn ever materially outpaces graph
publishing, every tweak forces a full version publish — the revisit trigger is
"function-only publishes become a nuisance", and the upgrade path is a per-function
lineage layered over the same individuals (additive; nothing in D changes).

### 3. Immutability + breaking flag (Q2 = a)

A published signature is **immutable in-place**. A signature change is a new graph
version carrying `breaking: true` on the registry entry (param added/removed/
retyped, return changed ⇒ breaking; label/description edits ⇒ not). Build SDK
codegen **refuses to regenerate across a breaking change without explicit human
acknowledgement**. An unstated change class defaults to breaking (fail-closed).

### 4. Deprecation at M2 (Q3 = a)

Registry entries carry `status: "active" | "deprecated"`. Events must not bind
**new** automations to a deprecated function; existing references keep resolving.
One enum field, shipped M2.

### 5. M2-definition vs v1.0-full split (Q4 = a) — verbatim into the contract

- **M2 — CE surface complete:** function definition + revision via CE-WRITE-1;
  `GET /api/functions` (list: `fn_iri, name, bound_kind, signature, version_iri,
  status, breaking`); `GET /api/functions/{iri}` (full signature + grounding entity
  IRIs + derived JSON Schema); breaking/deprecation semantics live. Build **starts**
  typed-binding codegen in M2 against this surface.
- **v1.0 — execution:** invocation semantics, where implementations live, runtime
  binding, SDK generation at scale. Nothing about *executing* a function is M2.

## Consequences

- CE owns the converter (one place, tested); Build and Events own zero RDF parsing —
  the divergence risk the contract was minted for is closed structurally.
- The registry is SPARQL-queryable like any graph content ("which functions are
  bound to kind X" is a SELECT, no new query surface).
- No new versioning machinery; version-lag/staleness semantics (CE-VERSION-1) apply
  to functions unchanged.
- The churn assumption in Decision 2 is the recorded revisit trigger (see above).
- Step C amends contracts.md CE-FUNCTION-1 with: the typing model, the derived-
  projection rule, `status`/`breaking` fields, `version_iri = CE-VERSION-1 IRI`, and
  the M2/v1.0 split verbatim from Decision 5.

## Alternatives considered

- **A — SHACL-referencing only (runner-up, 93):** same source of truth, no CE
  converter — but every consumer then writes its own SHACL→type mapping, recreating
  the divergence CE-FUNCTION-1 exists to prevent. Preferable only if CE could not
  own the converter.
- **B — Bespoke `weave:Function` OWL vocabulary (73):** most expressive; but OWL
  reasoning is deferred post-v1 so the expressiveness has no consumer, and it is the
  most build cost.
- **C — JSON-Schema-first with IRI annotations (84):** perfect codegen/tool-call
  ergonomics, but a second typing system beside SHACL, not graph-queryable, and
  violates the semantic-web-native rule for the model layer.
