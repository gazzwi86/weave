# ADR-001: Named graphs are workspace-granular, not tenant-granular

## Status

Proposed -- pending human sign-off on the relationship note below (QA finding, CE-TASK-001
retry). The naming-scheme decision itself has been shipped and is in production use; what needs
human confirmation is this ADR's account of what it does and does not supersede in the
release-gating program ADR.

## Context

CE-TASK-001's task brief (citing what it called "ADR-001" for the platform) specified a flat,
tenant-only named-graph scheme: BPMO OWL classes and framework SHACL shapes in
`urn:weave:g:framework`, per-tenant custom shapes in `urn:weave:g:tenant:{id}`'s shapes subgraph,
and version IRIs of the form `urn:weave:g:tenant:{id}:v{semver}` — with an explicit instruction
not to reintroduce any flat `weave:graph/*` scheme.

Reading the already-shipped platform code (PLAT-TASK-003 tenancy, PLAT-TASK-004 workspaces) before
implementing showed a different scheme already in production use:
`Workspace.named_graph_iri = urn:weave:tenant:{tenant_id}:ws:{workspace_id}`. The
`sparql`, `search`, and `settings` routers already resolve reads against this per-workspace IRI,
not a per-tenant one — a tenant can hold more than one workspace, each with its own working graph.
No literal `urn:weave:g:tenant:{id}` or `urn:weave:g:framework` graph exists anywhere in the
shipped code.

Introducing the brief's literal scheme alongside the shipped one would mean two competing
named-graph conventions live in the same codebase, with no clear rule for which one a given route
reads from — a worse outcome than the flat scheme the brief was trying to prevent.

## Decision

- CE-WRITE-1's working graph, scratch clone, and version snapshots all key off the
  already-shipped `Workspace.named_graph_iri` (`urn:weave:tenant:{tenant_id}:ws:{workspace_id}`),
  not a new tenant-only IRI.
- Version IRIs are minted as `{named_graph_iri}:v{semver}` — i.e.
  `urn:weave:tenant:{tenant_id}:ws:{workspace_id}:v{semver}` — preserving the brief's `:v{semver}`
  suffix convention, applied to the real (workspace-granular) base IRI instead of the brief's
  tenant-only one.
- Provenance activities are written to `{named_graph_iri}:prov`, following the same
  workspace-granular pattern.
- Framework SHACL shapes are not (yet) split into a separate `urn:weave:g:framework` graph; they
  ship as a static Turtle file (`ontology/shapes/framework.shacl.ttl`) loaded into an in-process
  `rdflib.Graph` at first use and cached — no framework named graph exists in Oxigraph today.
  Per-tenant custom shapes are out of scope for this task (no tenant ever supplies additional
  shapes yet); the loader has a natural extension point (`_load_shapes_graph`) for a future task
  to merge in a tenant shapes subgraph once one exists.

## Consequences

- Every future CE task that reads or writes a working graph must resolve
  `Workspace.named_graph_iri`, the same way `sparql.py`/`search.py`/`settings.py` already do — not
  invent a tenant-only IRI. This ADR is the record that the brief's literal graph-naming scheme was
  superseded, so later tasks citing that scheme should treat it as historical intent, not the
  current shipped convention.
- If per-tenant custom SHACL shapes are added later (a task brief said "full 13-kind BPMO ships in
  TASK-004"), that task must decide where a tenant's shapes subgraph lives relative to a workspace
  (a tenant may have several workspaces) — this ADR does not resolve that question, only notes it
  as open.

## Relationship to the program-level ADR-001 (QA finding, CE-TASK-001 retry)

This ADR shares a number with a separate, human-confirmed, release-gating program ADR:
[`docs/specs/weave/decisions/ADR-001-tenant-isolation.md`](../../../decisions/ADR-001-tenant-isolation.md)
(`confirmed_by: gazzwi86`, `status: Accepted`, scope: platform · constitution-engine ·
graph-explorer · build-engine). Two binding ADRs on the same subject, both named ADR-001, with no
cross-reference between them until now — the gap this section fixes.

**What this (CE) ADR supersedes from the program ADR:** only the literal named-graph IRI scheme
the program ADR's table proposes -- `urn:weave:g:framework` (shared upper framework) and
`urn:weave:g:tenant:{id}` (per-tenant instances), with version IRIs of the form
`urn:weave:g:tenant:{id}:v{semver}`. CE-WRITE-1 does not use that scheme. It reuses the
already-shipped, workspace-granular scheme from PLAT-TASK-003/004:
`urn:weave:tenant:{tenant_id}:ws:{workspace_id}`, with version IRIs
`{named_graph_iri}:v{semver}` (`operations/versioning.py`). This is a naming-scheme substitution
only — one tenant may hold several workspaces, each with its own working graph, which the
program ADR's flat per-tenant table does not model; the workspace-granular scheme is a
refinement, not a rejection, of "each tenant's instance data lives in a per-tenant named graph."

**What this ADR does NOT supersede** — the program ADR's actual isolation contract is unchanged
and this ADR does not touch it:

- **The isolation boundary itself.** One shared RDF store, per-tenant (now: per-workspace,
  tenant-embedded) named graphs, mandatory fail-closed query-rewriting. CE's scheme still embeds
  `tenant_id` in every graph IRI, so the rewriter's tenant-scoping invariant holds unchanged.
- **CE-WRITE-1 as the only write path.** The program ADR states "writes never issue raw triples;
  they go through CE-WRITE-1, which targets the graph derived from the request context, not from
  the payload." `routers/operations.py` / `operations/pipeline.py` are exactly this: the target
  graph is always resolved from the caller's own authenticated workspace context
  (`ApplyContext.named_graph_iri`), and `resolve_source_graph_iri` never trusts a payload-supplied
  IRI as a write target outside that.
- **The 403 + audit response contract for a forged cross-tenant target.** The program ADR states
  plainly: "a payload naming another tenant's graph is a 403 + audit." CE-TASK-001 initially
  shipped this as a `400 {invalid_target}` with no audit entry -- a genuine contract violation,
  not a reinterpretation. This is fixed in the same retry that adds this section: see
  `operations/pipeline.py`'s `ForeignTargetError` (distinguished from a genuinely malformed
  `InvalidTargetError`, which stays a 400) and `routers/operations.py`'s
  `_reject_foreign_target`, which emits a `security.cross_tenant.rejected` audit entry before the
  403 is raised.

**Status of this note:** the naming-scheme substitution above was already in production use
before this ADR was written (PLAT-TASK-003/004 shipped first) and is judged sound — one production
scheme beats two. What requires human sign-off is this account of the boundary: that the program
ADR's *isolation guarantee* is fully preserved under the workspace-granular scheme, and that only
the literal IRI table is superseded. Do not treat this ADR as reclassifying or loosening anything
in `ADR-001-tenant-isolation.md` until that sign-off lands.
