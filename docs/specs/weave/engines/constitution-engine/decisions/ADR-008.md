# ADR-008: CE-EVENT-1 beta transport is a transactional change-feed (poll), not SNS or WebSocket

## Status

Accepted

## Context

FR-015 requires CE to emit graph-change events
`{ change_type: "added"|"updated"|"deleted"|"constraint-violated", entity_iri, version_iri, actor, ts }`
with the AC "a commit produces one event". The transport was deferred as OQ-12
(SNS fan-out vs WebSocket vs change-feed). CE M2 ships `CE-EVENT-1` at **beta**.

Constraints that shape the decision:

- Every contracted consumer already has a mandated **poll-degrade path** (poll CE-READ-1 by
  since-version): Events Engine triggers (post-v1 engine, no M2 consumer exists yet), Platform
  activity widgets (M2), Onboarding activation (Should Have, poll fallback), Explorer live-refresh
  (explicitly post-v1 for the stream; M2 ships poll). So at M2, **no consumer needs push**.
- The M1 gate's exactly-once concern (Onboarding activation "exactly once per
  (tenant,user,milestone)") is enforced consumer-side; the transport only has to not lose events.
- Tenant isolation is a security sub-gate property (ADR-001, named-graph + fail-closed): tenant A
  must never see tenant B's change events.
- Law F: no test may depend on real cloud infrastructure; local dev runs without AWS.

## Decision

**Beta transport = transactional change-feed.**

1. **Write path:** every committed CE-WRITE-1 mutation (and every `constraint-violated` outcome)
   appends one row to an append-only `graph_change_events` table in Aurora Postgres, **in the same
   database transaction as the commit's version row**. Same tenant-RLS pattern as
   `ontology_versions` (data-model §Aurora). One commit, one event — guaranteed by the
   transaction, not by a relay. Rows carry a per-tenant monotonic `seq`.
2. **Read path:** `GET /api/events?since_seq={n}&limit={m}` (JWT-scoped to the caller's tenant)
   returns ordered events after `n` plus `latest_seq`. This is the contract's beta delivery
   mechanism: consumers poll it exactly as they would poll CE-READ-1 since-version, but get typed
   change events instead of re-diffing versions.
3. **Retention:** feed rows are retained 30 days (default, tunable via PLAT-SETTINGS-1); a
   consumer whose `since_seq` has aged out gets `410 Gone` and must re-baseline via CE-READ-1 —
   never a silent empty page.
4. **Post-v1 upgrade path (explicitly planned, not built):** the same table is the **outbox** for
   a push transport — an SNS fan-out relay (or WebSocket gateway for Explorer collab) reads the
   feed and publishes. Consumers keep the poll endpoint as their degrade path, which the contract
   already mandates. Nothing in the beta design is discarded by the upgrade.

Contract impact: `contracts.md` CE-EVENT-1 "transport deferred to tech-spec" line is replaced by
"beta transport = polled change-feed (`GET /api/events?since_seq=`); push fan-out post-v1
(ADR-008)". The event shape is unchanged; `seq` is additive.

## Consequences

**Positive**

- One commit = one event is transactionally guaranteed — the FR-015 AC test is a plain
  integration test (commit, then read the feed), no message-broker test double.
- Zero new infrastructure: reuses Aurora, the existing RLS tenant-isolation pattern, and the
  existing FastAPI surface. Local dev and CI need nothing beyond the existing Postgres container
  (Law F clean).
- The degrade path and the primary path are the same mechanism — no "stream down, fall back"
  mode transition to build or test at beta.
- Push upgrade is additive (outbox already exists), so the post-v1 SNS decision is deferred
  without re-work risk.

**Negative / accepted ceilings**

- Latency floor = consumer poll interval (default 30 s per Explorer FR-025). Acceptable: no M2
  consumer has a sub-poll-interval requirement.
- Polling N consumers × M tenants adds read load to the API tier, not the RDF store. Ceiling
  noted; the post-v1 fan-out removes it. (ponytail: poll feed now, SNS relay when a consumer
  actually needs sub-30 s delivery.)
- `constraint-violated` events for **rejected** commits must be written in their own transaction
  (the mutation rolled back); this is the one place "same transaction as the commit" does not
  apply — the implementing task brief must call it out.

## Alternatives considered

- **SNS fan-out (rejected for beta):** real push, per-consumer queues — but requires LocalStack
  in every consumer test, an outbox + relay anyway (to avoid dual-write inconsistency between
  commit and publish), IAM/topic Terraform, and delivery/retry semantics — all for zero M2
  consumers that need push. It is the planned post-v1 upgrade, reading this same feed table.
- **WebSocket (rejected):** solves browser live-refresh (Explorer collab), which is explicitly
  post-v1; needs connection state, auth-on-upgrade, and horizontal-scaling fan-out (Redis
  pub/sub) — the most infrastructure for the least M2 value. Candidate as a post-v1 *consumer*
  of the fan-out, not the transport itself.
- **No new surface — consumers keep re-diffing CE-READ-1 (rejected):** cheapest, but loses typed
  `change_type`/`actor` semantics, cannot represent `constraint-violated` (a rejected commit never
  becomes a version to diff), and fails the FR-015 "a commit produces one event" AC. The contract
  is already promised at beta to Platform widgets in M2.
