# ADR-011: Exercise ASK checks bypass the client-facing SPARQL validator (TASK-009)

## Status

Accepted

## Context

TASK-009's exercise checker (`onboarding/exercise_checker.py`) needs to run a SPARQL `ASK` query
to verify CE-02/CE-03/CE-03b completion (e.g. "does the learner's sandbox graph now have a Class
node labelled 'Outdoor Furniture'"). The engine's own client-facing SPARQL endpoint
(`POST /api/sparql`, validated by `rdf/query_rewriter.py::validate_query`) only allows
`SelectQuery` and `ConstructQuery` algebra types -- `ASK` is not in `_ALLOWED_QUERY_TYPES` and is
rejected. That validator exists to stop a *client-supplied* query (arbitrary user SPARQL text)
from doing something unsafe (write-shaped query, unbounded scan, wrong graph). It's a
trust-boundary gate for **untrusted input**.

The exercise checker's ASK strings are not untrusted input. They're hard-coded Python constants in
`onboarding/exercises.py`, reviewed in the same PR as the exercise content, never built from a
request body or any other caller-supplied value. No user input is ever concatenated into them.

## Decision

`check_completion()`'s `sparql_ask` branch calls `oxigraph_client.run_query(ask_string,
named_graph_iri)` directly, in-process, bypassing `validate_query`. `named_graph_iri` is also never
client-supplied -- it's resolved server-side from the caller's own sandbox pointer
(`_resolve_sandbox_named_graph` in `routers/onboarding.py`, itself scoped by `tenant_id`/`user_id`
via `store.get_sandbox_workspace_id`).

## Consequences

- Adding a new exercise with a `sparql_ask` completion means writing a new ASK string as a Python
  constant in `exercises.py`, PR-reviewed like any other code change -- never a config value read
  from a database, request body, or environment variable. If that changes (e.g. exercises become
  admin-authorable at runtime), this bypass must be revisited and the ASK string re-routed through
  `validate_query` (which would need `AskQuery` added to `_ALLOWED_QUERY_TYPES`) or an equivalent
  static-analysis gate.
- This is not a precedent for bypassing `validate_query` on any query built from, or containing,
  caller-supplied data. The rule stays: user input never reaches SPARQL text unparameterised or
  unvalidated (`.claude/rules/security.md`).

## Alternatives considered

- **Add `AskQuery` to `_ALLOWED_QUERY_TYPES`.** Rejected for M1: that widens what the *client-facing*
  `/api/sparql` endpoint accepts from any caller, not just the exercise checker's known-safe
  strings -- a bigger surface change than this task needs, and out of this task's scope to
  architect.
- **Route through `/api/sparql` over HTTP instead of calling `run_query` in-process.** Rejected:
  extra network hop and auth round-trip for no benefit, plus it still hits the same validator wall
  above.
