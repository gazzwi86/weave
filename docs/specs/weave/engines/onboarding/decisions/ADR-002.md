# ADR-002: Sandbox isolation topology — sandbox-as-workspace, tenant-local template, batch fork

## Status

Proposed

## Context

OQ-02 asked how the per-user writable Hammerbarn copy is keyed. The frame is already half-locked:
the program tenant-isolation ADR (`docs/specs/weave/decisions/ADR-001-tenant-isolation.md`,
Accepted) mandates one shared RDF store, named graphs, and mandatory fail-closed query-rewriting;
CE ADR-001 locks the shipped workspace-granular IRI scheme
`urn:weave:tenant:{tenant_id}:ws:{workspace_id}` with CE-WRITE-1 as the only write path (target
graph resolved from the caller's authenticated workspace context, never the payload). Store-per-
tenant is therefore off the table. The PRD pins three isolation boundaries and their tests
(§2.4): per-user sandbox, sandbox-vs-canonical 403 + audit, cross-tenant-read zero-leak.

A hidden constraint surfaced during design: the demo must appear in **every tenant's** switcher,
but the rewriter forbids any cross-tenant graph read — so a single global canonical graph cannot
be forked from directly without breaching the tenant boundary.

## Decision

**Sandbox-as-workspace, with the versioned seed batch as the unit of distribution:**

1. **Seed batch artefact.** The seed job (ADR-007) compiles the Hammerbarn content brief into an
   ordered, versioned CE-WRITE-1 operation-batch artefact (`hammerbarn-seed vX.Y.Z`). The batch —
   not a graph — is what crosses tenant boundaries, as code/content, never as a store read.
2. **Tenant-local canonical template.** Each tenant gets one "Hammerbarn Canonical" workspace,
   materialised by applying the seed batch and publishing a version (CE-VERSION-1). Workspace
   permissions make it read-only to everyone except the content-admin principal: any other write
   is rejected **403** by the existing workspace write-permission check and audited via
   PLAT-AUDIT-1. It is hidden from the switcher for non-admins.
3. **Per-user sandbox = a real Workspace row**, lazy-created at first demo access, keyed to the
   user, named-graphed as `urn:weave:tenant:{tid}:ws:{sandbox_ws_id}` like any workspace. The
   switcher's single "Hammerbarn Demo" entry resolves to the caller's own sandbox. Fork = apply
   the pinned seed batch into the sandbox draft via CE-WRITE-1 (`target=draft`) under the demo
   service principal; the sandbox records which batch semver it forked from.
4. **Reset = blue/green swap.** Create a fresh sandbox workspace, apply the latest seed batch,
   atomically swap `sandbox_workspace_id` in the onboarding state row, then delete the old
   workspace. The user's sandbox is always in a known state — fully old or fully new, never
   partial. Target ≤ 30 s (default, tunable); fork target ≤ 10 s p95.
5. **Zero CE changes.** Fork, reset, and all exercise writes use only CE-WRITE-1 / CE-READ-1 and
   platform workspace CRUD. The rewriter, RLS, and the 403 + audit contract are exercised, not
   extended. Onboarding remains a terminal consumer.

### The three boundaries, mapped to enforcement

| Boundary (PRD §2.4) | Enforced by | Release-gate test |
|---|---|---|
| Per-user sandbox | Sandbox is a distinct workspace; CE resolves the write target from the caller's workspace context, never the payload | User-A token cannot read/write user-B's sandbox graph (zero triples / 403) |
| Sandbox vs canonical (403) | Workspace write-permission on the template workspace; rejection audited via PLAT-AUDIT-1 | Non-content-admin write to canonical → 403 + audit row asserted |
| Cross-tenant zero-leak | CE's fail-closed query rewriter (program ADR-001); every graph IRI embeds `tenant_id` | Unscoped query under tenant-A/user-A JWT returns zero tenant-B and zero other-user triples |

## Consequences

- 100% reuse of shipped machinery (Workspace rows, rewriter, CE-WRITE-1 pipeline, PROV, version
  publishing); the isolation boundaries are the platform's existing ones, exercised harder.
- Workspace-row count grows by one per demo-active user per tenant — trivial at seed scale
  (a few thousand triples per sandbox); no delta/copy-on-write machinery to own.
- Fork-by-batch-application means fork and reset share one code path with the canonical seeding —
  one implementation, three uses.
- A user's sandbox does **not** auto-update on canonical re-seed (PRD pin); only reset moves it to
  the latest batch.
- Contract gap flagged: platform workspace CRUD (create/delete/permission) is shipped code but not
  yet a named `PLAT-*` contract clause — cited here as PLAT-SETTINGS-1 tenant-model machinery;
  the Platform spec should name it explicitly.

## Alternatives Considered

- **New `:sandbox:{user_id}` graph-IRI suffix inside one shared demo workspace** — fewer workspace
  rows, but CE routes resolve graphs via `Workspace.named_graph_iri`, so this needs CE-side
  changes, breaching the terminal-consumer boundary and adding a second graph convention.
- **Delta-overlay / copy-on-write from canonical** — storage-optimal but queries become
  union-plus-tombstones; real complexity for a seed of a few thousand triples.
- **Direct store-level graph COPY for forking** — fast, but a privileged raw-store operation
  outside every contract; batch re-application achieves the same result contract-pure.
- **Store-per-tenant** — already rejected by the Accepted program ADR; not re-opened.
