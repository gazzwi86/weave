# ADR-007: Hammerbarn seed application — idempotent Python CLI emitting CE-WRITE-1 batches

## Status

Proposed

## Context

Decision E2 locks the seed as a **live pipeline** — content flows through the real product
contracts, not a migration snapshot. The seed-lifecycle contract (EPIC-001) pins triggers
(ontology semver major bump → re-seed; manual dispatch always available), ownership (CE owns the
ontology/glossary/brand/governance run; onboarding integrates), atomicity (failed re-seed leaves
the previous canonical intact), and version pinning (sandboxes pin the batch they forked from).
The content itself is authored in `docs/specs/weave/hammerbarn-content-brief.md` (approved):
BPMO-grounded 13 kinds, 5 processes, actors, systems, data assets, punned Class/Concept
vocabulary, policies, glossary, brand. Build/Events seed portions (Kitchen Designer, automations)
are post-v1 and out of this ADR.

## Decision

- **Seed compiler + applier = one Python CLI** (`hammerbarn-seed`) in the repo, run with `uv`:
  - *Compile:* transform the content brief's entities/edges into ordered CE-WRITE-1
    operation batches (`add_node`/`add_edge` with local `ref` resolution), emitted as a versioned
    JSON artefact (`hammerbarn-seed vX.Y.Z`) committed alongside the CLI. Kinds and relationship
    types are validated against `GET /api/ontology/types` (CE-READ-1) at compile time — never a
    hand-copied kind list (ontology-standards rule).
  - *Apply:* POST the batches to `POST /api/operations/apply` under the **content-admin service
    principal** (PLAT-IDENTITY-1) with `target=draft` on the target workspace, then publish a
    version. CE's SHACL validation and PROV attribution apply as for any write — the pipeline is
    "live" by construction.
  - *Idempotent:* CE-WRITE-1's dedup (case-insensitive `label`+`kind` reuses existing nodes) plus
    an idempotency key per batch make re-runs converge; a partial failure is retried from the
    failed batch without duplicating earlier ones.
- **Three uses, one code path:** canonical template seeding (per tenant), per-user sandbox fork,
  and reset all apply the same versioned batch artefact (ADR-002).
- **Dispatch:** GitHub Actions `workflow_dispatch` (content admin) and a triggered run on CE
  ontology **major**-version bump (minor/patch → advisory review only, per the seed-lifecycle
  contract). Canonical promotion sits behind the roadmap's HITL publish/generate gate
  (content admin + Tech Lead).
- **Tests never touch cloud** (Law F): the CLI runs in CI against the in-process FastAPI app with
  an in-memory Oxigraph store; the dispatch workflow deploys nothing in test.

## Consequences

- The demo can never drift from the product: if CE-WRITE-1 or the SHACL shapes change
  incompatibly, the seed run fails loudly in CI before any user sees a broken demo.
- Content changes are reviewable diffs against the content brief and the compiled batch —
  the publish gate reviews exactly what will be applied.
- Seed application is slower than a bulk Turtle load (each batch walks the validation pipeline);
  at a few thousand triples this is well inside the ≤ 10 s fork p95 and ≤ 30 s reset targets.

## Alternatives Considered

- **Static Turtle bulk-load into the store** — fast, but bypasses CE-WRITE-1 validation and PROV,
  violating locked decision E2 and creating a second write path (the exact thing the program
  isolation ADR forbids).
- **Runtime seeding service** — a standing service for an operation that runs on dispatch and
  version bumps; a CLI + workflow is the minimum machinery.
