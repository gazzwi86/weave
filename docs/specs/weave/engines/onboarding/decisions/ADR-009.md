# ADR-009: TASK-003 i18n key convention and audit-bootstrap default

## Status

Proposed

## Context

TASK-003's DoR names "i18n key convention confirmed with the frontend package" -- but
`packages/frontend` has no i18n library or key convention today (grep for `i18n`/`next-intl`/
`useTranslations` returns nothing). The brief itself mandates "all copy = i18n keys with an `en`
catalogue" as the TASK-003 deliverable, so there is no existing convention to conflict with;
inventing one locally is in-scope, not a gap.

Separately, ADR-008 amended the anchor registry to carry `shipped: boolean` +
`planted_by: TASK-NNN`, replacing phase-based CI-audit gating. Its Consequences section says "M1
anchors are grandfathered `shipped: true` (their surfaces are live)" -- but TASK-003's Scope Note
is explicit that this task plants **zero** `data-tour-id` attributes (that lands in TASK-007/008/
010/012). Grandfathering M1 anchors `shipped: true` now would make the two-way audit fail
immediately (rule: `shipped: true` + attribute absent from code -> fail) against a real frontend
tree with no attributes anywhere -- contradicting ADR-008's own "atomic flip in the same PR that
plants the attribute" Decision-section rule, and blocking a green TASK-003 PR.

## Decision

- **i18n key format** (`packages/shared/onboarding/content/i18n.ts`): dot-namespaced, lowercase,
  hyphen-word segments -- `onboarding.<kind>.<id>.<field>`, e.g.
  `onboarding.tour.ce-overview.step1.title`. Enforced by regex in the key-format CI check
  (AC-003-06). Local to `packages/shared`; a future frontend-wide i18n library adopts or wraps
  this convention rather than the reverse (onboarding ships first).
- **`shipped` defaults `false` for every anchor at TASK-003**, including M1 ones, following
  ADR-008's Decision-section atomic-flip rule over its Consequences-section "grandfathered" aside.
  Each anchor's `planted_by` names the task that will flip it to `true` in the same PR that adds
  its `data-tour-id` attribute (TASK-007 for CE/Explorer, TASK-008 for the no-tour modals,
  TASK-014 for M2/post-v1 placeholders). The two-way audit is therefore vacuously green at
  TASK-003 (no anchor is `shipped: true` yet, so `missingShipped` is empty by construction) and
  starts enforcing drift the moment the first owning task flips its anchors.

## Consequences

- TASK-003's PR is green against the real frontend tree with zero frontend changes, matching its
  Scope Note.
- TASK-007/008/010/012/014 each flip a specific `shipped` bit atomically with their attribute --
  ADR-008's drift guarantee holds from their first PR onward.
- ADR-008's "grandfathered" aside is effectively superseded for the M1 set by this ADR; a future
  ADR-008 revision could fold this clarification back in.

## Alternatives Considered

- **Grandfather M1 anchors `shipped: true` at TASK-003, per ADR-008's literal text** -- fails the
  two-way audit against the real (attribute-free) frontend tree immediately; rejected.
- **Skip the audit's missing-shipped direction until a later task** -- weakens the gate instead of
  sequencing it correctly (Law 6/plugin-laws); rejected.
