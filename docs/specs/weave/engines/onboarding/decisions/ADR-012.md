# ADR-012: canvas_state exercise completions are client-asserted, not server-verified (TASK-009)

## Status

Accepted

## Context

TASK-009's exercise checker dispatches four completion kinds. Two of them are server-verified
against real backend state that the client cannot forge: `sparql_ask` (queries the caller's own
sandbox graph) and `nav_signal` -- wait, `nav_signal` is *not* server-verified either, it's
client-asserted (see below); only `sparql_ask` genuinely reads independent backend state today.

`canvas_state` (GE-01 spotlight-active, GE-02 heatmap-overlay-active) is meant to confirm the
learner actually triggered a Graph Explorer canvas interaction. But contract `GE-CANVAS-1` (the
canvas's own state) has no server-side read surface in this milestone -- canvas state (spotlight
target, active overlay) lives entirely client-side in the Explorer's React state, never persisted
or exposed via an API the backend could poll or query. There is nothing on the server to check
this against.

`nav_signal` (CE-01) has the identical problem for a different reason: it confirms the learner
*viewed* something (entity list, missing-property view), which is inherently a client-observed UI
event with no durable server trace to check.

## Decision

Both `canvas_state` and `nav_signal` are checked by matching the client's *claimed* signal strings
(`ExerciseCheckRequest.signals`) against the exercise's required signal/state string --
`check_completion()`'s `nav_signal` and `canvas_state` branches never touch the database or any
independent state store. This is a deliberate trust reduction relative to `sparql_ask`: a
malicious or buggy client could claim a signal it never actually triggered, and the server has no
way to catch that for these two kinds.

## Consequences

- CE-01/GE-01/GE-02 are lower-assurance completions than CE-02/CE-03/CE-03b. This is acceptable
  for M1 because the entire feature is onboarding progress tracking (a checklist/badge UX), not an
  access-control or billing gate -- the worst case of a forged `canvas_state`/`nav_signal` claim is
  a learner's own progress checklist shows a tick they didn't strictly earn. No data, permission,
  or paid-tier boundary depends on it.
- If a future milestone wants real GE-CANVAS-1 server verification, that requires the Graph
  Explorer to persist or expose canvas interaction state server-side (e.g. an audit event on
  spotlight/overlay activation) that this checker could then query the same way `sparql_ask` does
  -- out of scope for TASK-009.

## Alternatives considered

- **Block GE-01/GE-02 entirely until a server-verifiable signal exists.** Rejected: would drop two
  of the six required M1 exercises with no near-term unblock, and the risk being accepted (a
  learner's own progress checklist) doesn't warrant it.
- **Log the claimed signal as an audit event for later reconciliation.** Deferred, not built here --
  would need `PLAT-AUDIT-1` wiring not currently in this task's brief.
