# ADR-004: Activation detection — poll-first on CE-READ-1/CE-VERSION-1, CE-EVENT-1 behind a flag

## Status

Proposed

## Context

Activation milestones are detected from CE-grounded signals in the user's **own** workspace
(E5-S2): Business/Technical → first committed entity by that actor (Technical alternatively a
SPARQL run via CE-READ-1); Compliance → first governance/SHACL view via CE-READ-1. The PRD makes
CE-EVENT-1 a Should-Have with an explicit degrade: poll CE-READ-1 with a since-version. Meanwhile
CE-EVENT-1's transport (SNS / WebSocket / change-feed) is itself still undecided in
`contracts.md`. Building a consumer against an unpinned transport would fabricate a dependency
(the same trap Events' architecture D9 guards against). Admin activation stays manual self-mark
(OQ-08 accepted default) and needs no detector.

## Decision

- **Poll-first.** The M1 detector is a scheduled poller per demo-active, not-yet-activated user:
  resolve the latest published version of the user's own workspace via CE-VERSION-1, and when it
  advances past the stored cursor, run the milestone's named CE-READ-1 check (e.g. an ASK for an
  entity whose PROV attribution is the user's principal). Poll interval default 60 s, tunable;
  the poller stops for a user once all their applicable milestones have fired (the activation
  table is the stop condition).
- **Detection is idempotent by ADR-003** — `INSERT … ON CONFLICT DO NOTHING` — so the detector
  can safely re-check, race a future event consumer, or be re-run after a crash.
- **CE-EVENT-1 consumer behind a feature flag.** The detector exposes one internal entry point
  (`record_milestone(tenant, user, milestone, source)`); when CE pins the event transport, a thin
  consumer maps `{change_type:"added", actor}` events onto the same entry point and the flag
  flips. The poller remains as permanent degrade (the PRD requires it even after CE-EVENT-1 GA).
- **Locked milestones.** A milestone whose engine or signal is unavailable stays locked and the
  detector never evaluates it — no mis-fire (FR-022 failure mode).

## Consequences

- M1 ships with zero dependency on an unpinned contract; the degrade path is the tested,
  production path — not an afterthought.
- Activation latency is bounded by the poll interval (≤ ~60 s), well within the "first week"
  activation window the metric cares about; the toast feels near-immediate because the client
  refreshes checklist state on the triggering screen anyway.
- The event consumer, when it lands, is a small adapter — no rework of detection, idempotency,
  or outbox logic.

## Alternatives Considered

- **Build both now** — speculative against an undecided transport; likely rework; violates
  minimum-code.
- **Client-side detection** (SPA reports "user did X") — trivially spoofable, double-fires on
  refresh, and ties correctness to the browser session; rejected for a metric that gates product
  decisions.
