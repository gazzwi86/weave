# ADR-005: Tour-anchor strategy — data-tour-id attributes + typed registry in packages/shared

## Status

Proposed

## Context

OQ-06: tour steps and beacons must attach to UI elements owned by other engines' feature teams,
survive their refactors, and cope with screens that have not shipped yet (Build/Events surfaces
are post-v1; some CE/Platform surfaces are M2). "Tour anchors break when feature teams change UI"
is the PRD's highest-likelihood risk. The runtime failure mode is already pinned: an absent
anchor skips the step with a logged warning, never blocks (FR-008).

## Decision

- **First-class `data-tour-id` attributes.** Every tour/beacon target carries a
  `data-tour-id="<anchor-id>"` attribute in the owning feature's JSX. CSS selectors are never
  used as anchors.
- **Typed anchor registry** in `packages/shared` (`onboarding/anchors.ts`): a single `const` map
  `anchor id → { engine, area, phase }` where `phase ∈ m1 | m2 | post-v1`. Tour and beacon
  configs (ADR-006) may only reference registry keys — enforced by the config's zod schema
  (TypeScript `keyof` typing makes an unknown anchor a compile error).
- **Phase tags drive feature-flagging.** A tour/beacon whose any-anchor `phase` is later than the
  currently shipped phase set renders feature-flagged off — the uniform "no tour, beacon, or
  modal for a half-enabled area" epic AC falls out of one check.
- **CI audit.** A repo script greps the frontend for `data-tour-id` values and diffs against the
  registry both ways: an unregistered attribute or a registered-as-shipped anchor missing from
  the codebase fails CI. This is the drift alarm feature-team refactors trip *before* merge.
- **Runtime resilience unchanged:** absent anchor ⇒ skip step / hide beacon + `warn` log with the
  anchor id — defence in depth behind the CI audit.

## Consequences

- Other feature teams adopt a one-line convention (add an attribute + a registry entry); the
  registry PR review is the coordination point — cheap, additive, no runtime coupling.
- Not-yet-shipped screens have registry entries but no DOM — their tours are flagged off by phase
  tag, so nothing dangles and post-v1 tours are authored now without breaking anything.
- The registry is the single source for the PRD's guided-coverage audit (success criterion:
  every primary navigation area covered).

## Alternatives Considered

- **CSS selectors** — zero cross-team coordination but breaks on any refactor; directly feeds the
  top PRD risk; rejected.
- **Registry + selector fallback hybrid** — maximum resilience, double bookkeeping; the runtime
  skip + CI audit already covers the failure mode the fallback would.
