# ADR-003: State store + activation idempotency — Aurora RLS tables, unique-constraint insert, outbox

## Status

Proposed

## Context

Onboarding state — chosen path, tour resume points, beacon/modal dismissals, checklist state,
exercise completion, activation flags — must persist server-side per `(tenant, user)` (PRD §2.4;
localStorage is cache only). Activation must fire **exactly once** per
`(tenant, user, milestone)` — the celebration toast and the `onboarding-activation` publish to
PLAT-NOTIFY-1 must not double-fire even when the poll detector and a future CE-EVENT-1 consumer
race (FR-022). Onboarding owns no graph data by charter, so RDF is not an option. Scope note:
the human descoped EPIC-008 (analytics) from the M1 slice — this ADR deliberately contains **no
analytics tables, no SQS pipeline, and no event sink**; the outbox carries PLAT-NOTIFY-1
publishes only.

## Decision

- **Storage:** onboarding-owned tables in the existing Aurora PostgreSQL cluster, every table
  carrying `tenant_id` with **fail-closed row-level security** (no session context ⇒ zero rows) —
  the same pattern CE and Events use (Events ADR-003 precedent). Schema in
  `tech-spec/data-model.md`.
- **Surface:** an `/api/onboarding/*` router module inside the existing `packages/backend`
  FastAPI app — no new service, no new deployable. Caller identity resolves via PLAT-IDENTITY-1;
  every route is authz-checked and tenant-scoped before any read.
- **Idempotency:** the `activation` table has a primary key `(tenant_id, user_id, milestone_id)`;
  detection performs `INSERT … ON CONFLICT DO NOTHING`. Only the statement that actually inserts
  the row wins; in the same transaction it writes an `outbox` row. A re-trigger (poll + event
  race, redelivery, replay) conflicts and does nothing — exactly-once by construction, no
  distributed lock.
- **Transactional outbox:** a small dispatcher drains `outbox` rows to PLAT-NOTIFY-1
  (`onboarding-activation` type) and marks them dispatched; retries on failure. The in-app toast
  renders from the activation row itself (client polls/receives state), so a notify outage never
  blocks or duplicates the celebration.
- **Reset interaction:** sandbox reset clears `exercise_completion` rows (re-earnable, epic AC)
  but never deletes `activation` rows — activation is a lifetime milestone, not sandbox state.

## Consequences

- One store serves progress, resume, dismissals, checklist, and activation — single RLS pattern,
  single migration stream, single test fixture.
- Exactly-once needs no queue semantics; correctness rests on a database constraint, which
  mutation testing can meaningfully attack.
- When EPIC-008 lands (post-M1), analytics gets its own decision; nothing here presupposes it —
  the outbox event type taxonomy is open but currently has exactly one member.

## Alternatives Considered

- **PLAT-SETTINGS-1 cascade as the store** — that contract is settings *resolution*
  (Company→Domain→Workspace→Project, tighter-wins); per-user progress data is not a setting, and
  the cascade write model is not shaped for high-frequency progress updates. Semantic misuse.
- **localStorage** — banned by the PRD (cross-device resume).
- **SQS FIFO dedup for idempotency** — 5-minute dedup window only; not exactly-once forever.
- **Check-then-insert application logic** — race-prone under the poll/event double path.
- **RDF store** — onboarding owns no graph data; contradiction with the engine charter.
