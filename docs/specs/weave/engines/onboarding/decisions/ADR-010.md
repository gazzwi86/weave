# ADR-010: Training/What's-new config shape extension (TASK-012)

## Status

Accepted

## Context

TASK-012's brief scope note says "Frontend only + one state field. Renders TASK-003 config" --
but TASK-003's merged `TrainingEntrySchema`/`WhatsNewItemSchema` (`packages/shared/onboarding/content/schema.ts`)
had no `category`, `durationSeconds`, or `version` fields, while AC-012-01 requires seven named
categories (with `Build`/`Automation` flagged post-v1) and AC-012-04 requires
"version, date, headline, description" on every What's-new item. The brief names the requirement;
only the mechanical schema shape was left open.

## Decision

Extend both schemas additively (no breaking change to existing config consumers -- `collectKeys`,
`copy-budget`, `dead-cta` checks only read `*Key` fields, untouched):

- `TrainingEntrySchema` gains `category` (new `TrainingCategoryIdSchema` enum, required),
  `durationSeconds` (optional), `walkthroughBodyKey` (optional, pairs with the existing
  `writtenWalkthroughUrl` i18n key to render an inline disclosure).
- New `TrainingCategorySchema` + `TRAINING_CATEGORIES` config (`categories.ts`) -- seven
  categories, reusing the existing `EngineAvailabilitySchema` (`shipped`/`m2`/`post-v1`) for the
  post-v1 flag rather than inventing a parallel enum.
- `WhatsNewItemSchema` gains `version` (required).
- Seed content (`training.ts`, `whats-new.ts`) updated minimally: one playable-shaped video entry
  (still no real upload), one written-walkthrough entry (E2E needs something openable), one
  What's-new item with a version string.

## Consequences

- TASK-013 (launcher) and any future content PR inherit the richer shape for free.
- Category *availability* is a property of the category, not of individual entries -- an entry's
  own `category` field is a plain enum member; whether the category itself is "post-v1" is looked
  up via `TRAINING_CATEGORIES`, not duplicated per entry.
- No CI check (`collect-keys.ts`, `copy-budget.ts`, `dead-cta.ts`) needed a corresponding change --
  they only touch `*Key` fields.

## Alternatives Considered

- **Escalate as a blocker (Law 11)** -- rejected: the brief already names every field and category;
  only the schema mechanics were undecided, which is squarely an ADR-sized decision, not a spec
  gap needing human adjudication.
- **Per-entry `availability` field instead of a category-level lookup** -- more duplication for no
  behavioural gain; categories, not individual entries, are what "ships when the engine ships".
