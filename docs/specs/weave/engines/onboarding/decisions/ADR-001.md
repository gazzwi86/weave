# ADR-001: Tour/overlay framework — Driver.js + owned React wrapper

## Status

Proposed

## Context

OQ-01 asked build-in-house vs adopt a tour library. Requirements (E2-S1/S2/S3, FR-008–FR-012):
dimmed overlay + spotlight on a target element, tooltip with Back/Next + step indicator,
skippable/resumable (resume point server-side per `(tenant, user)`), fully keyboard-navigable
(Tab/Arrow/Enter/Escape), WCAG 2.1 AA zero-axe-violations, absent-anchor steps skip + log, all
copy from i18n keys, styling from `docs/standards/design/` tokens (no ad-hoc hex/px), and no
requirement to interact with the highlighted element to advance. The frontend is Next.js/React 19
with Radix primitives already installed.

Only the spotlight/positioning engine is at stake: the tour state machine, server-side resume,
role gating, and anchor resolution are onboarding-owned in every option. Beacons, welcome modals,
the checklist widget, and the help launcher are ordinary Radix/shadcn components — no library
needed there.

## Decision

Adopt **Driver.js** (MIT, zero runtime dependencies, ~5 kB gzip, framework-agnostic) for the
spotlight overlay, element highlighting, scroll-into-view, and reposition-on-resize mechanics,
wrapped in one owned React module (`TourEngine`):

- **The wrapper owns everything stateful:** step sequencing from the code-shipped tour config
  (ADR-006), resume-point persistence via `/api/onboarding/state`, role gating, absent-anchor
  skip + warn logging, and feature-flag checks against the anchor registry phase tags (ADR-005).
  Driver.js is invoked per-step as a dumb renderer; it never holds authoritative tour state.
- **Design tokens:** Driver.js popover/overlay class hooks are styled exclusively from design
  tokens; the `ui_verify` gate applies. No Driver.js default theme CSS ships.
- **Accessibility:** keyboard bindings (Tab/Arrow/Enter/Escape), `aria` attributes, and
  focus management are asserted in the wrapper's own tests plus the axe E2E lane — verified once
  in the wrapper, inherited by every tour.
- **React 19 note:** Driver.js is vanilla DOM, so React-version churn cannot break it; the wrapper
  is the only React-coupled surface.

## Consequences

- Spotlight masking, scroll, resize repositioning, and stacking-context edge cases — the defect
  surface of this epic — arrive from a maintained MIT library instead of being owned code.
- One new frontend dependency: mainstream, MIT, zero transitive deps — acceptable under Law A.
- The wrapper is the single choke point for tour a11y and token compliance; tests concentrate
  there (testing-strategy.md §TypeScript).

## Alternatives Considered

- **Intro.js** — AGPL for commercial use; excluded outright (licence risk, Law A written-ack
  burden for no capability gain).
- **Reactour** — React-component-based; React 19 compatibility unproven at decision time; smaller
  community than Driver.js.
- **Shepherd.js** — MIT core but heavier (floating-ui + deps) and drifting toward a "Pro" upsell;
  more CSS to fight for the design-token gate.
- **Build in-house on Radix + floating-ui** — perfect token/a11y control and zero new deps, but
  onboarding would own spotlight masking, scroll-into-view, and resize repositioning — the most
  defect-prone part of the epic, for zero product differentiation (Law E, minimum-code).
