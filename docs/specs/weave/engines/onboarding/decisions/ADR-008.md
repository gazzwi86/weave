# ADR-008: Per-anchor shipped signal

## Status

Proposed — amends ADR-005 (gating clause only; attribute + registry + runtime-skip mechanics
unchanged).

## Context

ADR-005 gates overlays by milestone phase tag: a `phase: "m2"` anchor keeps its tours flagged
off until "the M2 phase is shipped", and the two-way CI audit only fails on *shipped-phase*
entries missing from code. The red-team found the contradiction: onboarding config merges
independently of the owning surfaces (by design), so during that window an anchor whose
attribute was planted and then **dropped by an owner refactor** is indistinguishable from one
**not yet planted** — the audit stays green and the overlay silently no-ops. One milestone-wide
flag also flips all 14 anchors at once, though the owning surfaces (GE, CE, Platform) land at
different times.

## Decision

- **Registry entry gains two fields:** `shipped: boolean` (default `false`) and
  `planted_by: TASK-NNN` (the single onboarding task that plants the attribute — exactly one
  owner per anchor, making double-planting a reviewable claim).
- **Atomic flip:** `shipped` flips to `true` in the **same PR** that plants the
  `data-tour-id` attribute. The audit enforces atomicity from both sides (below), so
  flag-without-attribute and attribute-without-flag are both unmergeable.
- **Two-way CI audit keys on `shipped`, not phase:**
  1. `shipped: true` ∧ attribute absent from code → **fail** (drift alarm — an owner refactor
     that drops a planted anchor goes red on that PR).
  2. Attribute present in code ∧ (no registry entry ∨ `shipped: false`) → **fail** (forces the
     atomic flip; also the M1 unregistered-attribute rule, unchanged).
- **Per-anchor overlay gating:** a tour/beacon/modal is offered only when **every anchor it
  references** has `shipped: true`. Tours over different surfaces therefore flip independently
  (strictly finer than the milestone flag; the "never a half-enabled area" epic AC still falls
  out, per tour).
- **`phase` stays as descriptive metadata** (grouping, reporting, roadmap traceability) with no
  gating semantics. M1 anchors are grandfathered `shipped: true` (their surfaces are live).
- **Runtime resilience unchanged** (absent anchor ⇒ skip/hide + warn) — defence in depth behind
  the audit, exactly as ADR-005.

## Consequences

- Drift is impossible in both windows: pre-flip there is no attribute to drop and no overlay is
  offered (nothing can silently no-op); post-flip a dropped attribute fails CI on the PR that
  drops it. No other engine emits anything — attributes are planted by onboarding tasks and the
  audit runs in shared monorepo CI.
- The M2 release gate (m2 TASK-005) becomes mechanical: assert every M2 registry anchor is
  `shipped: true` — an unshipped Must-Have surface is a red gate, never a quietly-missing tour.
- Cost: one boolean + one id per registry entry and a two-line audit change; no new machinery.

## Alternatives Considered

- **Milestone phase flag (status quo, ADR-005)** — cannot distinguish dropped from not-yet-
  shipped during the independent-merge window; rejected (red-team Blocker).
- **Owning engines emit a readiness signal** (e.g. per-surface manifest each engine publishes) —
  cross-engine coordination and a new convention for information the onboarding monorepo already
  has at merge time; rejected as needless coupling.
- **Gate on runtime anchor presence only** — hides drift until a user session; the audit exists
  precisely to catch it pre-merge; rejected.
