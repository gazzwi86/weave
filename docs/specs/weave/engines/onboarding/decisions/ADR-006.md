# ADR-006: Content delivery — code-shipped typed config; video via S3 + CloudFront

## Status

Proposed

## Context

Two content questions, one posture. (1) Where do tour steps, beacon copy, welcome modals,
exercises, checklist items, training-library entries, and the What's-new feed live — code or a
database/CMS? The content-admin persona "curates tours and exercises", but v1 defines no
authoring UI. (2) OQ-04: where does training video stream from when produced? v1 ships
placeholders only (non-goal: producing videos).

## Decision

**Content = code-shipped versioned typed config** in `packages/shared` (`onboarding/content/`):

- TypeScript/JSON modules validated by zod schemas (data-model.md §Content config): tours,
  beacons, welcome modals, exercises, checklist item sets per path, training-library entries,
  What's-new items, and the role→starter-widget-set mapping (E3-S2). All user-facing strings are
  i18n keys, never literals.
- **Content-admin workflow = pull request.** Curation lands as reviewed, versioned diffs; the
  publish gate is code review plus the existing CI checks — no CMS built.
- **CI enforcement encoded as config tests** (they run on every content PR):
  - *Dead-CTA reconciliation:* every welcome modal with a "Take a tour" CTA references an
    existing tour for that area; no-tour areas carry the no-tour CTA set (E2 epic AC).
  - *Copy budgets:* tour tooltips ≤ 40 words, beacon tooltips ≤ 60 words (defaults, tunable —
    budget constants live beside the schema).
  - *Anchor validity:* config references only registry anchor ids (ADR-005).
  - *Phase/role tags present* on every tour, beacon, exercise, and checklist item.

**Video = S3 + CloudFront, in-stack.** Training cards are built around a CloudFront URL shape
(`https://<training-cdn>/onboarding/<video-id>/<rendition>.mp4`) with a native HTML `<video>`
player (poster = card thumbnail, captions via `<track>` when produced). v1 renders the
"Video — coming soon" placeholder state whenever the config entry has no `video_id`; a load or
playback failure falls back to the same placeholder/error card, never a broken player (E6-S1).

## Consequences

- Content correctness is testable and reviewable: the two epic-AC classes that only fail "when
  two stories disagree" (dead CTAs, half-enabled areas) become deterministic CI failures.
- No new vendor, no CMS surface, no runtime content fetch for overlays (config ships in the
  bundle; the What's-new unread dot uses per-user state from ADR-003).
- Real video later requires zero card rework — populate `video_id`, upload renditions, done.
  Signed CloudFront URLs can be added if training content ever becomes tenant-restricted.
- If a future phase needs live (non-deploy) content editing, that is a new decision — a CMS is a
  post-v1 candidate, not smuggled in now.

## Alternatives Considered

- **DB-stored, admin-editable content (mini CMS)** — live editing for the content admin, but a
  whole authoring/preview/permission surface the PRD never asks for in v1 (Law E, minimum-code).
- **Third-party video (Wistia/Vimeo)** — better player analytics/captions tooling; new vendor +
  cost + Law A bus-factor acknowledgement, for a v1 that ships placeholders. Revisit only if
  video analytics become a requirement.
- **Defer the video decision entirely** — cheapest now but risks card rework and an unplanned
  migration when videos land; the URL-shape contract costs nothing today.
