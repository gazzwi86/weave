# Epic: EPIC-004 - Authentication, RBAC & Agent Identity

## Overview

**Phase:** Phase 1 (MVP) — Platform Shell & Cross-Cutting Foundations
**PRD Reference:** [prd.md](../prd.md#epic-4-authentication-rbac--agent-identity)
**Status:** Backlog
**Priority:** Must Have

## Description

The identity and access spine for both humans and agents: SSO via Cognito (default) or Auth0
(multi-IdP), RBAC enforced at the API boundary against roles resolved through `PLAT-SETTINGS-1`, and
named least-privilege agent principals minted by the `PLAT-IDENTITY-1` registry. This epic establishes
the two distinct auth paths — Cognito for humans, IAM/STS for machines — and the canonical principal IRI
that threads every PROV-O record and audit entry.

## Acceptance Criteria (Epic Level)

- [ ] The human and machine auth paths are cleanly disjoint and both least-privilege: human requests
      authenticate via Cognito JWT and authorize via RBAC; agent access to AWS/secrets assumes an IAM
      role via STS (short-lived credentials, never raw secret values) — a single auth-path test confirms
      no human path can mint AWS credentials and no agent path uses a Cognito session.
- [ ] Every authorization denial — unauthorized API op, out-of-scope agent action, IdP outage — fails
      closed and is recorded: a role-lacking-permission op returns 403 + `PLAT-AUDIT-1` entry, an
      out-of-scope principal is denied + logged, and an IdP outage yields a defined error with no
      unauthenticated fallback session — verified by one fail-closed sweep across all three.
- [ ] The canonical principal IRI is single-sourced: the `PLAT-IDENTITY-1` registry reconciles platform
      agent classes, Build's 5 dark-factory roles, and Events' per-automation principals into one IRI
      that appears in PROV-O and in every `PLAT-AUDIT-1` entry, and maps to exactly one IAM role and one
      RBAC role at the Weave API boundary — verified by tracing one agent action end to end.

## Technical Notes

- Cognito is the default human IdP (email/password + hosted UI); Auth0 SAML/OIDC is configured per
      workspace (Google Workspace, Okta, Azure AD).
- RBAC is enforced at the API via JWT validation; roles control read/author/publish/admin per
      engine/area. The full role matrix lives in the tech spec.
- `PLAT-IDENTITY-1` records which IAM role maps to which canonical principal IRI and to which RBAC role.
      Admins can view agent principals and scopes in Settings.
- The "Weave platform operator" is a platform-internal identity that never appears in any client
      workspace's RBAC (see EPIC-009 self-improvement authority).
