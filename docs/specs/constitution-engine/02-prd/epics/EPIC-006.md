# Epic: EPIC-006 - SHACL Validation (Cross-Cutting)

## Overview

**Phase:** Phase 1 — Validated-Mutation Spine (MVP)
**PRD Reference:** [prd.md](../prd.md#epic-6-shacl-validation-cross-cutting)
**Status:** Backlog
**Priority:** Must Have

## Description

The validated-mutation spine the entire engine depends on: exactly **one** mutation entry point
(`POST /api/operations/apply`, CE-WRITE-1) that clones the target graph, applies the batch, runs
SHACL, and commits only if zero `sh:Violation`. Provides actionable error messages and a standalone
whole-graph validation endpoint. No other content area may write except through this gate.

## Acceptance Criteria (Epic Level)

- [ ] There is **exactly one** mutation entry point to the trusted graph; a CI test asserts **no**
      auto-apply path and **no** raw SPARQL-Update write surface exists (the prototype's legacy
      auto-apply `POST /api/llm/mutate` bypass is absent) (FR-003).
- [ ] Prospective validation **clones** the target graph, applies the batch to the clone, runs SHACL,
      and commits to the real graph only if zero `sh:Violation` — a batch with any single violation
      commits **nothing** (all-or-nothing on the clone) (FR-004).
- [ ] Severity routing is uniform: `sh:Violation` → HTTP 422 with the graph unchanged;
      `sh:Warning` and `sh:Info` are surfaced as advisory and **never** block — each severity is
      exercised by an integration test (FR-005).
- [ ] Error messages name the violated shape, the offending node/path, and the constraint with a
      plain-English explanation; with the AI "fix this" helper unavailable, the raw structured
      violation (focus node, path, severity, message) is still shown so the user can fix it manually.
- [ ] `GET /api/validate` returns a full tenant-scoped SHACL report (JSON-LD/Turtle) over all active
      shapes; a non-existent version → 404; a request without a valid JWT → 401.

## Dependencies

## Technical Notes

- This epic **is** CE-WRITE-1's validation core (EPIC-010 wraps the public write contract around it).
  The single-entry-point invariant (FR-003) is a security property gated at the phase-boundary
  security-review.
- Clone-then-validate-then-commit is grounded in the prototype's clone-store commit gate
  (`store.py`); the heavier batch reasoner runs separately at publish (EPIC-008), not here.
- Validation runs `inference='none'` (closed-world data-quality enforcement — the "Polikoff rule"
  OWL/SHACL split); OWL DL semantics are not load-bearing for SHACL.
- Standards: SHACL (`sh:Violation`/`sh:Warning`/`sh:Info`), SHACL validation report (JSON-LD/Turtle).
