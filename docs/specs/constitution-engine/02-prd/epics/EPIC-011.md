# Epic: EPIC-011 - Authoring Surfaces — Chat AND Forms (Cross-Cutting)

## Overview

**Phase:** Phase 1 — Validated-Mutation Spine (MVP)
**PRD Reference:** [prd.md](../prd.md#epic-11-authoring-surfaces--chat-and-forms-cross-cutting)
**Status:** Backlog
**Priority:** Must Have

## Description

Delivers the two v1 authoring surfaces (decision B4) that every content area shares: a persistent NL
chat panel available on all Constitution screens, and SHACL-shape-driven guided forms that let a
business user author without AI and without RDF. Every AI proposal is shown as a plain-English
operation list and confirmed through the single validated mutation pipeline. Server-side conversation
persistence (E11-S4) is carved out to Phase 4.

## Acceptance Criteria (Epic Level)

- [ ] The chat panel mounts on **every** Constitution screen, defaults to that screen's content area,
      and every AI proposal shows a **human-readable operation list / plain-English summary** as the
      default review surface — a raw-Turtle view is available **on demand for the ontologist/power-user
      persona only**, never the default (FR-002).
- [ ] Both surfaces funnel through the **same** single validated mutation pipeline (CE-WRITE-1);
      guided forms render fields from the selected class's SHACL shapes and apply partial-update
      semantics (E2-S2) — neither surface has a private write path.
- [ ] With the AI provider unavailable, chat returns HTTP 503 with a clear offline message **while
      guided forms, browse, and SPARQL surfaces remain fully functional** — a user can complete any
      authoring task via forms at 503; the structured operation list still renders without the prose
      summary.
- [ ] Form-side validation is layered: a required field left blank produces an inline error before
      submit, and a server-side `sh:Violation` returns 422 with the violation shown against the field.

## Technical Notes

- Guided forms (B4) reuse the prototype Inspector / NodeEditModal / AddNodeForm patterns; forms are
  **driven by SHACL shapes** for the selected kind — the form is generated from the shape, not
  hand-built per class.
- The op-list-not-Turtle default (FR-002) is load-bearing for the non-technical persona; Turtle is a
  power-user affordance only.
- Partial-update semantics (E2-S2 / FR-034) apply to form submissions exactly as to chat edits — only
  named properties are retracted/asserted.
- The 503-degrade behaviour is the engine-wide reliability contract (FR-001/FR-018/FR-033): NL surfaces
  fail soft; forms/browse/SPARQL stay live. Accessibility: chat + forms are WCAG 2.1 AA with a
  zero-violations axe gate and full keyboard navigation.
