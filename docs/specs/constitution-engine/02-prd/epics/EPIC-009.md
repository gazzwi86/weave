# Epic: EPIC-009 - Provenance & Version Lifecycle

## Overview

**Phase:** Phase 1 — Validated-Mutation Spine (MVP)
**PRD Reference:** [prd.md](../prd.md#epic-9-provenance--version-lifecycle)
**Status:** Backlog
**Priority:** Must Have

## Description

Makes every change auditable and the model versionable: every committed batch produces an append-only
PROV-O activity (recording human-vs-AI authorship) and emits the same event to PLAT-AUDIT-1, and the
engine maintains a draft→published version lifecycle with immutable version IRIs and a server-computed
diff. This is the provenance + versioning half of the MVP spine and the basis for CE-DIFF-1 /
CE-VERSION-1.

## Acceptance Criteria (Epic Level)

- [ ] **100%** of committed changes across all content areas carry **both** a PROV-O `prov:Activity`
      (in append-only `weave:graph/prov`, recording LLM = `prov:SoftwareAgent`, approving human =
      `prov:Person`, and the PLAT-IDENTITY-1 principal IRI) **and** a PLAT-AUDIT-1 event — and a failed
      audit emit is retried and the discrepancy logged, never silently accepted as audited (FR-006).
- [ ] PROV-O records are immutable and append-only: **no** principal, including admin, can delete a
      provenance record (it is not an available operation) — prior `weave:graph/prov` content is
      byte-identical after any later commit.
- [ ] Publishing snapshots the draft into an **immutable** version IRI, generates a PROV-O change log
      from the diff, and leaves all prior published versions **byte-identical**; `?version=latest`
      resolves to the **newest published** version (decision B2); a publish that fails the consistency
      check (Phase 4) leaves prior versions and the draft intact with no partial version IRI.
- [ ] `GET /api/ontology/diff?from=&to=` returns a **server-computed** diff of added/removed/**modified
      nodes AND edges** (an edge-only change appears in `modified`); a diff against a non-existent
      version → 404; a diff of a version against itself → empty diff, not an error (CE-DIFF-1).
- [ ] `GET /api/ontology/versions` is the single source for `is_latest` and canonical version-lag
      (count of published versions strictly between a pin and latest); consumers read lag, never
      recompute it (CE-VERSION-1).

## Dependencies

- **Blocked by:** EPIC-006 (every PROV-O stamp rides the single validated mutation pipeline).

- **Blocks:** EPIC-010 (CE-READ-1's `?version=` pinning and `/versions` endpoint depend on this

## Technical Notes

- This epic owns **CE-DIFF-1** and **CE-VERSION-1** (they land in Phase 1, not Phase 2 — the roadmap
  is explicit they are not re-claimed by the contract-hub work).
- The diff must be **server-computed** over both nodes and edges — an edge-only change is the canonical
  test case that catches client-side diffing shortcuts.
- PROV-O human-vs-AI authorship is the load-bearing trust distinction from the prototype ("AI proposed
  / human approved"); model the LLM as `prov:SoftwareAgent` and the approver as `prov:Person` via
  `prov:wasAssociatedWith` the approval activity.
- Canonical version-lag is defined **once** here (CE-VERSION-1); "stale" default threshold lag ≥ 2,
  tunable — consumers must not re-derive it.
- Standards: PROV-O, named-graph versioning (semver version IRIs), DCTERMS, append-only audit.
