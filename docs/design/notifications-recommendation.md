# Notifications — role & type recommendation

Grounding: `PLAT-NOTIFY-1` (channel + type preferences, backend done); tenancy ruling ("publish
notifies members"); SES email channel is **post-v1** (WS1-GAP ruling) — so v1 ships **in-app only**,
with the preference model already channel-shaped so email slots in later without migration.

## Types (v1 set — 8, grouped)

| Group | Type | Fires when | Default recipients |
|---|---|---|---|
| Model | `model.version.published` | A constitution version goes draft → published | **All members** (the ruling's "publish notifies members") |
| Model | `model.change.mention` | An entity you authored/steward is edited or deleted | Author + data steward |
| Model | `model.conformance.regression` | SHACL rejects spike / conformance drops | Compliance officer, enterprise architect |
| Build | `build.request.completed` | A requested application finishes (success) | Requester |
| Build | `build.request.failed` | Generation/gate failure needs a human | Requester + workspace admin |
| Governance | `audit.chain.invalid` | Chain verification fails (integrity incident) | Workspace admin + compliance officer (not suppressible) |
| Account | `billing.cap.warning` | Budget burn crosses 80% / 100% of cap | Workspace admin |
| Account | `member.added` | You are added / your role changes | Affected member |

Deliberately **not** notification types: routine reads, every write (that is the audit trail's job),
placeholder-phase features. Notification = "something changed that plausibly needs *you*"; the
audit trail remains the complete record.

## Role → default matrix (summary)

- **Workspace admin:** everything except `model.change.mention` (opt-in).
- **Compliance officer:** conformance + chain + publishes.
- **Enterprise architect / data steward:** model group.
- **Engineer:** build group + publishes.
- **Analyst/SME and other roles:** publishes + own mentions + own account events.

All defaults are per-type toggleable in Settings → Notifications (the missing section — assessment
finding F-D24) except `audit.chain.invalid`, which is mandatory for admin + compliance (an
integrity incident must not be silenceable).

## Channels

- **v1: in-app** — bell with unread badge, panel listing (icon, one-line summary, relative time,
  deep link), mark-read/mark-all. Digest = none in v1 (volume is low at seed scale).
- **post-v1: email (SES)** — per-type immediate/daily-digest/off; the preference schema carries a
  `channel` dimension from day one so this is additive.

## UX surfaces

1. **Bell (header):** icon + unread count badge (assessment F-D03 replaces the text label); panel
   groups by day; empty state gets a short "what notifies you" hint + link to preferences.
2. **Settings → Notifications:** matrix UI — rows = types (grouped as above), columns = channels
   (in-app now; email column pre-rendered disabled with a `post-v1` phase pill, consistent with
   the rail's placeholder pattern).
3. **Toast** only for events generated by *your own current action* (e.g. build request accepted);
   never for ambient events (they go to the bell).

## Resolved (user MCQ, 2026-07-09)

`model.version.published` defaults to **all members, batched per session** — successive publishes
within a short window collapse into one "v0.3.0→v0.3.4 published" entry. Honours the "publish
notifies members" ruling without authoring-time spam.
