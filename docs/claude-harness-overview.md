## 1. What the harness is

The Weave harness is a Claude Code configuration that turns Claude into a **spec-driven
dark-factory builder** for the Weave product itself. It implements:

1. **SDLC skills** вЂ” composable per-artifact skills that produce high-quality spec documents
   with research rounds, model right-sizing, and section-by-section HITL review
2. **Persona agents** вЂ” orchestration shells (PO, Architect, Engineer, QA) that call skills
   in sequence and hold the Laws, handoff contracts, and phase boundaries
3. **Dark factory loop** вЂ” an autonomous implementation mode that builds entire phases
   unattended, pausing only at phase-completion gates for human review
4. **Hooks** вЂ” lifecycle guards that enforce quality (no secrets, uv-only, anatomy freshness,
   phase-gate HITL, drift detection)
5. **OKF knowledge graph** вЂ” a unified Open Knowledge Format bundle spanning all docs; rendered
   to `docs/viz.html` for interactive browsing

**What it is NOT:** Weave's product features (Build Engine, Constitution Engine). This harness
builds Weave. The Build Engine is a separate sub-system that Weave will generate for customers.

## 6. Persona agents

Agents are orchestration shells. They hold Laws (non-negotiable constraints), a sequenced
list of skill invocations, and HITL contracts. They do NOT produce artifacts directly вЂ”
they delegate to skills.

| Agent | File | Triggered by | Calls skills |
|---|---|---|---|
| Product Owner | `agents/product-owner.md` | `/po` | po-brief, po-prd, po-roadmap, po-epic |
| Tech Architect | `agents/tech-architect.md` | `/architect` | arch-stack + 12 arch-* skills |
| Engineer | `agents/engineer.md` | implement skill (subagent) | scaffold, TDD implementation |
| QA | `agents/quality-assurance.md` | implement skill (subagent) | qa skill + reporting |
| Prototyper | `agents/prototyper.md` | `/prototype` | prototype skill |

### Agent Laws (universal, baked into every agent)

- **Law A** вЂ” common-stack first (Weave defaults, В§ Stack)
- **Law B** вЂ” functional, browser-runnable, automation-tested (Playwright for UI)
- **Law C** вЂ” council-graded quality for enterprise claims (7-persona, в‰Ґ 4.0/5)
- **Law D** вЂ” stacked PRs (one PR per EPIC; small commits stack within a phase)
- **Law E** вЂ” complexity as a budget (cyclomatic в‰¤ 10, cognitive в‰¤ 15, fn в‰¤ 50 lines)
- **Law F** вЂ” synthetic verification only (LocalStack, not real cloud in tests)

## 7. Dark factory вЂ” how it works

The dark factory mode runs the implementation loop autonomously between human-review gates.

### Starting a run

```bash
/implement          # continue from current state (reads progress.json)
/implement TASK-003 # start a specific task
```

### PDAC loop mechanics

```
1. Read .claude/state/progress.json в†’ next unstarted task whose deps are done
2. Set /goal: "all tasks in current phase done, or stop after 60 turns"
   Haiku evaluates the condition after each turn
3. Per task (PLAN в†’ DELEGATE в†’ ASSESS в†’ CODIFY):
   PLAN:     Read task brief + dependency summaries from .claude/state/summaries/
             Verify DoR checklist is satisfied
   DELEGATE: Invoke engineer subagent (worktree isolation)
             - Write ALL failing tests first (TDD)
             - Implement until tests pass
             - /simplify + /code-review
             - Commit to feature branch
   ASSESS:   Invoke qa subagent (worktree isolation)
             - 10-category validation
             - Mutation testing (в‰Ґ 70%)
             - If FAIL: feed report back to engineer, retry (max 3)
   CODIFY:   Write .claude/state/summaries/TASK-NNN.md
             Update + commit .claude/state/progress.json
             Open PR
4. When all phase tasks complete:
   Stop hook fires в†’ phase_gate() detects phase completion в†’ re-prompts Claude
5. Claude presents: kanban summary, PR list, phase gate checklist
6. AskUserQuestion: Approve в†’ next phase | Amend в†’ describe changes | Reject в†’ replan
7. If Approve: advance phase in progress.json, loop to next phase
```

### Overnight / unattended pattern

```
Push to backlog-ready branch (or daily cron)
в†’ /implement reads committed progress.json
в†’ Builds tasks autonomously
в†’ Commits progress.json after each task
в†’ Opens PRs for human review
Gate: PR approval = human advance signal
```

## 8. Hooks вЂ” what fires when and why

All hooks live in `.claude/scripts/hooks.py` (Python, module-dispatched). Each hook module
owns one logical concern.

### Active hooks

| Event | Matcher | Handler | Effect |
|---|---|---|---|
| PreToolUse | Edit\|Write | `check-no-secrets` | Blocks hardcoded secrets before write |
| PreToolUse | Edit\|Write + Bash | `check-uv-over-pip` | Blocks `pip install` (uv only) |
| PostToolUse | Edit\|Write | `mark-anatomy-stale` | Appends stale marker to area wiki page |
| PostToolUse | Edit\|Write | `commit-progress` | Auto-commits `.claude/state/progress.json` when changed |
| Notification | вЂ” | `notification` | Logs Claude messages to stderr |
| UserPromptSubmit | вЂ” | `user-prompt-submit` | Injects project memory context |
| Stop | вЂ” | `phase_gate` | Fires HITL gate when all phase tasks are done |
| Stop | вЂ” | `drift_check` | Context hygiene suggestions via local LLM (optional) |
| SubagentStop | вЂ” | `subagent_stop` | Injects task summary back to parent implement loop |
| PreCompact | вЂ” | `pre_compact` | Snapshot state before context compaction |
| SessionStart | вЂ” | `check_setup_status` | Warns if git hooks are not wired |
| SessionEnd | вЂ” | `session_end` | Flush pending state |

### Phase gate mechanics

The `phase_gate` function in `stop.py` reads `.claude/state/progress.json`. When all
tasks show `done` status:

```
Hook fires в†’ writes HITL prompt to stderr в†’ sys.exit(2) (Claude re-reads stdin)
Claude re-starts в†’ formats kanban + phase gate checklist
AskUserQuestion: Approve / Amend / Reject
Approve в†’ implement skill continues to next phase
```

The `stop_hook_active: true` payload guard prevents re-entry from `/goal`'s own Stop hook.

### Anatomy staleness tracking

The `mark_anatomy_stale` function in `wiki.py` runs on every Edit/Write. It:
1. Maps the edited file to its wiki area via `common.area_for_path()`
2. Only acts on files under tracked roots: `apps/`, `packages/`, `infra/`, `src/`, `lib/`
3. Appends `<!-- stale: <path> -->` to that area's `docs/wiki/<area>.md`
4. The pre-push git hook (`check-anatomy-fresh`) blocks if stale markers remain

## 9. OKF knowledge graph

The harness maintains an **Open Knowledge Format (OKF) v0.1** bundle spanning all docs.
This makes every spec, standard, and wiki page navigable as a connected knowledge graph.

### What the bundle contains

| Bundle path | Types included |
|---|---|
| `docs/specs/weave/engines/<entity>.md` | Brief, PRD, Roadmap, Tech Spec artifacts, ADRs, Task Briefs |
| `docs/standards/` | Coding Standards |
| `docs/wiki/` | Code Area anatomy pages |
| `docs/index.md` | Bundle root (OKF metadata only) |
| `docs/log.md` | Bundle change history |

### OKF conformance rules

Every non-reserved `.md` file must have:
- YAML frontmatter block (hard error if absent)
- Non-empty `type` field (hard error if absent)

Recommended fields: `title`, `description`, `tags`, `timestamp`.

Reserved files: `index.md` (no frontmatter in sub-dirs; only `okf_version` at root),
`log.md` (no frontmatter; ISO 8601 date headings only).

### Generating the graph

```bash
# Validate conformance (hard errors only)
uv run .claude/scripts/okf_validate.py docs/

# Validate with warnings as errors (CI mode)
uv run .claude/scripts/okf_validate.py docs/ --strict

# Render interactive graph
uv run .claude/scripts/okf_visualize.py docs/ -o docs/viz.html
open docs/viz.html
```

The visualizer renders:
- **Nodes** вЂ” one per OKF concept, coloured by `type`, sized by content length
- **Edges** вЂ” markdown cross-links between concept files
- **Panel** вЂ” click any node to see its full content, outgoing links, and "Cited by" backlinks

Cross-links between spec files use bundle-relative markdown links:
```markdown
[constitution-engine brief](engines/constitution-engine.md#brief)
```

### Nested sub-bundle note

`/anatomy` validates `docs/wiki/` as its own sub-bundle for code anatomy checks.
`/okf-validate` (and the unified graph) operates on the full `docs/` bundle.
The root `docs/index.md` has only `okf_version`; `docs/wiki/index.md` (when created by
`/anatomy`) is a non-root index within the unified bundle вЂ” treated as a navigation aid,
not an OKF concept.

                    # answer questions for brief, PRD, roadmap, epics
                    # each section shown before write; approve/amend/reject

# Step 3 вЂ” architecture (after PO artifacts are approved)
/architect          # generates stack в†’ C4 в†’ API в†’ data model в†’ task briefs
                    # each section shown before write

# Step 5 вЂ” build

                    # pauses at phase boundaries for your approval

## 13. Conventions

- **Spec location**: `docs/specs/weave/engines/<entity>.md`
- **Template location**: `.claude/spec-templates/<artifact>.md` (stays in `.claude/`)
- **Progress**: `.claude/state/progress.json` (always committed; routines depend on it)
- **Skills**: `.claude/skills/<skill-name>/SKILL.md`
- **Agents**: `.claude/agents/<persona>.md`
- **Hooks**: `.claude/scripts/hooks.py` + modules in `.claude/scripts/modules/`
- **No `pip install`**: uv only (enforced by hook вЂ” exits 2 on `pip` usage)
- **No hardcoded secrets**: AWS Secrets Manager only (enforced by hook)
- **Conventional commits**: `feat:`, `fix:`, `docs:`, `test:`, `chore:` (required by commitlint hook)
