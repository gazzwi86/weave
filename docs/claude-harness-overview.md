# Weave Harness — Complete Guide

> The Weave AI development harness. Read this before touching harness configuration.
> Last updated: 2026-06-29 (Pass 2 overhaul).

---

## 1. What the harness is

The Weave harness is a Claude Code configuration that turns Claude into a **spec-driven
dark-factory builder** for the Weave product itself. It implements:

1. **SDLC skills** — composable per-artifact skills that produce high-quality spec documents
   with research rounds, model right-sizing, and section-by-section HITL review
2. **Persona agents** — orchestration shells (PO, Architect, Engineer, QA) that call skills
   in sequence and hold the Laws, handoff contracts, and phase boundaries
3. **Dark factory loop** — an autonomous implementation mode that builds entire phases
   unattended, pausing only at phase-completion gates for human review
4. **Hooks** — lifecycle guards that enforce quality (no secrets, uv-only, anatomy freshness,
   phase-gate HITL, drift detection)
5. **OKF knowledge graph** — a unified Open Knowledge Format bundle spanning all docs; rendered
   to `docs/viz.html` for interactive browsing

**What it is NOT:** Weave's product features (Build Engine, Constitution Engine). This harness
builds Weave. The Build Engine is a separate sub-system that Weave will generate for customers.

## 3. Command reference

### Primary workflow commands

| Command | What it does |
|---|---|
| `/po` | Product Owner orchestration: brief → PRD → roadmap → epics. Delegates to po-* skills, section-by-section HITL at each artifact. |
| `/architect` | Tech Architect orchestration: stack confirmation → C4 → OpenAPI → data model → flows → class → CI/CD → testing → DoD/DoR → infra → ADRs → task briefs. |
| `/elicit` | Structured elicitation using one of four methods: 20Q, Six Hats, Five Whys, Stochastic Reasoning. Use before `/po` for new problem areas. |
| `/implement` | Dark factory orchestration: PDAC loop (Plan → Delegate → Assess → Codify) per task, with phase-gate HITL. |
| `/qa` | 10-category task validation against the task brief. |
| `/spec-review` | Pre-scaffold spec completeness gate. Checks all artifacts before any code is written. |
| `/status` | Progress dashboard: kanban board, current phase, blocked tasks. |

### Knowledge graph commands

| Command | What it does |
|---|---|
| `/anatomy` | Full regeneration of `ANATOMY.md` + `docs/wiki/<area>.md` OKF bundle + `docs/viz.html`. |

## 4. The SDLC flow — step by step

### From idea to implementation

```
1. /elicit          Establish context — 20Q for new areas, Six Hats for contested ones
2. /po              Brief → PRD → roadmap → epics (section-by-section HITL at every section)
3. Human review     Read approved docs/specs/weave/engines/<entity>.md, 02-prd/, 03-roadmap/
4. /architect       Stack → C4 → OpenAPI → data model → … → task briefs (HITL at every section)
5. /spec-review     Completeness gate — must pass before any code
6. /implement       Dark factory loop: auto-builds each task; phase gate at phase end
7. /qa              QA validates each task (auto-called by implement; run manually to recheck)
```

### The spec cascade


## 6. Persona agents

Agents are orchestration shells. They hold Laws (non-negotiable constraints), a sequenced
list of skill invocations, and HITL contracts. They do NOT produce artifacts directly —
they delegate to skills.

| Agent | File | Triggered by | Calls skills |
|---|---|---|---|
| Product Owner | `agents/product-owner.md` | `/po` | po-brief, po-prd, po-roadmap, po-epic |
| Tech Architect | `agents/tech-architect.md` | `/architect` | arch-stack + 12 arch-* skills |
| Engineer | `agents/engineer.md` | implement skill (subagent) | scaffold, TDD implementation |
| QA | `agents/quality-assurance.md` | implement skill (subagent) | qa skill + reporting |
| Prototyper | `agents/prototyper.md` | `/prototype` | prototype skill |

### Agent Laws (universal, baked into every agent)

- **Law A** — common-stack first (Weave defaults, § Stack)
- **Law B** — functional, browser-runnable, automation-tested (Playwright for UI)
- **Law C** — council-graded quality for enterprise claims (7-persona, ≥ 4.0/5)
- **Law D** — stacked PRs (one PR per phase, small commits)
- **Law E** — complexity as a budget (cyclomatic ≤ 10, cognitive ≤ 15, fn ≤ 50 lines)
- **Law F** — synthetic verification only (LocalStack, not real cloud in tests)

## 7. Dark factory — how it works

The dark factory mode runs the implementation loop autonomously between human-review gates.

### Starting a run

```bash
/implement          # continue from current state (reads progress.json)
/implement TASK-003 # start a specific task
```

### PDAC loop mechanics

```
1. Read .claude/state/progress.json → next unstarted task whose deps are done
2. Set /goal: "all tasks in current phase done, or stop after 60 turns"
   Haiku evaluates the condition after each turn
3. Per task (PLAN → DELEGATE → ASSESS → CODIFY):
   PLAN:     Read task brief + dependency summaries from .claude/state/summaries/
             Verify DoR checklist is satisfied
   DELEGATE: Invoke engineer subagent (worktree isolation)
             - Write ALL failing tests first (TDD)
             - Implement until tests pass
             - /simplify + /code-review
             - Commit to feature branch
   ASSESS:   Invoke qa subagent (worktree isolation)
             - 10-category validation
             - Mutation testing (≥ 70%)
             - If FAIL: feed report back to engineer, retry (max 3)
   CODIFY:   Write .claude/state/summaries/TASK-NNN.md
             Update + commit .claude/state/progress.json
             Open PR
4. When all phase tasks complete:
   Stop hook fires → phase_gate() detects phase completion → re-prompts Claude
5. Claude presents: kanban summary, PR list, phase gate checklist
6. AskUserQuestion: Approve → next phase | Amend → describe changes | Reject → replan
7. If Approve: advance phase in progress.json, loop to next phase
```

### Overnight / unattended pattern

```
Push to backlog-ready branch (or daily cron)
→ /implement reads committed progress.json
→ Builds tasks autonomously
→ Commits progress.json after each task
→ Opens PRs for human review
Gate: PR approval = human advance signal
```

## 8. Hooks — what fires when and why

All hooks live in `.claude/scripts/hooks.py` (Python, module-dispatched). Each hook module
owns one logical concern.

### Active hooks

| Event | Matcher | Handler | Effect |
|---|---|---|---|
| PreToolUse | Edit\|Write | `check-no-secrets` | Blocks hardcoded secrets before write |
| PreToolUse | Edit\|Write + Bash | `check-uv-over-pip` | Blocks `pip install` (uv only) |
| PostToolUse | Edit\|Write | `mark-anatomy-stale` | Appends stale marker to area wiki page |
| PostToolUse | Edit\|Write | `commit-progress` | Auto-commits `.claude/state/progress.json` when changed |
| Notification | — | `notification` | Logs Claude messages to stderr |
| UserPromptSubmit | — | `user-prompt-submit` | Injects project memory context |
| Stop | — | `phase_gate` | Fires HITL gate when all phase tasks are done |
| Stop | — | `drift_check` | Context hygiene suggestions via local LLM (optional) |
| SubagentStop | — | `subagent_stop` | Injects task summary back to parent implement loop |
| PreCompact | — | `pre_compact` | Snapshot state before context compaction |
| SessionStart | — | `check_setup_status` | Warns if git hooks are not wired |
| SessionEnd | — | `session_end` | Flush pending state |

### Phase gate mechanics

The `phase_gate` function in `stop.py` reads `.claude/state/progress.json`. When all
tasks show `done` status:

```
Hook fires → writes HITL prompt to stderr → sys.exit(2) (Claude re-reads stdin)
Claude re-starts → formats kanban + phase gate checklist
AskUserQuestion: Approve / Amend / Reject
Approve → implement skill continues to next phase
```

The `stop_hook_active: true` payload guard prevents re-entry from `/goal`'s own Stop hook.

### Anatomy staleness tracking

The `mark_anatomy_stale` function in `wiki.py` runs on every Edit/Write. It:
1. Maps the edited file to its wiki area via `common.area_for_path()`
2. Only acts on files under tracked roots: `apps/`, `packages/`, `infra/`, `src/`, `lib/`
3. Appends `<!-- stale: <path> -->` to that area's `docs/wiki/<area>.md`
4. The pre-push git hook (`check-anatomy-fresh`) blocks if stale markers remain

## 9. OKF knowledge graph

The harness maintains an **Open Knowledge Format (OKF) v0.1** bundle spanning all docs.
This makes every spec, standard, and wiki page navigable as a connected knowledge graph.

### What the bundle contains

| Bundle path | Types included |
|---|---|
| `docs/specs/weave/engines/<entity>.md` | Brief, PRD, Roadmap, Tech Spec artifacts, ADRs, Task Briefs |
| `docs/standards/` | Coding Standards |
| `docs/wiki/` | Code Area anatomy pages |
| `docs/index.md` | Bundle root (OKF metadata only) |
| `docs/log.md` | Bundle change history |

### OKF conformance rules

Every non-reserved `.md` file must have:
- YAML frontmatter block (hard error if absent)
- Non-empty `type` field (hard error if absent)

Recommended fields: `title`, `description`, `tags`, `timestamp`.

Reserved files: `index.md` (no frontmatter in sub-dirs; only `okf_version` at root),
`log.md` (no frontmatter; ISO 8601 date headings only).

### Generating the graph

```bash
# Validate conformance (hard errors only)
uv run .claude/scripts/okf_validate.py docs/

# Validate with warnings as errors (CI mode)
uv run .claude/scripts/okf_validate.py docs/ --strict

# Render interactive graph
uv run .claude/scripts/okf_visualize.py docs/ -o docs/viz.html
open docs/viz.html
```

The visualizer renders:
- **Nodes** — one per OKF concept, coloured by `type`, sized by content length
- **Edges** — markdown cross-links between concept files
- **Panel** — click any node to see its full content, outgoing links, and "Cited by" backlinks

Cross-links between spec files use bundle-relative markdown links:
```markdown
[constitution-engine brief](engines/constitution-engine.md#brief)
```

### Nested sub-bundle note

`/anatomy` validates `docs/wiki/` as its own sub-bundle for code anatomy checks.
`/okf-validate` (and the unified graph) operates on the full `docs/` bundle.
The root `docs/index.md` has only `okf_version`; `docs/wiki/index.md` (when created by
`/anatomy`) is a non-root index within the unified bundle — treated as a navigation aid,
not an OKF concept.

**Run `/po` and answer the questions. That's it.**

More precisely:


## 12. Model right-sizing

Skills declare their model tier. No guessing — the tier is fixed per artifact type.

| Task type | Model | Rationale |
|---|---|---|
| Elicitation / creative brief writing | claude-opus-4-8 | Wide reasoning, novel framing |
| Architecture design, C4 diagrams | claude-opus-4-8 | Spatial reasoning, tradeoff analysis |
| Task brief writing | claude-opus-4-8 | Self-contained, must be complete |
| PRD stories, flows, OpenAPI | claude-sonnet-5 | Structured but not exploratory |
| Data models, class diagrams | claude-sonnet-5 | Precision over creativity |
| Stack confirmation, open dimensions | claude-haiku-4-5 (detect) + HITL | Fast pattern matching; human confirms |
| DoR/DoD, YAML/config generation | claude-haiku-4-5 | Mechanical, template-following |
| Validation, lint checks | claude-haiku-4-5 | Fast, cheap, deterministic |
| Code implementation (engineer) | claude-sonnet-5 | TDD iteration |
| Security review | claude-opus-4-8 | High-stakes, adversarial |

## 13. Conventions

- **Spec location**: `docs/specs/weave/engines/<entity>.md`
- **Template location**: `.claude/spec-templates/<artifact>.md` (stays in `.claude/`)
- **Progress**: `.claude/state/progress.json` (always committed; routines depend on it)
- **Skills**: `.claude/skills/<skill-name>/SKILL.md`
- **Agents**: `.claude/agents/<persona>.md`
- **Hooks**: `.claude/scripts/hooks.py` + modules in `.claude/scripts/modules/`
- **No `pip install`**: uv only (enforced by hook — exits 2 on `pip` usage)
- **No hardcoded secrets**: AWS Secrets Manager only (enforced by hook)
- **Conventional commits**: `feat:`, `fix:`, `docs:`, `test:`, `chore:` (required by commitlint hook)

## 14. Key design decisions

| Decision | What | Reference |
|---|---|---|
| Skill granularity | Per artifact/template (not per persona, not per section); each skill maps 1:1 to a spec template — templates are the natural artifact seams | Harness redesign, 2026-06-25 |
| HITL gates | Spec = section-by-section; implement = phase-gated groups (PO-defined) | Harness redesign, 2026-06-25 |
